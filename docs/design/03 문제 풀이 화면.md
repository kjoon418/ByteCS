**전제:** `00 공통 디자인 원칙.md`. 이 문서는 이 페이지의 목적·구성 요소만 기술한다. (이 화면이 서비스의 히어로이므로 가장 상세하다.)

# 1. 어떤 페이지인가
- **문제 풀이 화면** — 일일 세션 중 **문제 하나를 푸는** 화면. 서비스의 히어로.
- 도메인 근거: 기능 1(주관식 단답·결정적 판정·**근접 신호**·무낙인·**정답 공개 후 직접 입력**·**심화 정보**), 기능 2(**문제별 자유 구성 힌트**), 기능 2.5(오답 교정 힌트), 기능 5(스크랩).
- 사용자는 여기서: ①문제를 읽고 ②정답을 **직접 적어** 제출하고 ③막히면 **힌트를 펼치고** ④끝내 안 되면 **정답을 열어**(보고 다시 직접 입력) 풀고 ⑤맞힌 뒤 **더 알아보기**로 흥미를 잇는다.
- **디딤 문제(파고들기)도 이 화면을 재사용할 예정** — MVP에는 없다. **[결정 2026-07-16] 디딤 UI는 시안에서 제거, 로드맵(디딤 문제 연결)에서 재도입.**

# 2. 이 페이지의 테마 강조점
- **차분·집중.** 문제 하나에 시선 집중, 여백 넉넉. 코드 스니펫 대응(고정폭).
- **비처벌이 핵심.** 틀림에 빨강·경고·벌점 절대 금지. "아직이에요, 다시 해볼까요?" 톤.
- **또렷한 시각 피드백(리뷰 반영).** '정답 확인하기'를 누른 순간 정답/불일치/근접(오탈자) 각 상태가 **분명하고 즉각적인 시각+햅틱 변화**로 느껴져야 한다(밋밋함 금지). 단 강렬함이 곧 처벌색은 아님 — 정답은 긍정(success), 불일치·근접은 중립·격려 톤으로 또렷하게.
- 파고들기(디딤 문제)의 **'내려갔다 올라오는' 깊이감** 전환은 로드맵에서 도입 시 표현한다(MVP 제외 — 위 §1 결정 참고).

# 3. 구성 요소 (구체)
> **핵심 위계: Primary Action은 '정답 제출' 하나.** 힌트·정답 보기·신고는 secondary로 시각 강도를 낮춘다.

## A. 상단 — 세션·맥락 바
1. **세션 진행 인디케이터** — 오늘의 한입 중 몇 번째 **본 문제**인지(예: `2 / 5`). **시간 카운트다운 아님.** 담백하게.
2. **파고들기 맥락 배지 + 복귀 경로** — 디딤 문제로 내려간 상태일 때만. "〈원래 문제〉를 위한 더 쉬운 문제예요" + **돌아가기**. **여러 겹 가능** → 깊이(스택) 표현 가능해야 함. **MVP 제외 — 로드맵에서 디딤 도입 시 재도입(위 §1 결정).**
3. **나가기/중단** — 재개되므로 **부담 없는 나가기**(경고 모달 지양, "언제든 이어서" 톤).

## B. 문제 영역
4. **문제 질문 텍스트** — 화면의 주인공. 청킹·여백. **코드 스니펫은 모바일 폭에서 가로 스크롤 없이 읽히도록**(짧은 줄 큐레이션 전제) — 가로 스크롤로 컨텍스트가 끊기지 않게. 부득이한 긴 코드만 가로 스크롤 컨테이너로 격리하고 본문 흐름은 유지(리뷰 반영).
5. **난이도 표시(선택·은은)** — MVP는 표시 속성. 로드맵의 개인 수준 맞춤 적응 출제·디딤 문제(둘 다 도입 시)의 소비처로 이어질 예정. 압박 안 되게 최소화. **[반영 2026-07-19: D5]** 풀기 전 노출(은은한 표시)을 현행대로 유지 확정 — 실기기 QA에서 시작 전 위축 신호가 관찰되면 재검토.
6. **개념 칩** — ⚠️ **풀기 전에는 숨긴다**(개념명이 정답을 스포일할 수 있음: 정답 '해시 충돌' ↔ 칩 '해시 충돌'). **정답 확인·공개 이후에만** 노출.

## C. 답 입력 영역 (Primary)
7. **주관식 단답 입력 필드** — 텍스트 입력. **입력 최적화 키패드**, **자동 포커스**. 재시도 시 이전 입력 유지·수정 가능.
8. **정답 제출 CTA** — **동사형**("정답 확인하기"). **엄지 영역 하단 고정·확장**. 화면의 유일한 Primary.

## D. 피드백 (상태별 — 무낙인, 또렷하게)
9. **정답 상태** — **또렷한** 긍정 피드백(success 색 + 체크 + **햅틱** + 짧은 축하 모션), "맞았어요!" → 다음 문제로 부드럽게. 이때 개념 칩·간단 해설·[더 알아보기] 노출 가능.
10. **불일치 상태(오답 아님)** — **분명하지만 비처벌인 인라인 넛지**("아직이에요, 다시 해볼까요?"). 제출 순간 눈에 띄는 상태 변화를 주되 **빨강·경고 금지**, 토스트로 휙 사라지게 하지 말 것(설명이 남아야 함). 입력 유지·재시도.
11. **근접(오탈자) 신호(리뷰 반영)** — 불일치가 오탈자 수준으로 가까우면 "거의 맞았어요, 오타를 확인해보세요"를 별도 톤으로 표시(정답·개념 비노출). '진짜 오답인지 오타인지'의 혼란을 덜어 준다. **정의 재생형에만 적용.** **[반영 2026-07-19: D1]** 단, 아래 17번 **정답 직접 입력(따라 입력)** 구간에서는 유도형에도 이 신호를 그대로 쓴다(문제 유형 무관 예외 — 근거는 명세 '정답 판정' 절).
11-b. **[반영 2026-07-19: M1] 유도형 중립 넛지 — 로드맵(클라 구현 미포함).** 유도형은 (첫 시도 등 **따라 입력 이전** 구간에 한해) 근접 신호 대신 표기 점검을 권하는 중립 넛지("숫자·기호 표기를 다시 확인해보세요")를 쓸 수 있다(정답·개념 비노출, §D-10 일반 불일치와 같은 톤). 도메인 근거는 명세 '정답 판정' 절 참고. 이번 단계는 도메인 문구만 확정하고 화면 반영은 로드맵. **[반영 2026-07-19: D1]** 정답 공개 후 따라 입력 구간은 이 중립 넛지 대상이 아니다 — 그 구간은 11번의 근접 신호(유형 무관)를 쓴다.
12. **오답 교정 힌트(자동·push)** — 특정 **흔한 오답**을 냈을 때 **요청 없이 자동으로 뜨는 카드**. 예: "'프로세스'는 …라 이 문제의 답과는 달라요. 다시 도전!" **정답 비노출, 오답 처리 아님**.

## E. 힌트 (pull · 약→강 · 문제별 자유 구성)
13. **[힌트 보기] 진입점** — secondary. **사용자가 누를 때만**. (힌트가 없는 문제면 노출 안 함)
14. **단계적 힌트 스텝** — 눌러서 **약→강**으로 하나씩 공개. ⚠️ **고정 L1/L2 사다리 아님(리뷰 반영)** — 힌트 개수·종류는 문제마다 다르다. 각 힌트는 '학습적'이어야 하고(풀이용 단순 실마리 지양), "더 보기"로 다음 것, **이미 연 힌트는 계속 보임**.
15. **[더 쉬운 문제로 풀어보기] = 디딤 문제 진입** — 선행 개념(배경지식) 힌트 안의 진입점. 누르면 파고들기(이 화면 재사용 + A-2 맥락 배지)로 내려감. **MVP 제외 — 로드맵에서 디딤 도입 시 HintCard에 진입 버튼·라벨을 재도입(위 §1 결정).**

## F. 정답 공개 + 심화 정보
16. **[정답 보기] 진입점** — **사용자 명시 요청 시에만**. 최후의 도움. 누르면 **모범답안 + 짧은 해설**. (도움 신호로 내부 복습에 반영되지만 화면에 벌점처럼 보이게 하지 말 것.)
17. **정답 직접 입력(리뷰 반영)** — ⚠️ 정답 공개 후에는 **정답을 직접 입력**해야 다음으로 넘어간다(입력이 허용답 집합과 일치하면 통과 — 화면의 대표 정답 문자를 그대로 베낄 필요는 없다. 눈으로만 보고 넘기지 않게. **[반영 2026-07-19: C6]**). '벌'이 아니라 '손으로 써 보며 익히기'로 느껴지게 안내(예: "정답을 다시 적어 볼까요?"). 이 서비스가 진행을 요구하는 유일한 지점. **[반영 2026-07-19: D1]** 이 구간(따라 입력)에서 불일치가 오탈자 수준으로 가까우면 **문제 유형과 무관하게**(유도형 포함) 11번 근접 신호를 보여준다 — 이미 공개된 정답을 옮겨 적다 낸 오타에 막히지 않게 하기 위함.
18. **[더 알아보기] = 심화 정보** — 정답 처리(맞힘 또는 정답 공개) 후 그 개념의 흥미로운 추가 정보. **[결정 2026-07-16] 별도 조작 없이 바로 보인다**(토글 아님 — 확인하려 매번 한 번 더 누르는 마찰 제거). 없으면 표시 안 함. 진행을 막지 않음.

## G. 보조·엣지
19. **스크랩 토글(리뷰 반영)** — 이 문제를 개인 북마크에 저장/해제(별·북마크 아이콘, secondary). 스크랩 목록에서 나중에 재열람.
20. **지난 문제 다시 보기(리뷰 반영)** — 세션 내 이미 지나온 본 문제로 되돌아가 읽기 전용 확인(문제·내 답·모범답안) 후 현재로 복귀. 상단 진행 인디케이터/뒤로에서 진입.
21. **오류 신고 진입점** — 오버플로우(⋯) 등 **눈에 띄지 않되 접근 가능** → 07 신고 모달.
22. **로딩** — 문제 레이아웃 **스켈레톤**(스피너 지양).
23. **접근성** — 글자 확대 대응, 대체텍스트, 포커스 순서(문제 → 입력 → 제출 → 힌트).

# 인터랙션 흐름
1. 스켈레톤 → 문제. 입력 필드 자동 포커스.
2. 답 제출 → **일치**: 또렷한 축하 → (있으면)더 알아보기 바로 노출 → 다음 문제 / **불일치**: 또렷하지만 비처벌 넛지(낙인 X) / **근접(오탈자)**: "거의 맞았어요" 안내.
3. 흔한 오답이면 **오답 교정 힌트 카드** 자동 표시(정답 비노출).
4. 막히면 **[힌트 보기]** → 약→강 펼침(문제별 구성). (선행 개념 힌트에서 **[더 쉬운 문제로]** → 디딤 문제(깊이 전환) → 풀고 **원래 문제로 복귀**하는 흐름은 MVP 제외 — 로드맵에서 재도입.)
5. 끝내 안 되면 **[정답 보기]** → 모범답안 → **정답 직접 입력**(허용답 집합과 일치하면 통과, 불일치가 오탈자 수준이면 **[반영 2026-07-19: D1]** 유형 무관 근접 신호) → (있으면)더 알아보기 바로 노출 → 다음 문제.
6. 세션 마지막 본 문제를 끝내면 **04 세션 완료**로.

# 정답 상태 (리뷰 반영, 2026-07-16 / 갱신 2026-07-16 / 시안 충실화 2026-07-16)
> 오너 추가 시안 `03 문제 풀이 화면 디자인 (정답 맞췄을 때).html`을 **참고**해 확정. 시안 중 **디딤 맥락 배지·XP 문구·공유 버튼은 도메인 밖으로 제외**(디딤은 로드맵, XP·공유는 미도입 결정), **영문 라벨은 한국어로 치환**(난이도 점 마커·기존 진행 표기·"더 알아보기"). **[결정 2026-07-16]** 심화 정보는 시안처럼 **정답 시점에 바로 노출**한다 — §F-18이 토글식 "선택적으로 펼친다"에서 "별도 조작 없이 바로 보인다"로 바뀌었고, 시안의 펼쳐진 모습이 곧 실제 기본 상태다.
>
> **[결정 2026-07-16, 시안 충실화]** 직전 구현이 시안(52~121행)과 시각적으로 어긋난다는 오너 피드백을 반영해 세 가지를 다시 맞춘다: ①입력란 확정 전환의 시각 강도(자리 이동 없는 타이포 일치 + 확인 라인 + 절제된 체크 배지 등장), ②더 알아보기를 시안 구조(제목·리드·항목 카드·인용)로 렌더, ③허용답 나열 대신 **대표 정답** 하나만 표시(§C-7 이하 참고). 아래는 이 갱신을 반영한 최신 사양이다.

- **대표 정답 표시** — 정답 공개·지난 문제·스크랩 재열람·정답 확정 입력란 어디에서도 허용답을 나열하지 않는다. 문제마다 지정된 **대표 정답 하나**(예: "인덱스 (index)"처럼 한/영 병기가 자연스러운 문제는 병기)만 보여준다. 확정 입력란의 표시 내용은 **사용자가 실제로 제출한 문자열이 아니라 이 대표 정답**이다 — 여러 표현이 정답으로 인정돼도 화면엔 항상 같은 표기가 뜬다.
- **입력란의 확정 전환** — 정답을 맞히면 입력 가능한 칸이 사라지고, [AnswerTextField]와 **같은 높이·타이포·패딩**을 유지한 채(자리 이동감 없이) **success 톤(테두리·배경)의 확정된 기록**으로 바뀐다. 더 이상 편집 대상이 아니므로(시안도 `<input>`이 아니라 `<div>`로 그린다) 엔터로 인한 재제출 경로도 함께 사라진다. 우측 원형 체크 배지는 시안의 무한 bounce보다 절제된 **한 번의 스케일 인**으로 등장한다.
- **확인 라인** — 확정 입력란 바로 아래, 체크 글리프 + "완벽해요! 정확한 정답입니다."(success 톤, XP 없음)가 함께 뜬다. 이 한 줄이 예전 "맞았어요!" 카드의 확인 역할을 대신하므로, 그 카드는 이 상태에서 더 이상 그리지 않는다(중복 제거).
- **문제 영역의 감쇠** — 질문·난이도·코드 스니펫은 정답 확정 후 시각적으로 한 발 물러난다(옅어짐). 스크린리더 낭독은 그대로 유지한다(시각 강도만 낮추고 정보는 지우지 않는다).
- **정보 위계** — 확정된 입력 + 확인 라인 → 개념 칩 → 해설(카드 없이 본문 텍스트 — 확인 라인이 이미 카드 역할을 하므로 그 아래는 시안처럼 flat하게 흐른다) → [더 알아보기](**[결정 2026-07-16] 바로 노출, 토글 없음**, §F-18, 시안 구조 렌더) 순서. 해설은 정답 시 반드시 노출된다(명세). 힌트 진입점([힌트 보기])은 정답 후에는 더 이상 의미가 없으므로 감춘다.
- **더 알아보기 구조** — 자유 텍스트 한 덩어리가 아니라 시안 구조(71~114행)로 렌더한다: 섹션 라벨 "더 알아보기" + 가는 구분선 → 본 카드(제목 행 + 리드 문단 + 항목 서브카드들: 각 제목+설명, 테두리 카드) → 인용 카드(있으면 — 인용 표식 + 이탤릭 톤). 항목이 0개거나 인용이 없어도 본 카드는 자연스럽게 동작한다(부분 구조 허용). enrichment가 없으면 섹션 자체가 없다.
- **하단 CTA** — "다음 문제로"가 기본 Primary다. **[결정 2026-07-16] 세션의 마지막 본 문제를 맞혔을 때만 예외**로, 04 완료 화면으로 곧장 넘기지 않고 이 피드백 화면(개념·해설·심화)을 그대로 유지한 채 CTA가 "한입 마치기"로 바뀐다 — 마지막 문제라는 이유로 해설을 못 보고 넘어가는 것을 막기 위해서다. 눌러야 완료 화면으로 전환된다(아래 "완료 전환" 절 참고). 시안의 "해설 다시보기"는 **채택하지 않음** — 이 화면은 정답 후에도 피드백(개념·해설)이 화면에 계속 남아 있어(토스트로 사라지지 않음, §D-10) 다시 불러올 필요가 없다. 별도 버튼을 두면 이미 보이는 것을 또 보여주는 중복 진입점이 된다(과설계 회피).
- **은은한 축하 연출** — 시안의 파티클 오버레이 대신, 확정된 입력·확인 라인 블록이 짧게 페이드·스케일 인 되는 정도로 절제한다. 04 세션 완료(컨페티)보다 약해야 한다(위계: 문제 정답 < 세션 완료).
- **재열람과의 분리** — 이 확인 라인·체크 배지 등장 연출은 03 세션 풀이(정답 확정)에만 있다. **[반영 2026-07-19: D6]** 추가 학습이 폐지되어 '세션 밖 추가 연습' 경로 자체가 사라졌으므로(앱 레인에서 관련 화면·컴포넌트 정리), 남은 예외는 **스크랩 재열람**뿐이다 — 스크랩 재열람은 "확인" 맥락이지 "축하" 맥락이 아니므로, 기존 "맞았어요!" 카드(CorrectFeedback)를 그대로 쓰고 확인 라인·배지 연출은 새어 들어가지 않는다.

# 완료 전환 (결정 2026-07-16)
- 세션의 **마지막 본 문제**를 맞혀도 04 완료 화면으로 즉시 넘어가지 않는다. 위 "정답 상태"의 피드백(확정된 입력·축하·개념 칩·해설·더 알아보기)을 그대로 보여주고, 하단 Primary CTA만 "다음 문제로" 대신 **"한입 마치기"**로 바뀐다.
- **목적** — 마지막 문제라는 이유만으로 해설·심화 정보를 볼 기회를 잃지 않게 한다(다른 문제와 동등한 피드백 경험).
- "한입 마치기"를 누르면 그제서야 04 세션 완료 화면으로 전환된다. 완료 화면이 받는 요약(푼 문제 수·스트릭)은 이 마지막 제출 응답에 이미 실려 있으므로, 전환을 미루는 동안에도 유실 없이 보존된다.
- 서버 쪽 완료 처리(세션 상태 전이)는 마지막 제출 시점에 이미 끝나 있다 — 이 지연은 순수하게 **클라이언트 화면 전환 타이밍**이다.
