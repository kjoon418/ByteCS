**전제:** `00 공통 디자인 원칙.md`. 이 문서는 이 요소(모달/플로우)의 목적·구성만 기술한다.

# 1. 어떤 페이지인가
- **콘텐츠 오류 신고** — 서빙된 문제·힌트 등에서 오류를 발견한 사용자가 신고하는 **모달(또는 짧은 플로우)**. 독립 페이지가 아니라 03 문제 풀이에서 진입한다.
- 도메인 근거(콘텐츠 신뢰성 가드레일): 검수 게이트를 통과한 콘텐츠도 오류가 있을 수 있어, 사용자 신고 → **큐레이터가 수정·회수**한다. 게이트를 뚫은 오류의 안전망.
- 목적: 학습을 끊지 않으면서(가볍게) 오류를 알리게 한다.

# 2. 이 요소의 테마 강조점
- **가볍고 빠름.** 학습 흐름을 방해하지 않는 최소 입력. 바텀시트/모달로 인라인 처리.
- **감사·안심 톤.** "알려줘서 고마워요, 확인할게요." 신고가 곧 기여임을 긍정적으로.

# 3. 구성 요소 (구체)
1. **진입점** — 03의 오버플로우(⋯) 등에서 "이 문제에 오류가 있어요". 눈에 띄지 않되 접근 가능.
2. **신고 유형 선택(간단)** — 예: 정답이 틀림 / 문제 설명 오류 / 힌트 오류 / 기타. **단일 선택형**으로 인지 부하 최소(UX 1).
3. **자유 서술(선택)** — 짧은 설명 입력(선택 사항, 강제 X).
4. **Primary CTA — [신고 보내기]** — 동사형. 전송은 백그라운드(학습 흐름 방해 X).
5. **완료 피드백** — "알려줘서 고마워요! 확인 후 반영할게요." (감사·안심, 토스트 가능하나 충분히 읽히게)
6. **막다른 길 방지** — 신고 후에도 사용자는 **바로 문제 풀이로 복귀**(닫기). 신고했다고 해당 문제가 즉시 사라지거나 학습이 막히지 않음.

# 인터랙션 흐름
1. 03에서 오류 발견 → 진입점 → 이 모달.
2. 유형 선택(+선택 서술) → **[신고 보내기]** → 감사 피드백 → 03으로 복귀.

# 이 요소 주의
- ⛔ 신고를 무겁게(로그인 강제·긴 폼) 만들지 말 것.
- ⛔ "신고 처리 중"으로 학습을 막지 말 것(비동기·백그라운드).
- 신고 누적 자동 숨김 등은 도메인 열린 질문 → 이 시안에서 다루지 않음.
