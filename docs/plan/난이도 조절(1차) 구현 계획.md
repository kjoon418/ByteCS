# 난이도 조절 1차 구현 계획 — 선호 난이도 선택 + 가중 출제

> **문서 성격.** 테스터 피드백 "내 수준에 맞게 난이도를 조절할 수 있으면 좋겠다"를 수용하기 위한 1차 구현 계획. 오너와의 결정(2026-07-21, 아래 기록)을 전제로 하며, **새 세션이 이 문서만으로 착수 가능**하도록 작성한다.
>
> 도메인 근거: 명세서 기능 1 '문제 형태' — "문제는 난이도를 가진다. MVP에서는 표시 속성이며, 로드맵의 **개인 수준 맞춤 적응 출제**가 이를 소비하는 지점이므로 일급 속성으로 존치한다." 이번 작업은 그 소비를 시작하는 **로드맵 앞당김**이지 새 축의 발명이 아니다.

---

## 1. 배경과 목적

- 프로토타입 배포 후 가장 많이 나온 테스터 피드백이 **난이도 조절**이었고, 오너가 수용을 결정했다.
- 현재 난이도는 `Difficulty`(EASY/보통 MEDIUM/HARD) 3단계 **표시 전용** 속성이다. 세션의 새 개념 문제는 안 푼 문제 풀에서 **난이도 무관 균등 무작위**로 배정된다(기능 1.5).
- 콘텐츠 분포(시드 107문제): 쉬움 24 / 보통 58 / 어려움 25. 카테고리·개념 단위로 쪼개면 특정 난이도가 비는 조합이 흔하므로 **폴백이 필수**다.
- 선례: **관심 카테고리 2차 정책**(기능 7 — 가중치로만 반영 · 완전 배제 없음(소량 유지) · 복습 편입 무관)이 이번 요구와 동형이다. 그 도메인 원칙을 그대로 재사용한다.

## 2. 오너 결정 기록 (2026-07-21)

## 3. 도메인 규칙 (확정 — 명세서 반영 대상)

### 3.1 선호 난이도 (Preferred Difficulty)
- **사용자 소유 설정**이다(게스트 포함 — `UserSettings`의 세션 분량과 동일한 소유 모델). 언제든 설정 화면에서 변경할 수 있다.
- 값은 3단계(쉬움/보통/어려움) 중 하나이며, **미설정 상태가 존재한다**. 미설정이면 현행 균등 무작위 배정이 그대로 유지된다(강제 없음 — 무낙인·저부담 정합).
- 사용자당 **전역 1개**다. 카테고리별 선호 난이도(분야별 실력 차 반영)는 콘텐츠 분포가 받쳐줄 때 재검토한다(로드맵).

### 3.2 가중 출제
- 선호 난이도는 **새 개념 문제 배정에만 가중치**로 반영된다.
  - **복습 편입은 난이도와 무관하게 유지**된다(정착 보호 — 관심 카테고리 정책과 동일 근거).
  - D8 **반복 폴백**(안 푼 문제 소진 시 이미 푼 문제 재출제)도 난이도 가중의 영향을 받지 않는다.
- **완전 배제 없음.** 어떤 난이도도 0이 되지 않는다 — 선호에 가까울수록 더 자주, 멀수록 덜, 그러나 모든 난이도가 소량은 계속 나온다(오너 원안 "골고루 받되 선호에 가까운 문제가 더 많이"와 일치, 편식·콘텐츠 고갈 방지).
- **결정적 가중 규칙.** 판정 원칙과 정합하게 규칙 자체는 결정적이다(LLM 없음). 무작위 추첨은 기존 '새 개념 무작위 배정'이 이미 허용한 축이므로, 균등 추첨을 **가중 추첨**으로 바꾸는 것뿐이다.
- **가중치 수치는 운영 판단**으로 분리한다(관심 카테고리 선례와 동일 — 명세에는 원칙만). 구현 기본값은 **거리 기반 6:3:1** — 선호와의 거리 d=0 → 6, d=1 → 3, d=2 → 1. 예: 쉬움 선호 시 쉬움 6 : 보통 3 : 어려움 1(60/30/10), 보통 선호 시 6:3:3(50/25/25). 코드에서 상수로 분리해 튜닝 가능하게 둔다.
- **폴백(콘텐츠 희박 대응).** 가중 추첨은 **후보(안 푼 승인 문제)가 존재하는 난이도**에만 적용한다 — 특정 난이도 풀이 비면 그 난이도의 가중치가 자연히 0이 되고 남은 난이도끼리 배정된다. 전 난이도 소진 시에는 기존 D8 반복 폴백이 그대로 동작한다.

### 3.3 첫 세션 완료 후 제안 (DF1)
- 노출 조건: **선호 난이도 미설정 && 제안에 아직 응답하지 않음** && 세션 완료 화면 도달.
- 선택지: 상태 서술형 3택(§4.4) + **"지금은 괜찮아요"**(거절). 어느 쪽이든 **1회 응답하면 다시 묻지 않는다**(잔소리 금지 — 긍정 동기 원칙). 거절 시 "설정에서 언제든 바꿀 수 있어요"를 은은히 안내한다.
- 제안 카드는 완결 축하를 가리지 않는다(성취 맥락 보존 — 디자인 04의 위계 유지).

### 3.4 2차 예고 — 적응 조정 (구현하지 않음, 방식만 확정)
- 신호: 이미 결정적으로 기록되는 '스스로 맞힘 / 도움받아 맞힘 / 정답 공개(포기)' 구분을 재사용한다. 새 수집 인프라 불필요.
- 규칙 방향: 최근 K개 **새 개념 본 문제**(복습·반복 폴백 제외 — 복습은 정착 검증이지 수준 신호가 아님) 기준, 스스로 맞힘 비율이 임계 이상이면 상향 제안, 정답 공개·포기 비율이 임계 이상이면 하향 제안. 평가 시점은 세션 완료, 제안 후 쿨다운.
- **제안+승인(DF2).** 서비스는 제안만 하고 사용자가 수락해야 반영된다. 하향 제안 문구는 낙인 없이("기본기를 더 단단히 다져볼까요?") — 톤은 2차 디자인에서 확정.
- K·임계값·쿨다운은 1차 운영 데이터(§5 Stage 6 지표)로 검증 후 확정한다 — 명세서 [열린 질문]에 등재.

## 4. 시스템 변경 설계

### 4.1 DB — Flyway `V7__preferred_difficulty.sql`
```sql
ALTER TABLE users ADD COLUMN preferred_difficulty VARCHAR(16) NULL;
ALTER TABLE users ADD COLUMN difficulty_prompt_done BOOLEAN NOT NULL DEFAULT FALSE;
```
- `preferred_difficulty` NULL = 미설정(균등 배정 유지). 값은 `Difficulty` enum 이름.
- `difficulty_prompt_done` = 완료 화면 제안에 응답(선택 또는 거절)했는지. 설정 화면에서 선호를 직접 지정한 경우에도 TRUE로 올린다(이미 아는 사용자에게 제안 불필요).

### 4.2 서버
- **`UserSettings` 확장** (`account/domain/UserSettings.kt`): `preferredDifficulty: Difficulty?` 추가. (`difficultyPromptDone`은 설정이라기보다 노출 상태이므로 `User` 직속 필드로 두는 것을 권장 — 구현 시 판단.)
- **`PATCH /me/settings` 확장** (`UserController`): `preferredDifficulty` 필드 추가(설정 시 promptDone도 TRUE). 거절 기록용으로 같은 PATCH에 `difficultyPromptDone: true`만 보내는 형태를 허용한다(별도 엔드포인트 불요).
- **조회**: 기존 내 정보/설정 응답에 `preferredDifficulty` 포함. 세션 완료 응답(또는 완료 상태 조회)에 `needsDifficultyPrompt: Boolean` 포함 — 클라이언트가 노출 조건을 재계산하지 않게 서버가 단일 출처로 판단한다.
- **가중 배정** (session 도메인의 새 개념 문제 선정부): 균등 무작위 → 선호 설정 시 **난이도 가중 무작위**(§3.2 규칙·폴백). 선호 미설정이면 기존 경로 그대로(회귀 없음).

### 4.3 클라이언트 (Compose Multiplatform — android/wasmJs 공통)
- **설정 화면(디자인 06)**: '선호 난이도' 항목 추가 — 상태 서술형 3택 + '자동(미설정)'. 현재 값 표시.
- **세션 완료 화면(디자인 04)**: `needsDifficultyPrompt`일 때 제안 카드 — 상태 서술형 3택 + "지금은 괜찮아요". 응답 즉시 PATCH, 카드 닫힘. 완결 축하 아래 배치(위계 침범 금지).
- **온보딩(디자인 01): 변경 없음** (DF1).

### 4.4 문구 (무낙인 — 상태 서술형)
| 내부 값 | 선택지 문구(안) |
|---------|----------------|
| EASY | "CS를 이제 막 시작해요" |
| MEDIUM | "기본기를 다지는 중이에요" |
| HARD | "도전적인 문제를 원해요" |
| (미설정) | "자동으로 골고루 받을래요" (설정 화면 전용) |

제안 카드 리드 문구(안): "문제 난이도를 OO님께 맞춰볼까요?" — 수준 평가 뉘앙스("실력에 맞는") 배제, 조절 주체가 사용자임을 드러낸다. 최종 문구는 구현 시 UX 가이드(긍정·대화형 톤) 기준으로 다듬는다.

## 5. 구현 단계와 커밋 계획

각 커밋은 서비스 온전 동작 + 테스트 통과를 전제로 한다(버전 관리 지침).

| Stage | 내용 | 커밋(안) |
|-------|------|---------|
| 0 | 명세서·디자인 문서 갱신(§6) + 본 계획 문서 | `docs: 난이도 조절 1차(선호 난이도·가중 출제) 명세 반영` |
| 1 | V7 마이그레이션 + UserSettings/User 확장 + PATCH·조회 API + 테스트 | `feat: 선호 난이도 설정 저장·변경 API` |
| 2 | 가중 배정 + 폴백 + 회귀 테스트(복습 무관·미설정 경로 무회귀 포함) | `feat: 선호 난이도 가중 출제` |
| 3 | 세션 완료 응답 `needsDifficultyPrompt` + 테스트 | Stage 1 또는 2에 병합 가능 |
| 4 | 클라이언트 설정 화면 항목 | `feat: 설정 화면 선호 난이도 선택` |
| 5 | 클라이언트 완료 화면 제안 카드 | `feat: 세션 완료 화면 난이도 제안 카드` |
| 6 | `/admin/stats` 난이도 지표(선호 설정 분포 · 난이도별 정답 공개율) — 2차 임계값 검증 근거 | `feat: 관리자 지표에 난이도 통계 추가` (스트레치 — 1차 출시 후순위 가능) |

### 테스트 관점 (필수 커버)
- 가중 추첨: 선호별 가중치 적용(통계적 단정이 아니라 **가중치 계산·후보 필터링의 결정 로직**을 단위 테스트로), 난이도 풀 소진 시 폴백, 전 소진 시 D8 반복 폴백 유지.
- 회귀: 선호 미설정 사용자의 배정이 기존과 동일 경로, **복습 편입이 선호와 무관**함.
- 설정 API: 값 검증, 게스트 동작, promptDone 전이(직접 설정 시 포함), 가입 승계 시 보존.
- needsDifficultyPrompt: 미설정·미응답에서만 true, 응답 후 false.

## 6. 명세서·문서 갱신 목록 (Stage 0)

- **도메인 명세서**
  - 도메인 용어: **선호 난이도(Preferred Difficulty)** 등록.
  - 기능 1 문제 형태: 난이도 "MVP에서는 표시 속성" 서술 갱신 — 선호 가중 출제가 소비 시작 [반영 2026-07-21: DF1~DF4].
  - 기능 1.5 문제 선정: 새 개념 무작위 배정에 "선호 난이도 설정 시 가중 무작위(완전 배제 없음·폴백·복습 무관)" 추가.
  - 기능 6 또는 1.5: 첫 세션 완료 후 1회 제안(DF1) 기록.
  - 로드맵: '개인 수준 맞춤 적응 출제' 항목 분해 — 1차(선호 가중)는 [구현], 적응 조정은 로드맵 유지 + **제안+승인 방식 확정(DF2)** 명기.
  - 열린 질문: 가중치 수치 튜닝 · 적응 조정 K/임계/쿨다운 · 카테고리별 선호 난이도 추가.
- **디자인 문서**: 04(완료 화면) 제안 카드, 06(설정) 항목 추가. 01(온보딩)은 "난이도 선택을 두지 않기로 결정(DF1)" 각주만.

## 7. 비범위 (1차에서 하지 않는 것)

- 적응 조정 실행(2차 — DF2·DF3), 카테고리별 선호 난이도, 숙련도 기반 문제 개인화, 온보딩 변경, 난이도 재큐레이션(콘텐츠 난이도 값 자체의 재검수).
