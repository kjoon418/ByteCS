# 문제 콘텐츠 파이프라인 구현 계획 (관리자 등록·LLM 초안·검수)

이 문서는 "신뢰도 있는 문제 품질을 유지하면서 다량의 문제를 지속적으로 축적하는 파이프라인"의 구현 계획이다.
**독자는 이 계획을 실행할 새 AI 세션**이다. 이 문서와 `docs/` 하위 지침만으로 전체 작업을 수행할 수 있어야 한다.

> **문서 상태**: 오너와의 합의로 확정된 계획 (2026-07-18). 아래 '확정된 결정' 절의 항목은 재논의 없이 그대로 구현한다.
> 구현 중 이 계획과 기존 지침·명세가 충돌하면, 작업을 멈추고 오너에게 되묻는다(CLAUDE.md 반문 규칙).

---

## 1. 목적과 배경

- "매일 새로운 학습 경험"이 서비스 핵심 메리트이므로, 문제를 지속적으로 공급하는 파이프라인이 필요하다.
- 도메인 명세서는 이미 콘텐츠 승인 파이프라인을 [결정]으로 확정해 두었다(가드레일 §콘텐츠 신뢰성): **AI는 초안 제안자, 전량 사람(큐레이터) 검수 게이트, 승인 상태(초안→검수중→승인/반려), 결정적 구조 검증, 사용자 신고 → 수정·회수.** 현재 구현만 "시딩분을 승인 취급"하는 MVP 상태다. 이번 작업은 그 명세를 실체화한다.
- LLM API 종량 비용을 피하기 위해, **관리자가 구독 중인 LLM(ChatGPT/Claude 등) 대화창을 사용하는 프롬프트 팩 방식**을 기본으로 한다.

### 현재 상태 (as-is, 2026-07-18 기준 사실)

| 항목 | 현재 상태 |

## 2. 확정된 결정 (오너 결정 2026-07-18)

1. **관리자 화면: 서버 렌더링 웹.** Spring Boot 안에서 Thymeleaf(+필요 시 소량 JS)로 `/admin/**` 페이지를 렌더링한다. 별도 프론트 빌드·배포 없음. Compose Wasm 웹은 사용자용 웹의 로드맵으로 남긴다.
2. **권한 모델: ADMIN 단일.** `UserRole`에 `ADMIN`만 추가한다. 초기엔 오너가 관리자=큐레이터를 겸한다. CURATOR 분리(초안 등록·검수 권한 분리, 큐레이터 계정 관리)는 로드맵. 단, 권한 체크는 역할 추가만으로 확장되게 설계한다.
3. **초안 반입: 프롬프트 팩 + JSON 붙여넣기.** 관리자 페이지가 기존 콘텐츠 컨텍스트를 포함한 생성 프롬프트를 만들어 주고, 관리자가 구독 LLM 대화창에서 문제 JSON을 생성해 붙여넣는다. LLM API 직접 호출은 하지 않는다(반입 경계를 인터페이스로 두어 후일 API 모드 추가 가능하게).
4. **중복 방지: 이중 장치.** ① 생성 프롬프트에 기존 개념 풀·문제 요지·대표정답 목록을 주입해 생성 단계에서 회피, ② 등록 시 PostgreSQL `pg_trgm` 기반 결정적 유사도 검사로 '중복 의심' 경고를 검수 화면에 표시(차단이 아니라 경고 — 최종 판단은 큐레이터). 임베딩(pgvector) 유사도는 로드맵.
5. **1차 범위에 운영 기능 4종 포함:** 신고 처리 콘솔 · 콘텐츠 수정·회수 · 개념·카테고리 관리 · 품질 대시보드(거짓 오답 후보).
6. **Flyway 도입.** 운영 DB에 관리자가 축적하는 콘텐츠는 재생성 불가능한 자산이 되므로, 이 작업에서 `ddl-auto: update`를 버리고 Flyway 마이그레이션 + `ddl-auto: validate`로 전환한다.
7. **기존 107문제는 폐기하지 않고 '초안'으로 이관.** 시드 로더는 local·test 프로파일 전용으로 강등하고, 운영에서는 반입 화면으로 107문제를 초안 등록 → 검수해 승인분만 서빙한다("검수 전 투입 금지" 유지).

## 3. 목표 워크플로 (큐레이터 관점 시나리오)

1. 관리자가 `/admin`에 로그인한다(폼 로그인, ADMIN 전용).
2. **[생성]** '초안 반입' 화면에서 생성 조건(문제 수, 카테고리·개념 편중, 유형 분포)을 고르면 **생성 프롬프트**가 만들어진다. [복사] 버튼으로 복사해 구독 중인 LLM 대화창에 붙여넣고, LLM이 낸 **문제 JSON**을 받아 온다.
3. **[반입]** 그 JSON을 반입 화면에 붙여넣으면 서버가 파싱 → 도메인 불변식 검증 → no-leak 검사 → 중복 검사를 수행하고 **반입 리포트**(성공 n건 초안 저장 / 실패 m건 사유 / 중복 의심 k건)를 보여준다. 유효한 문제만 '초안' 상태로 저장된다.
4. **[검수]** '검수 큐'에서 문제를 하나씩 열어 내용을 확인·수정하고 **승인** 또는 **반려**(사유 기록)한다. 중복 의심 경고가 붙은 문제는 기존 문제와 나란히 비교해 판단한다.
5. **[서빙]** 승인된 문제만 사용자 세션·추가 학습·복습 선정에 편입된다.
6. **[운영]** 신고가 들어오면 신고 콘솔에서 확인해 수정하거나 회수한다. 대시보드에서 정답률·상위 오답을 보고 허용답을 보강한다. 개념 관리에서 오타·중복 개념을 병합하고 카테고리를 정리한다.

관리자 개입이 필요한 일은 "프롬프트 복사 → LLM 대화 → JSON 붙여넣기 → 문제별 승인 클릭"뿐이다(요구사항 6: 최소 노력).

## 4. 도메인 설계

### 4.1 승인 상태 (ApprovalStatus)

`Problem` 애그리거트에 승인 상태를 도입한다. **승인 단위는 문제 애그리거트**다 — 힌트·오답 교정 힌트·허용답·심화 정보는 문제에 소유되므로 함께 승인된다. 개념(Concept)은 별도 승인을 두지 않는다(승인된 문제를 통해서만 사용자에게 드러나므로).

```
DRAFT(초안) → IN_REVIEW(검수중) → APPROVED(승인) | REJECTED(반려)
REJECTED → IN_REVIEW            (수정 후 재검수)
APPROVED → RETRACTED(회수)      (승인 해제 → 서빙 중단)
RETRACTED → IN_REVIEW           (수정 후 재검수)
```

- 명세의 4상태(초안/검수중/승인/반려)에 **RETRACTED를 상태로 실체화**한다. 명세는 회수를 행위("승인 해제 → 서빙 중단")로 이미 확정했고, 반려(서빙된 적 없음)와 회수(서빙되다 중단)는 운영상 구별이 필요하다. → 명세서 갱신 대상(§8).
- **승인 전이 시 결정적 구조 검증을 강제**한다(명세 가드레일): 허용답 비어있지 않음 · 개념 태깅 1개 이상 · **문제 유형(type) 태깅 존재(null 금지)** · 힌트/오답 교정 힌트에 자기 정답 비노출(no-leak). 위반 시 승인 전이가 도메인 예외로 거부된다.
  - 주의: 현재 `Problem.type`은 nullable이다. 초안은 null을 허용하되, **승인 시점에는 non-null을 강제**한다(명세 수용 기준 23).
- **승인 상태에서의 직접 수정 허용.** 관리자=검수자이므로 수정=재승인으로 간주한다(허용답 보강 같은 경미 수정을 막지 않기 위함). CURATOR 분리 시 재검토. 수정 시에도 불변식·no-leak은 항상 검증된다.

### 4.2 서빙 게이트

사용자향 모든 조회·선정 경로에 `approval_status = APPROVED` 필터를 건다. 대상 경로를 전수 열거한다(구현 시 각 리포지토리 쿼리를 확인하고, 세션·추가학습·복습·스크랩·카테고리 이력·문제 단건 조회를 모두 커버하는 테스트를 둔다):

| 경로 | 규칙 |
|---|---|
| 세션 새 개념 문제 배정 | APPROVED만 후보 |
| 추가 학습 문제 선정 | APPROVED만 후보 |
| 복습 문제 선정(유도형 다른 문제 우선 포함) | APPROVED만 후보 |
| 문제 단건 조회(GET /api/problems/**) | 비승인 문제는 존재하지 않는 것처럼 응답(404 계열 — 초안 유출 차단, 명세 수용 기준 15) |
| 스크랩 상세 | **회수된 문제는 '더 이상 볼 수 없음' 처리**(명세 §5 결정). 서버는 회수 상태를 응답에 실어 주고, 클라이언트는 플레이스홀더를 보여준다(소규모 클라이언트 작업 포함) |
| 카테고리별 학습 이력 | 스크랩과 동일하게 회수 문제는 '볼 수 없음' 처리 |
| **진행 중 세션에 이미 배정된 문제** | **그대로 풀 수 있게 유지**한다. 회수는 신규 노출 중단이지 진행 중 세션 파괴가 아니다(세션 중단·재개 보장과 정합). → 명세 열린 질문 10의 부분 확정으로 기록(§8) |

### 4.3 반려 사유·감사 기록

- 상태 전이 이력(누가·언제·어느 상태로·반려/회수 사유 텍스트)을 별도 테이블로 기록한다(`content_approval_log` 등). 큐레이터가 늘었을 때의 감사 추적 및 반려 사유 재확인 용도. 화면은 문제 상세에서 이력 열람 정도로 최소화.

### 4.4 개념 병합의 결정적 규칙

개념은 이름 기준 find-or-create라 오타·표기 차이("스레드"/"쓰레드")로 중복이 생긴다. 병합 merge(source → target) 규칙:

1. `problem_concept`의 source 태깅을 제자리에서 target으로 치환한다(순서 보존 — 대표 개념 위치 유지). 치환 결과 한 문제에 target이 중복되면 **뒤쪽 항목을 제거**하고 인덱스를 압축한다.
2. 사용자 개념 숙련도에 두 레코드가 모두 있으면 **다음 복습 시점이 이른 쪽 + 숙련도가 낮은 쪽을 채택**한다(보수적 — 복습을 더 자주 하는 방향으로만 틀린다). 하나만 있으면 target으로 이전.
3. source 개념을 삭제한다.
4. source·target의 카테고리가 다르면 병합 전 경고를 표시한다(진행은 가능 — 큐레이터 판단).

### 4.5 오답 제출 집계 (대시보드 데이터)

- 새 테이블 `problem_wrong_answer_stat(problem_id, normalized_answer, count, last_submitted_at)`. 판정이 MISMATCH/NEAR_MISS일 때 정규화 답 기준 upsert(사용자 식별자 저장 안 함 — 집계만).
- 대시보드가 이것으로 문제별 시도 수·정답률·near-miss 비율·상위 오답 5개를 보여준다. 상위 오답 중 "사실상 정답인데 허용답에 없는 표기"(거짓 오답)를 큐레이터가 발견해 허용답을 보강한다 → 명세 [열린 질문] 2·3(허용답 완전성)을 데이터로 푸는 루프.

## 5. 관리자 웹 설계

### 5.1 인증·보안

- `UserRole.ADMIN` 추가. 관리자 계정은 자가 가입 불가 — 기동 시 환경변수(`BYTECS_ADMIN_EMAIL`, `BYTECS_ADMIN_INITIAL_PASSWORD`)가 설정돼 있고 ADMIN이 없으면 생성하는 부트스트랩 러너(BCrypt 저장, 기존 인코더 재사용).
- `/admin/**` 전용 `SecurityFilterChain`을 **별도로** 추가한다(@Order로 기존 API 체인보다 앞): 폼 로그인 + 세션(IF_REQUIRED) + **CSRF 활성** + `hasRole("ADMIN")`. 기존 JWT 무상태 API 체인은 손대지 않는다.
- 관리자 페이지는 Thymeleaf 템플릿(`server/src/main/resources/templates/admin/**`). 디자인 시스템 준수 의무 없음(내부 도구) — 깔끔한 기본 스타일이면 충분하다.

### 5.2 화면 목록

| # | 화면 | 핵심 요소 |
|---|---|---|
| A1 | 로그인 | 폼 로그인 |
| A2 | 대시보드(홈) | 상태별 문제 수, 카테고리별 승인 문제 수('준비 중' 카테고리 표시), 미처리 신고 수, 시도 수 상위·정답률 하위 문제, 거짓 오답 후보(상위 오답) 바로가기 |
| A3 | 문제 목록 | 필터(승인 상태·카테고리·유형·난이도·텍스트 검색), 페이지네이션 |
| A4 | 문제 상세·편집 | 전 필드 편집(지문·개념 순서·허용답·대표정답·유형·난이도·힌트·오답 교정·심화), 구조 검증 결과 인라인 표시, 중복 의심 표시(유사 문제 나란히 비교), 상태 전이 버튼(검수 시작/승인/반려/회수 — 사유 입력), 상태 이력, 오답 집계 표시 |
| A5 | 문제 수동 등록 | A4와 같은 폼의 신규 모드(LLM 없이 단건 작성) |
| A6 | 초안 반입 | 2단계. **Step1 프롬프트 생성**: 조건 입력(문제 수·카테고리/개념 편중·유형 분포) → 프롬프트 표시 + 복사 버튼. **Step2 JSON 반입**: 텍스트 영역에 붙여넣기 → 반입 리포트(성공/실패 사유/중복 의심) → 유효분 초안 일괄 저장 |
| A7 | 검수 큐 | 초안·검수중 문제 목록 → A4로 진입해 승인/반려. "다음 문제" 흐름으로 연속 검수 |
| A8 | 신고 콘솔 | 신고 목록(미처리 우선) → 해당 문제 A4 바로가기 → 수정 또는 회수 → 신고 처리 완료 표시 |
| A9 | 개념 관리 | 개념 목록(이름·카테고리·연결 문제 수·미분류 필터), 이름 변경, 병합(§4.4), 카테고리 지정 |

### 5.3 프롬프트 팩 구성

프롬프트 템플릿은 서버 리소스(`server/src/main/resources/prompt/problem-generation-v1.md`)로 두고 버전을 파일명에 명시한다(프롬프트 개선 이력 추적). 생성 시 서버가 플레이스홀더를 채운다:

1. **역할·목표**: CS한입의 문제 생성 어시스턴트. 출력은 JSON만(설명 문장 금지).
2. **JSON 스키마·필드 규칙**: `docs/dev/문제 데이터 생성 가이드.md` §2를 요약 이관(허용답 정규화 규칙, 대표 정답 불변식, 병기 표기 등재, no-leak, 힌트 학습적 가치·고정 사다리 금지, 유형 태깅 기준 — 모르면 null, 심화 전 문제 작성, conceptCategories 규칙). **반입 JSON 스키마는 기존 시드 스키마(`ProblemSeedDto`)와 동일하게 유지한다** — 검증 로직·테스트를 그대로 재사용하고, 107문제 이관도 같은 경로로 처리하기 위함.
3. **기존 콘텐츠 컨텍스트(중복 방지)**: 전체 개념 풀(이름·카테고리·연결 문제 수) + 기존 문제 요지 목록(대표 개념 · 대표정답 · 지문 앞 60자). "이 목록과 같은 것을 묻는 문제를 만들지 말 것, 기존 개념의 재사용(개념당 1~5문제)은 권장" 지시 — 개념 재사용과 문제 중복은 다르다는 점을 명시.
4. **생성 지시**: 요청 개수(권장 기본 20 — 검수 배치 단위), 카테고리·개념 편중, 유형 분포(기본 정의 재생형 70 : 유도형 30), 표기 다양성.

> 컨텍스트가 커지면(문제 수천 개 규모) 카테고리별 분할 생성으로 전환한다 — Step1에서 카테고리를 고르면 그 카테고리의 컨텍스트만 주입. 1차에서는 전체 주입으로 충분하다(107문제 ≈ 수천 토큰).

### 5.4 반입 파이프라인

```
JSON 텍스트
 → ① 파싱 (구문 오류: 위치 보고)
 → ② 문제별 도메인 조립 (init require 위반: 문제 인덱스·필드·사유 보고)
 → ③ no-leak 검사 (기존 로더 검증 재사용)
 → ④ 배치 내부 중복 검사 + DB 중복 검사 (pg_trgm 지문 유사도 ≥ 임계(초기 0.5, 상수로 조정 가능)
      또는 [동일 대표 개념 + 정규화 대표정답 동일] → '중복 의심' 마킹)
 → ⑤ 유효분만 초안 저장, 반입 리포트 렌더
```

- 부분 성공을 허용한다: 유효한 문제는 저장하고, 실패분은 사유와 함께 보고한다(관리자는 실패분만 LLM에서 수정해 재반입하거나 A4에서 수동 수정).
- 개념은 반입 시점에 find-or-create로 생성한다. 초안 단계의 개념 생성은 무해하다 — 개념은 승인된 문제를 통해서만 사용자에게 드러난다(§4.1). 미분류 개념은 A9에서 정리.
- ②·③은 현재 `ProblemDataLoader`에 있는 조립·검증 로직을 **application 계층 서비스로 추출해 공용화**한다(로더·반입·수동 등록이 같은 검증을 공유). 계층 배치는 `docs/dev/Spring 아키텍처 가이드.md`·`Spring 검증 로직 배치 가이드.md`를 따른다.

## 6. 인프라 변경

### 6.1 Flyway

- 의존성 추가(`flyway-core` + PostgreSQL 지원), `ddl-auto: validate`로 전환.
- `V1__baseline.sql`: 현재 엔티티에서 생성되는 스키마 전체를 SQL로 고정(로컬에서 `ddl-auto: create`로 뽑아 정리). 운영 DB가 아직 없으므로 baseline 마이그레이션 부담이 없다 — **지금이 도입 적기**.
- 기존 로컬·CI DB는 재생성 또는 `baselineOnMigrate` 사용.
- 이후 이 계획의 스키마 변경(approval_status, 로그 테이블, 오답 집계, pg_trgm)은 각각 V2, V3, … 마이그레이션으로 추가한다.
- `V*__enable_pg_trgm.sql`: `CREATE EXTENSION IF NOT EXISTS pg_trgm;` (+ 지문 컬럼 GIN 트라이그램 인덱스). 관리형 PG에서 확장 설치 권한을 확인할 것(체크리스트 §9).
- 세션 저장이 필요한 관리자 폼 로그인은 기본 인메모리 세션으로 충분(관리자 1인, 재로그인 부담 없음).

### 6.2 시드 로더 강등

- ✅ **(Phase 1 리뷰 반영으로 조기 적용됨)** `ProblemDataLoader`를 `@Profile("local")`(현행 `!test`에서 변경)로 좁혔다. 운영 프로파일에서는 어떤 자동 시딩도 없다. Phase 1이 로더 시딩분을 APPROVED로 스탬핑하면서, 프로파일 축소를 Phase 5까지 미루면 운영 첫 기동 시 미검수 107문제가 자동 서빙되는 위험이 실체화되어 앞당겼다(리뷰 Major).
- local 프로파일 시딩분은 로더에서 APPROVED로 저장한다(로컬 개발 편의 — 명세의 "MVP는 시딩분을 승인 취급" 유지).
- 107문제의 운영 반입은 A6 반입 화면에 `problems-generated.json` 내용을 붙여넣는 것으로 수행한다(스키마 동일 — §5.3). 초안으로 저장되므로 검수 게이트가 그대로 적용된다.

## 9. 운영 전환 체크리스트 (실서비스 배포 시)

1. 운영 PG에 Flyway 마이그레이션 적용(신규 DB이므로 V1부터). `pg_trgm` 확장 설치 권한 확인.

5. 기존 메모리의 운영 전 백필 SQL 항목(concept_id 드롭·representative_answer 백필·concept.category 백필)은 **V1 베이스라인에 흡수되는지 확인**(신규 DB면 자연 해소, 기존 데이터 이전이 생기면 별도 마이그레이션).

- **LLM API 자동 생성 모드**: 반입 경계(§5.4 파이프라인의 입력이 "JSON 텍스트"라는 사실)는 유지한 채, 프롬프트 팩 대신 서버가 API를 호출해 JSON을 받아 오는 어댑터 추가. 비용 통제(배치 상한) 포함.
- **임베딩 기반 의미 중복 검사**: pgvector + 임베딩 API. 표기가 달라도 의미가 같은 중복 탐지.
