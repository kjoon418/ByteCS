# 핵심 목적
핵심 목적은 기획 과정에서 개선될 수 있다.

**상위 목적 (존재 이유)**
1. 취업·실무를 준비하는 개발자(지망생)가 **CS 공부를 시작하고 지속하는 부담**을 줄인다.

**수단 목적 (상위 목적을 실현하는 방법 — 위계순)**
1. **짧은 문제 풀이로 능동 학습을 만든다.** 사용자는 읽고 흘리는 수동 학습이 아니라, 5분 안에 직접 문제를 풀며 지식을 기억에 남긴다. (히어로 — 핵심 활동)
2. **막혔을 때 포기 대신 넘게 한다.** 문제가 어렵거나 배경지식이 없어도, 단계적 힌트로 이탈하지 않고 문제와 씨름하게 한다. 그래도 막히면 무낙인으로 정답을 열어 따라 적으며 익힌다. ('더 쉬운 문제로 이어지는 길'=디딤 문제는 로드맵) (히어로 보조 — 진입 장벽 제거)
3. **푼 것을 복습·정착시키고, 다시 오게 만든다.** 한 번 풀고 잊는 게 아니라 약한 지식을 다시 만나 정착시키고, 완결 성취·스트릭 같은 **긍정 동기**로 매일 가볍게 돌아오게 한다(강제·죄책감이 아니라 성취감으로). (지속·정착)
4. **큐레이션과 AI로 신뢰할 수 있고 확장 가능한 학습 콘텐츠를 공급한다.** (콘텐츠 공급 기반)

> 위계 근거: 목적 1(공부 부담 제거)이 서비스의 존재 이유다. 수단 목적 1(문제 풀이)이 사용자가 처음 체감하는 히어로 활동이고, 수단 목적 2(힌트 — 디딤 문제는 로드맵)가 이 히어로를 받쳐 '포기'를 막는다. 수단 목적 3(복습·정착)은 히어로 활동을 오래 가는 가치로 잇고, 수단 목적 4(콘텐츠 공급)는 그 아래에서 전체를 떠받친다. 수단 목적 2·3의 세부 위계는 [논의 예정].

# 타깃 사용자 페인포인트
> 이 절은 기획자 추정에 근거한다. 실제 사용자 리서치로 검증·갱신이 필요하다. [논의 예정 — 리서치 검증]

취업·실무를 위해 CS를 공부하려는 개발자가 반복적으로 겪는 문제를 정리한다. **주 타깃에는 공부 의지가 높지 않은 층이 포함되므로**, 시작 부담 제거(P1)만큼 '다시 오게 하는' 지속 동기(P5)도 중요하다. 각 페인포인트가 어느 기능/가드레일로 해소되는지 함께 매핑한다.

| # | 페인포인트 | 해소 수단 |
|---|-----------|----------|
| P1 | **"CS 공부는 어디서 뭘 봐야 할지 막막하고, 시간 내기 부담된다"** — 시작 자체가 무겁다. | 5분 단위 문제 풀이(짧게 진입) — 기능 1 |
| P2 | **"문제가 어렵거나 배경지식이 없으면 그냥 포기한다"** — 막히는 순간 이탈한다. | **힌트 (히어로)** — 단계적 힌트 → 정답 공개 → 따라 입력 → 복습 재노출 — 기능 2 (디딤 문제 연결은 로드맵) |
| P3 | **"풀어도·읽어도 금방 잊는다"** — 학습이 정착되지 않는다. | 복습·정착(개념 숙련도 기반 재출제) — 기능 3 |
| P4 | **"AI가 틀린 CS 지식을 주면 어쩌지"** — 콘텐츠를 믿기 어렵다. | 콘텐츠 신뢰성 가드레일(큐레이션 검증) — 가드레일 절 |
| P5 | **"혼자서는 꾸준히 못 한다"** — 공부 의지가 높지 않아 다시 안 열게 된다. | 완결 성취·스트릭 등 **긍정 동기**(강제·죄책감 배제) — 기능 6 |
| P6 | **"단편 지식만 외우는 건 지루하다"** — 흥미가 안 생겨 이어가기 어렵다. | **유도형 문제**(개념을 재료로 여러 단계를 거쳐 답을 도출 — 암기가 아닌 적용) — 기능 1 문제 유형 / 정답 후 '더 알아보기'(심화·흥미 정보) — 기능 1 |

> P6의 해소는 콘텐츠 부가물('더 알아보기')뿐이었으나, **문제 유형**이 일급 축으로 확정되면서 **문제 자체의 차원**에서도 대응이 생겼다. **유도형**은 외운 단편을 되뇌는 것이 아니라 개념을 재료로 답을 도출하게 하므로, '단편 암기의 지루함'을 히어로 활동 안에서 던다(기능 1 문제 유형). '더 알아보기'는 그 위에 얹히는 보조 수단으로 남는다.

> P2가 이 서비스의 존재 이유와 가장 강하게 맞닿는다. "5분이면 된다"는 진입 약속은, 막혔을 때 포기하지 않게 하는 힌트가 없으면 지켜지지 않는다. 그래서 힌트를 히어로로 둔다.

# 도메인 용어 정리
이 절에서 각 도메인의 영어 명칭을 한 번씩 드러낸 뒤, 이후 서술에서는 한글 명칭만 사용한다.

- **사용자(User):** 취업·실무를 위해 CS를 공부하는 개발자(지망생). 서비스에서 문제를 풀고, 힌트로 막힘을 넘고, 복습으로 지식을 정착시킨다.
- **문제(Problem):** 사용자가 5분 안에 풀며 학습하는 단위. **주관식 단답**(정답을 직접 적는 형태)이며, 하나 이상의 **개념**을 다루고, **문제 유형**(정의 재생형·유도형)을 가지고, 난이도를 가지며, 단계적 **힌트**를 딸고 있다. (형태·유형·판정 방식 — [결정])
- **문제 유형(Problem Type):** 한 문제가 사용자에게 **무엇을 인출시키는가**를 정하는 축. **정의 재생형**과 **유도형** 둘로 나뉜다. '주관식 단답'이 정하는 **답변 형식**과는 독립된 축이다(기능 1 문제 유형). 근접 신호 적용 여부와 복습 문제 선정이 이 축을 참조한다.
- **정의 재생형(Definition Recall):** 지문이 개념의 정의·성질을 주고 사용자가 그 **개념의 이름(용어)** 을 떠올려 적는 문제 유형. **정답이 곧 개념 이름**이다. (예: 후입선출 자료구조를 설명하고 `스택`을 묻는 문제)
- **유도형(Derivation):** 지문이 재료(코드·구성·상황)를 주고 사용자가 개념을 적용해 **여러 단계를 거쳐 답을 도출**하는 문제 유형. **정답이 개념 이름이 아니며**, 수식·숫자 답이 허용된다. (예: 코드를 보고 시간 복잡도를 답하는 문제, /26 서브넷의 호스트 수를 묻는 문제)
- **허용답 집합(Accepted Answer Set):** 한 문제의 정답으로 인정되는 표기들의 집합. 큐레이터가 대표답과 허용 표기(한/영 표기, 대표 유의어 등)를 등재한다. 정답 판정은 사용자 입력을 정규화한 뒤 이 집합과 결정적으로 대조한다. (구체 정규화 규칙은 구현 노트로 분리 — [결정])
  - **대표 정답(Representative Answer) — 표시용 [결정 2026-07-16]:** 화면(정답 공개·지난 문제·스크랩 상세, 정답 처리 응답)은 허용 표기를 **나열하지 않고 대표 정답 하나만** 보여준다(예: "스레드 (thread)"). 문제마다 큐레이터가 하나를 지정하며, 한/영 병기가 자연스러우면 병기 표기를, 아니면 단일 표기를 쓴다. **불변식:** 대표 정답은 정규화 기준으로 허용답 집합에 포함되어야 한다 — 정답 공개 후 '따라 입력'에서 사용자가 화면의 대표 정답을 그대로 치면 반드시 통과해야 하기 때문. 정규화는 구두점을 접지 않으므로, 병기 표기("스레드 (thread)")는 그 문자열 자체도 허용답 집합에 등재한다.
- **모범답안(Model Answer):** 사용자가 **정답 공개**를 요청했을 때 보여주는, 그 문제의 기준 정답과 짧은 해설. 허용답 집합의 대표답을 사람이 읽기 좋게 제시한 것.
- **정답 공개(Answer Reveal):** 사용자가 스스로 요청해 모범답안을 여는 행위. 서비스가 강제로 오답을 선고하는 대신, 사용자가 원할 때만 발동한다. 도움을 받은 신호로서 복습·숙련도에 반영된다. **정답 공개 후에는 정답을 직접 입력해야 다음으로 넘어간다**(입력이 허용답 집합과 일치하면 통과 — 화면의 대표 정답 문자와 완전히 같을 필요는 없다. 눈으로만 보고 넘기지 않게 — 재생으로 정착 강화, 기능 1). **[반영 2026-07-19: C6]** 이전에는 '모범답안을 그대로 따라 입력'으로 서술했으나, 요구되는 것은 대표 정답 문자열의 전사(轉寫)가 아니라 **정답을 다시 판정 가능한 형태로 직접 입력하는 행위** 자체다.
- **근접 신호(Near-miss Signal):** 사용자의 제출이 허용답과 불일치하지만 **오탈자 수준으로 가까울 때**(결정적 편집거리) 주는 안내 — "거의 맞았어요, 오타를 확인해보세요". 정답 자체나 개념은 노출하지 않고, '진짜 오답인지 오타인지'의 혼란만 덜어 준다. 결정적이므로 판정 원칙과 정합한다. **정의 재생형에만 적용된다** — 유도형에서는 편집거리가 가까운 답이 대개 유효한 다른 답이라 이 신호가 오작동한다(기능 1 문제 유형).
- **심화 정보(Enrichment) — '더 알아보기':** 문제를 맞히거나 정답 공개로 학습한 뒤, 그 개념에 대해 **선택적으로** 볼 수 있는 추가·흥미 정보(더 깊은 배경지식, 역사적 맥락, 관련 개념 등). 큐레이션 콘텐츠이며 승인 게이트 대상. 학습 강화 + 흥미 + 재접속 동기(P6).
  - **구조 [결정 2026-07-16]:** 자유 텍스트 한 덩어리가 아니라 **제목(title) + 리드 문단(body) + 보조 항목 카드(items: 각 제목+설명, 순서 보존) + 인용(quote, 선택)** 의 고정 구조를 가진다(정답 상태 시안 78~121행). 시안이 정의한 위계를 화면이 그대로 렌더할 수 있게 하며, items·quote는 없으면 생략된다(graceful).
- **오답 교정 힌트(Misconception Hint):** 사용자가 흔히 저지르는 특정 오답을 제출했을 때 **요청 없이 자동으로** 표시되는, "왜 그 답이 다른지"를 짚고 재도전을 권하는 큐레이션 힌트. 큐레이터가 (예상 오답 표기 집합, 교정 메시지) 쌍으로 등재한다. 표시되어도 오답으로 처리하지 않는다(무낙인 유지). 사용자가 단계적으로 여는 힌트(기능 2)와 달리 **오답 제출이 트리거**이며 정답을 노출하지 않는다.
- **개념(Concept):** CS 지식의 최소 주제 단위(예: "해시 충돌", "TCP 3-way handshake", "정규화 제3정규형"). 문제보다 작은 지식 축이며, **힌트 분기·복습·진도**가 모두 이 축 위에서 돈다. **각 개념은 정확히 하나의 카테고리(대분류)에 속한다 [결정 2026-07-17].** (일급 도메인 — [결정])
- **카테고리(Category) — CS 대분류 [결정 2026-07-17]:** 여러 개념을 묶는 CS 지식의 **대분류**. 개념보다 한 층 위의 분류 축이며(카테고리 1 — 개념 N), **개념에 귀속**된다. **8개 고정 대분류**로 시작한다: 자료구조 · 알고리즘 · 운영체제 · 네트워크 · 데이터베이스 · 컴퓨터구조 · 소프트웨어공학 · 보안. 화면 표시(문제가 속한 분야)·학습 이력 분류·(2차)관심 기반 출제 가중치가 이 축을 참조한다. **대표 분류 원칙:** 한 문제가 여러 개념을 다뤄 여러 카테고리에 걸칠 수 있으나, 표시·이력에서는 **대표 개념(첫 번째 개념)의 카테고리 하나**를 그 문제의 대표 분류로 삼는다. 문제가 아직 없는 카테고리는 목록에 두되 '준비 중'으로 노출한다. (자세히는 기능 7)
- **선행 관계(Prerequisite):** 개념 사이의 방향 있는 관계. "개념 A를 이해하려면 개념 B가 먼저 필요하다". 개념 그래프를 이룬다. (**MVP 미포함 — 로드맵(디딤 묶음)**. MVP는 선행 간선을 두지 않는다.)
- **힌트(Hint):** 사용자가 문제에서 막혔을 때 포기 대신 넘게 돕는, **큐레이션으로 사전 작성된** 안내. 문제마다 **필요한 만큼(0~N개) 자유롭게 구성**되며, 약→강 순서로 필요한 만큼만 펼쳐 본다. 각 힌트는 **학습적 가치**를 가져야 한다 — 단순히 정답을 맞히게만 하는 실마리('풀이용')는 지양하고 그 개념을 이해하게 돕는다. (생성 방식·구성 — [결정])
- **힌트 레벨(Hint Level):** 한 문제의 힌트가 놓인 약→강 순서. 약한 것부터 공개되고, '더 보기'를 누를 때만 다음 것이 열린다. (고정된 종류·개수 사다리가 아니라 문제별 구성)
- **힌트 종류(예시 — 열려 있음):** 배경지식 보충(선행 개념 설명, 디딤 문제로 이어질 수 있음 — 디딤 연결은 **MVP 미포함·로드맵**)·접근 관점·코드 예시 등, **문제 성격에 맞는 것만** 큐레이터가 고른다(예: 코드를 안 봐도 되는 문제엔 코드 힌트를 넣지 않는다). 학습에 도움이 안 되는 단순 키워드 실마리는 지양한다.
- **디딤 문제(Stepping-stone Problem):** 배경지식 보충으로도 "그 배경지식 자체를 모른다"가 드러났을 때 이어지는, **부족한 선행 개념을 다루는 더 쉬운 문제.** 별도 엔티티가 아니라 '문제'가 선행 개념 맥락에서 맡는 역할이다. (**MVP 미포함 — 로드맵(디딤 묶음)**. 도메인 개념으로만 보존한다.)
- **본 문제(Primary Problem):** 일일 학습 세션에 배정되어 **세션 완료 분량으로 카운트되는** 문제. 재시도와 구별되며(재시도는 분량에 포함되지 않는다), 디딤 도입 시에는 디딤 문제와도 구별된다.
- **파고들기 스택(Learning Path Stack):** 사용자가 막힌 지점을 따라 디딤 문제로 내려갔다가 되돌아오는 경로. 여러 겹 재귀가 가능하다(문제 A→디딤 B→디딤 C→…→C 풀고 B 복귀→B 풀고 A 복귀). (돌아옴·다층 — [결정]. **MVP 미포함 — 로드맵(디딤 묶음)**)
- **일일 학습 세션(Daily Session) — '오늘의 한입':** 하루 단위로 사용자에게 배정되는 소량의 **본 문제** 묶음. 시간이 아니라 본 문제 분량으로 정의되며, 다 마치면 완료된다. 새 개념 문제와 복습 개념 문제가 섞여 배정된다. **[반영 2026-07-19: D6]** 하루에 **여러 세션**을 만들 수 있다 — "조금 더 풀기"는 완료된 세션 뒤에 이어지는 **새 세션**이며, 별도 활동이 아니라 이 세션과 완전히 같은 것이다. "오늘의 세션"은 오늘 날짜에 만들어진 세션 중 **가장 최근 것**을 뜻한다.
- ~~**추가 학습(Extra Study)**~~ **[반영 2026-07-19: D6 — 폐지]** 세션과 별개로 존재하던, 분량·완결 없는 자유 풀이 활동. **하루 여러 세션 허용**으로 세션에 완전히 흡수되어 폐지되었다 — "조금 더 풀기"는 이제 새 세션을 시작하는 진입점이다(위 일일 학습 세션 참고). 이 항목은 옛 명칭을 찾는 독자를 위해 자리만 남겨 둔다.
- **개념 숙련도(Concept Mastery):** 한 사용자가 한 개념을 얼마나 정착시켰는지의 누적 신호(맞힘·틀림·힌트/정답 공개 사용에서 결정적으로 파생). 개념별 **다음 복습 시점**이 여기서 간격 반복으로 정해진다. (구체 가중치·간격 공식은 구현 노트로 분리)
- **문항–답 연합(Item–Answer Binding):** 사용자가 개념을 이해해서가 아니라 **그 문항의 정답 문자열을 외워서** 맞히는 상태. **유도형** 문제에서 짧은 간격으로 같은 문항을 반복할 때 생길 수 있다(유도 과정을 건너뛴다). (실제 발생 정도는 미검증 — [열린 질문] 17번)
- **복습(Review):** 복습 시점이 도래한 개념을 일일 세션에서 다시 만나 정착시키는 활동. **이미 푼 그 문제를 다시 내는 것이 기본**이다(간격 반복 기반). 예외적으로 **유도형** 문제에 한해, 아직 안 낸 다른 문제가 있으면 그것을 우선 낸다(기능 3 복습 문제 선정).
- **학습 상태(Learning State):** 한 사용자에게 귀속되는 개인 데이터 묶음 — 개념 숙련도·다음 복습 시점, 진행 중 세션(파고들기 스택은 디딤 도입 시 추가 — MVP 미포함), 풀이 이력(맞힘·힌트 사용·정답 공개), **스크랩한 문제 목록**, 지속 동기 상태(스트릭 등). 공용 콘텐츠(문제·개념·힌트)와 구별되는 **사용자 소유** 데이터다. **[결정 2026-07-17]** 풀이 이력·개념 숙련도·복습 스케줄은 세션에서 동일하게 축적되며, 복습 문제 선정·세션 새 개념 배정의 'unseen(아직 안 푼)·배정 이력'은 **[반영 2026-07-19: D6]** 사용자가 그날까지 만든 **모든 세션**(하루 여러 세션 포함)의 풀이 이력 위에서 돈다(추가 학습 폐지로 '두 활동의 합집합' 규정은 불필요해졌다).
- **스크랩(Scrap):** 사용자가 다시 보고 싶은 문제를 저장해 두는 개인 북마크. 학습 상태에 귀속되며, 스크랩 목록에서 언제든 다시 열람한다.
- **게스트(Guest):** 가입하지 않고 학습을 시작한 사용자. 학습 상태가 정상적으로 쌓이며, 가입 시 그 상태가 계정으로 **승계**된다.
- **콘텐츠(Content):** 서비스가 공급하는 학습 자산의 총칭 — 문제·허용답 집합·개념·힌트·오답 교정 힌트·심화 정보. 서비스 공용 자산이며, 큐레이터 승인을 거친 것만 서빙된다.
- **큐레이터(Curator):** 콘텐츠를 작성·검수·승인하는 서비스 운영 주체. AI 초안의 게시 여부를 결정하는 신뢰성 게이트다.
- **콘텐츠 승인 상태(Content Approval State):** 콘텐츠가 사용자에게 서빙될 수 있는지를 정하는 상태 — 초안 / 검수중 / 승인 / 반려. 승인 상태만 서빙된다.

> 용어 규율: 하나의 도메인에 여러 표현을 혼용하지 않는다(예: '문제'를 '퀴즈'·'카드' 등으로 섞어 부르지 않는다). 새 용어가 필요하면 이 절에 먼저 등록한다.

# 핵심 가치 흐름 개요
서비스가 가치를 전달하는 기본 흐름이다. 흐름은 **먼저 닫히는 독립 루프(1~3)** 와 **그 위에 얹히는 정착(4)** 으로 나뉜다.

**독립 가치 루프 — 5분 안에 닫힌다 (수단 목적 1·2)**
1. **만난다:** 사용자가 5분 안에 풀 수 있는 문제 하나를 만난다. (짧게 진입 — P1 해소)
2. **막히면 넘는다:** 어렵거나 배경지식이 없으면, 약한 힌트부터 필요한 만큼 펼쳐 본다(배경지식 보충 힌트 포함). 그래도 부족하면 낙인 없이 **정답을 열어 직접 따라 적으며** 익히고, 그 개념은 나중에 복습으로 다시 만난다. **포기 대신 이해로.** (P2 해소 — 히어로)
   > 디딤 문제로 내려갔다 되돌아오는 경로(파고들기 스택)는 **로드맵**이다. MVP에서 P2는 힌트 → 정답 공개 → 따라 입력 → 복습 재노출로 받친다.
3. **푼다:** 문제를 풀며 그 개념을 능동적으로 익힌다. (수단 목적 1)

**정착 — 푼 것을 오래 남긴다 (수단 목적 3)**
4. **다시 만난다:** 약한 개념(틀렸거나 힌트를 많이 쓴 개념)을 나중에 복습으로 다시 만나 정착시킨다. (P3 해소)

이 흐름의 두 전제:
- **한 세션(5분)에서 가치가 완결된다.** 사용자는 복습(4)에 도달하지 않아도 만남→막힘 넘기→풀이(1~3)만으로 '오늘 CS 한입'을 마친다.
- **개념이 축이다.** 복습(재출제)은 **개념 노드 + 문제↔개념 태깅** 위에서 돌고, 콘텐츠가 늘수록 이 구조가 복습 가치를 키운다.
  - ⚠️ 여기서 공유되는 것은 **개념 노드와 태깅까지**다. 디딤 문제가 요구하는 **개념 간 선행 간선**(선행 관계)은 복습이 참조하지 않는다 — 기능 3의 숙련도·복습 시점·복습 문제 선정 어디에도 선행 관계가 쓰이지 않는다. 선행 간선은 디딤 문제 전용 비용이며, "하나의 구조가 두 가치를 함께 키운다"는 간선에 대해서는 성립하지 않는다.

> 5분 세션의 구체 흐름(하루 분량·시작/끝·재개·복습 편입)은 기능 1.5에서 확정한다.

# 기능별 상세 명세

## 1. 문제 풀이
> 문제 형태와 정답 판정을 아래에 확정해 담는다. 문제가 어떻게 세션 단위로 사용자에게 전달되는지(하루 분량·시작/종료·재개)는 기능 1.5 참고.

### 목적
사용자가 5분 안에 직접 문제를 풀며 CS 개념을 능동적으로 익히게 한다. 읽고 흘리는 수동 학습이 아니라, **정답을 직접 떠올려 적는** 재생(recall) 행위로 기억에 남긴다.

### 이뤄낼 가치
- "언제 시간 내서 공부하지"가 아니라 "5분이면 한 문제"로 진입 부담을 없앤다. (수단 목적 1, P1)
- 보기 중 고르는 재인이 아니라 직접 떠올리는 재생이므로, 같은 시간에 더 오래 남는다(정착 — P3와 상호 보강).
- 문제는 하나 이상의 **개념**을 다루므로, 푸는 순간 그 개념의 학습·숙련도 축적이 함께 일어난다.

### 문제 형태 (결정)
- 문제는 **주관식 단답**이다 — 사용자가 정답을 직접 적는다. (객관식·플래시카드·코드형·서술형은 채택하지 않음 — 서술형은 결정적 판정을 깨므로 미도입. 통합·시스템형 지식은 서술형 대신 '다개념 연결 단답' 등으로 로드맵에서 다룬다.)
- 문제는 **하나 이상의 개념**에 연결된다(문제 N — M 개념). 이 연결이 힌트 분기·복습의 근거가 된다.
- 문제는 **난이도**를 가진다. MVP에서는 표시 속성이며, 로드맵의 **개인 수준 맞춤 적응 출제**(숙련도 기반 난이도·문제 개인화 — 아래 로드맵 절 참고)가 이를 소비하는 지점이므로 일급 속성으로 존치한다. 디딤 문제('더 쉬운 문제' 정의, 로드맵)도 도입되면 같은 속성을 소비한다. **[결정 2026-07-16]** **[반영 2026-07-19: D5]** MVP 풀기 전 노출(은은한 표시)을 현행대로 유지 확정 — 실기기 QA에서 시작 전 위축 신호가 관찰되면 재검토한다.

> 도메인 결정 근거: 정착(P3)이 핵심 가치이므로, 재인(객관식)보다 재생(주관식)을 택한다. 재생은 직접 떠올리게 해 기억에 오래 남는다. 그 대가로 '정답 판정'이라는 난제를 떠안으며, 이를 아래 결정적 판정으로 닫는다.
>
> ⚠️ **'면접의 설명하기에도 가깝다'는 근거는 유형별로 다르다.** **정의 재생형**은 설명하기의 **역방향**이다 — 면접은 이름을 주고 설명을 요구하지만, 정의 재생형은 설명을 주고 이름을 요구한다. 따라서 이 근거는 정의 재생형에 대해서는 **성립하지 않는다**(정착 근거만으로 선다). **유도형**에 대해서는 성립한다 — 개념을 재료로 여러 단계를 거쳐 답에 이르는 과정이 면접의 '설명하기'가 요구하는 인출과 같은 방향이기 때문이다. 유도형 도입으로 이 근거가 **부분 회수**된다. (아래 '문제 유형' 참고)

### 문제 유형 (결정)
> **'문제 형태'와는 다른 축이다.** 위 '문제 형태'는 사용자가 **어떤 형식으로 답하는가**(주관식 단답)를 정하고, 문제 유형은 **무엇을 인출시키는가**를 정한다. 두 축은 독립이며, 아래 두 유형은 **모두 주관식 단답**이고 모두 결정적 허용답 대조로 판정된다.

- **정의 재생형.** 지문이 개념의 정의·성질을 주고, 사용자는 그 **개념의 이름(용어)** 을 떠올려 적는다. **정답 = 개념 이름.** 예: "가장 나중에 넣은 데이터가 먼저 나오는 후입선출(LIFO) 자료구조는?" → 허용답 `{스택, stack}`.
- **유도형.** 지문이 재료(코드·구성·상황)를 주고, 사용자는 개념을 적용해 **여러 단계를 거쳐 답을 도출**한다. **정답 ≠ 개념 이름** — 개념 이름을 알아도 유도하지 않으면 답이 나오지 않는다. 예: 코드를 보고 시간 복잡도 답하기 → 허용답 `{o(n^2), …}`, /26 서브넷의 호스트 수 → 허용답 `{62}`.
  - **유도형은 수식·숫자 답을 허용한다.** 유도의 결과는 자연어 용어가 아닐 수 있다. 표기 흔들림은 허용답 큐레이션(변형 등재)으로 흡수한다(아래 정답 판정의 '구두점을 일괄 정규화하지 않는다' 결정과 정합).
- **근접(오탈자) 신호는 정의 재생형에만 적용한다.** 유도형에는 근접 판정을 하지 않는다(근거는 아래).
- 두 유형은 MVP에 **모두 포함**된다. 유형은 문제의 속성이며, 복습 문제 선정(기능 3)과 근접 신호(아래 정답 판정)가 이 축을 참조한다.

> 배제 목록과의 정합: '문제 형태'가 채택하지 않은 객관식·플래시카드·**코드형**·서술형은 **전부 답변 형식**이다. 여기서 '코드형'은 **사용자가 코드를 작성해 제출하는 형식**을 뜻하지, 코드를 재료로 주는 문제(유도형)를 배제한다는 뜻이 아니다 — 콘텐츠 신뢰성 가드레일이 "문제·힌트·심화 정보의 **코드 스니펫**"을 명시적으로 전제하므로, 코드 지문을 읽고 단답하는 문제는 처음부터 허용된다.

> 근접 신호를 정의 재생형에만 두는 근거: 근접 신호는 "**편집거리가 작다 ⇒ 오타다**"를 가정한다. **정의 재생형에서는 참이다** — 답이 자연어 단어(개념 이름)이고 개념 이름끼리는 편집거리가 멀어(`TCP`↔`UDP`는 2), 편집거리 1은 사실상 오타다(`collsion`→`collision`). **유도형에서는 거짓이다** — 답이 수식·숫자라는 **밀집한 답 공간의 한 점**이라 이웃이 전부 **유효한 다른 답**이고, 한 글자가 곧 의미(지수·차수·자릿수)다. 허용답이 `o(n^2)`인 문제에 사용자가 `o(n)`을 제출하면 편집거리 1이라 "거의 맞았어요, 오타를 확인해보세요"가 뜬다 — **유도를 틀린 사용자에게 유도가 맞았다고 알려주는** 오작동이다(허용답 `{1024}`에 `1023`을 낸 숫자답도 같다). 대가로 유도형은 오탈자를 **허용답 등재로만** 흡수하지만, 무제한 재시도가 있어 막다른 길이 되지는 않는다.

### 정답 판정 (결정)
주관식의 유일한 난제는 "같은 답의 여러 표기(예: `충돌`/`해시 충돌`/`collision`)를 어떻게 맞다고 판정하나"이다. 이를 결정적으로 고정한다.

- **결정적 허용답 대조.** 각 문제는 큐레이션된 **허용답 집합**을 가진다. 사용자 입력을 정규화(대소문자·공백·유니코드 NFC 등)한 뒤 이 집합과 문자열 대조해 결정적으로 판정한다. 또 다른 AI 판정은 쓰지 않는다(검증에 비결정성을 들이지 않기 위함 — 힌트 큐레이션과 동일 원칙). AI 의미 채점은 로드맵.
  - **구두점은 일괄 정규화하지 않는다(결정).** `o(n^2)`·`TCP/IP`·`C++`처럼 기호가 의미를 갖는 답이 깨지기 때문. 마침표·괄호 등 표기 흔들림은 **허용답 큐레이션(변형 등재)** 으로 대응한다. (구두점 자동 폴딩은 필요 시 로드맵에서 재검토)
- **'일치'의 도메인 의도:** 같은 개념을 가리키는 표기는 정답으로 본다. 한/영 표기와 대표 유의어는 큐레이터가 허용답 집합에 등재한다. 세부 정규화·표기 동일성 규칙은 구현 노트로 분리한다.
- **근접(오탈자) 신호 — 정의 재생형에만 적용.** **정의 재생형** 문제에서 불일치이지만 허용답과 오탈자 수준으로 가까우면(결정적 편집거리 임계 내) "거의 맞았어요, 오타를 확인해보세요"류 **근접 신호**를 준다. '진짜 오답인지, 오타 때문인지'를 몰라 좌절하는 것을 막는다. 정답·개념 문자열은 노출하지 않으며(이미 사용자가 떠올린 표기의 오타만 짚음), 결정적이라 판정 원칙과 정합한다. **[반영 2026-07-19: D11]** 편집거리 임계는 길이 3자 미만 근접 없음·7자 이하 편집거리 1·8자 이상 편집거리 2로 확정(열린 질문 2 참고).
  - **유도형에는 근접 판정을 하지 않는다.** 유도형의 답은 수식·숫자라 편집거리가 가까운 이웃이 전부 유효한 다른 답이고, 근접 신호를 주면 유도를 틀린 사용자에게 유도가 맞았다고 알려 준다(근거는 위 '문제 유형'). 유도형의 오탈자는 허용답 큐레이션(변형 등재)으로 흡수한다.
  - **[반영 2026-07-19: M1] 유도형 불일치의 중립 넛지.** 근접 신호는 유도형에 줄 수 없지만, 유도형의 불일치도 표기 실수(부호·자릿수·기호 누락 등)로 인한 것일 수 있다. 이 경우 정답·개념은 노출하지 않는 **중립 넛지**("숫자·기호 표기를 다시 확인해보세요")를 줄 수 있다 — "편집거리가 가깝다"는 판단이 아니라 표기 점검을 권하는 일반 안내이므로, 오작동 근거(위 문단)와 충돌하지 않는다. ※클라이언트 구현은 로드맵(디자인 03 참고).

### 무낙인·사용자 주도 흐름 (결정)
> 관통 원칙: **서비스는 막지도 벌하지도 않는다.** 힌트·재시도·정답 공개는 모두 사용자가 원할 때만 발동한다. (존재 이유 — 공부 부담 제거와 정합)

- **불일치는 오답 선고가 아니다.** 대조가 불일치하면 '틀렸다'로 낙인찍지 않고, 사용자가 **무제한 재시도**할 수 있다.
- 제출한 답이 큐레이션된 흔한 오답과 일치하면, 요청 없이도 **오답 교정 힌트**가 자동으로 떠 "왜 그 답이 다른지"를 짚는다(기능 2.5). 이때도 오답으로 확정되지 않는다.
- 재시도 중 언제든 **힌트(약→강)**를 열 수 있다(기능 2). (디딤 문제로 파고들기는 **MVP 미포함 — 로드맵**)
- 정의 재생형에서 불일치가 오탈자 수준으로 가까우면 **근접 신호**로 "거의 맞았어요"를 알려, 진짜 오답인지 오타인지의 혼란을 덜어 준다(정답 비노출). 유도형에는 이 신호가 뜨지 않는다.
- 사용자가 **[정답 보기]**를 명시적으로 요청하면 **모범답안**을 공개한다(정답 공개). **[결정 2026-07-17]** 선행 오답을 요구하지 않는다 — 원하면 **시도 전에도** 열 수 있다(무낙인·사용자 주도). 이는 도움을 받은 신호로서 복습·숙련도에 반영된다(기능 3). ⭐️ **정답 공개 후에는 정답을 직접 입력해야 다음으로 넘어간다**(입력이 허용답 집합과 일치하면 통과 — 화면의 대표 정답 문자를 그대로 베낄 필요는 없다. **[반영 2026-07-19: C6]**) — 눈으로만 보고 넘기지 않고 손으로 한 번 써 보며 정착시킨다(벌이 아니라 학습 강화이며, 이 서비스가 진행을 요구하는 유일한 지점).

### 사용자 시나리오
- **성공(스스로 맞힘):** 사용자가 답을 적어 제출하고 허용답 집합과 일치하면 정답 처리된다. 힌트·정답 공개 없이 맞힌 것은 그 개념의 강한 숙련 신호가 된다.
- **성공(재시도 후 맞힘):** 첫 시도가 불일치해도 낙인 없이 다시 적어 맞힌다. 그 사이 힌트를 열었을 수 있다.
- **성공(근접 신호로 오타 교정):** 사용자가 `collsion`처럼 오타를 냈다면 "거의 맞았어요, 오타를 확인해보세요"가 떠, 개념은 알았음을 확인하고 바로 고쳐 맞힌다.
- **성공(정답 공개로 학습):** 끝내 못 떠올려 [정답 보기]로 모범답안을 확인한 뒤, **정답을 직접 입력**하고 넘어간다(입력이 허용답 집합과 일치하면 통과 — 손으로 써 보며 정착). 이 개념은 복습 대상으로 표시된다.
- **거짓 오답 주의(큐레이션 품질):** 사용자가 사실상 맞게 적었는데 허용답 집합에 그 표기가 없으면 계속 불일치가 나 좌절할 수 있다. 이 위험은 허용답 집합의 완전성(큐레이션 품질)에 달려 있다. (완충 장치 추가 여부는 [열린 질문])

### 수용 기준
- 제출한 답이 문제의 허용답 집합과 정규화 후 일치하면 정답으로, 아니면 불일치로 **결정적으로** 판정된다(같은 입력 → 같은 판정, AI 판정·비결정성 없음).
- 불일치는 오답으로 확정되지 않으며, 사용자는 [정답 보기]를 요청하기 전까지 횟수 제한 없이 재시도할 수 있다.
- 사용자가 [정답 보기]를 요청한 경우에만 모범답안이 공개되고, 그 사실이 복습·숙련도 신호로 기록된다.
- **정의 재생형** 문제에서 불일치가 오탈자 수준으로 가까우면(결정적 편집거리 임계 내) 근접 신호가 표시되며, 이때 정답·개념 문자열은 노출되지 않는다. **유도형 문제에서는 어떤 불일치에도 근접 신호가 표시되지 않는다.**
- 정답 공개 후에는 정답을 직접 입력해야 다음 문제로 진행되고(입력이 허용답 집합과 일치하면 통과 — 화면의 대표 정답 문자와 완전히 같을 필요는 없다), 입력 전에는 넘어갈 수 없다.
- 힌트·정답 공개 없이 맞힌 경우와 도움을 받아 맞힌 경우가 숙련도 신호에서 구분된다. (구분의 구체 반영은 기능 3에서 확정)

### 정답 후 심화 정보 — '더 알아보기' (결정)
문제를 맞히거나 정답 공개로 학습한 뒤, 그 개념에 대한 심화·흥미 정보를 제공해 학습을 강화하고 흥미로 이어간다(P6). 단편 암기의 지루함을 덜고 재접속 동기를 만든다.
- **정답 처리 이후에만, 별도 조작 없이 바로.** **[결정 2026-07-16]** 예전엔 토글을 열어야 보이는 선택적 펼침이었지만, 확인하려 매번 한 번 더 누르는 마찰을 없애기 위해 정답 처리 시점에 바로 노출하는 것으로 바꿨다. 다만 진행을 막지는 않는다 — 봐도 그만, 안 봐도 다음으로 갈 수 있다.
- **큐레이션 콘텐츠.** 심화 정보는 문제·개념에 딸린 큐레이션 콘텐츠로 승인 게이트를 거친다(콘텐츠 신뢰성 가드레일 대상).
- **없어도 됨(graceful).** 모든 문제가 심화 정보를 갖지는 않는다. 없으면 그냥 다음으로.

**수용 기준(심화 정보)**
- 심화 정보는 정답 처리(맞힘 또는 정답 공개) 이후 별도 조작 없이 바로 노출되며, 진행을 막지 않는다.
- 승인된 심화 정보만 서빙된다(신뢰성 게이트).

## 1.5 일일 학습 세션 (오늘의 한입)
### 목적
"5분이면 된다"는 진입 약속을 지키는 그릇. 사용자가 부담 없이 시작해, 정해진 소량을 끝내면 완결감을 얻고, 복습까지 자연스럽게 녹여 매일 가볍게 이어가게 한다.

### 이뤄낼 가치
- 시작 부담 제거: "오늘의 한입"이라는 작고 명확한 목표 하나만 준다. (P1)
- 완결감: 소량을 끝내면 "오늘 완료"로 마무리되어 성취가 남는다(지속 동기).
- 복습 누락 없음: 복습이 별도 숙제가 아니라 오늘 세션에 섞여 자연히 이뤄진다. (P3, 기능 3과 연결)

### 세션의 정의 (결정)
- **세션은 시간이 아니라 '본 문제 분량'으로 정의된다.** 정해진 수의 **본 문제**를 끝내면 세션이 완료된다. "5분"은 이 분량을 시간으로 번역한 진입 약속이지, 강제 종료 타이머가 아니다.
- **본 문제 수는 고정 소량이다.** **[결정]** 기본값 **5개**, 사용자가 설정에서 조정 가능(1~50, 사용자 소유 설정). **[반영 2026-07-19: D3]** "5분" 카피와의 정합을 위해 기존 기본값 10에서 하향했다 — 실기기 QA 실측 시간을 근거로 확정. (열린 질문 절 5번 참고)

> 결정 근거: 무제한 재시도·여러 겹 파고들기(기능 1·2)는 문제별 소요가 들쭉날쭉해, 시간 타이머로 자르면 학습·몰입이 끊겨 부정 경험이 된다(무낙인 원칙과 충돌). 분량 기반은 파고들어도 끊기지 않고 완결감을 주므로 존재 이유(부담 제거)와 정합한다.

### 문제 선정 (결정)
- **서비스가 자동으로 오늘 세션의 본 문제를 배정한다.** 사용자는 "오늘의 한입"을 받기만 하면 되고, 무엇을 풀지 고르는 부담이 없다.
- 배정은 **새 개념 문제와 복습할 개념 문제를 섞어** 구성한다. 복습이 별도 활동이 아니라 세션에 인터리빙된다. (섞는 비율·복습 대상 선정 규칙은 기능 3에서 확정 — 여기선 '섞인다'는 뼈대만 [결정])
- **새 개념 문제는 아직 풀지 않은 문제 중 무작위로 배정한다.** [결정] 고정 순서(예: 등록 순)로 주면 모든 사용자가 같은 순서로 같은 문제를 받아 학습 경험이 획일화되므로, 남은 칸을 채우는 새 개념 문제는 안 푼 문제 풀에서 무작위로 뽑는다. (복습 편입 문제의 우선 배치·도래 순 규칙은 기능 3을 따른다.)
- **[반영 2026-07-19: D8] 세션 소진 시 반복 폴백.** 세션에서 **안 푼 문제 소진 시 이미 푼 문제로 반복 폴백**한다 — 콘텐츠가 적은 현 단계에서 매일 완결·스트릭을 보장하기 위함이다(세션 완료 카운트에는 그대로 포함된다). 단, **반복 재출제 문제의 재정답은 해당 개념의 복습 도래 전(다음 복습 시점이 아직 미래)인 경우 숙련도 갱신에서 제외한다.** **복습 도래로 재출제된 문제의 정답은 정상적으로 숙련도에 갱신한다**(이것이 간격 반복의 본질이므로, 무조건 제외하면 간격 반복 자체가 무력화된다). 콘텐츠가 충분히 쌓이면 이 폴백 자체의 재검토(예: 폴백 없는 짧은 세션)를 오너와 논의한다.
- (사용자가 주제·개념을 직접 고르는 선택형 세션은 로드맵.)
- **관심 카테고리 반영은 2차다 [결정 2026-07-17].** 1차는 자동 배정을 그대로 두고, 2차에서 사용자의 관심 카테고리를 **새 개념 문제 배정에 가중치**로 반영한다(관심 분야는 더 자주, 비관심 분야는 덜 — 그러나 **완전히 배제하지 않고 소량 유지**). **복습 편입은 관심과 무관하게 유지된다**(정착 보호). 자세히는 기능 7.

### 완료 (결정)
- 배정된 본 문제를 모두 마치면 세션이 **완료**로 전이되고, 완결 신호("오늘 CS 한입 완료")를 준다.
- **하루 여러 세션 + 세션 일원화(결정) [반영 2026-07-19: D6·D9 — 이전 '열린 질문 6' 확정, 추가 학습 폐지].** 하루(서버 로컬 KST 자정 경계)에 세션은 **여러 개** 만들 수 있다. 완료 후 "조금 더 풀기"는 **새 세션**을 만든다 — 문제 선정(위 '문제 선정' 절, 소진 시 반복 폴백 포함)·판정·따라 입력·심화 정보 등 사용자 풀이 경험은 첫 세션과 **완전히 동일**하다. 이전의 '추가 학습'(분량·완결 없는 별도 자유 풀이 활동)은 **폐지**되었다 — 별도 개념으로 구분할 이유가 사라졌기 때문이다.
  - **"오늘의 세션"** = 오늘 만들어진 세션 중 **가장 최근 것**. 진행 중이면 이어서 풀고, 완료돼 있으면 완료 상태를 보여준다.
  - **스트릭은 그날 첫 세션 완료로만 기록된다(하루 멱등)** — 같은 날 두 번째 이후 세션을 완료해도 스트릭이 중복 반영되지 않는다. 다만 각 세션은 완료 카운트·완결 화면을 저마다 동일하게 받는다(둘째 세션도 '완료'다).
  - **[반영 2026-07-19: M2, D6로 갱신]** "조금 더 풀기"로 만든 세션의 완료가 스트릭에 추가로 잡히지 않는다는 점을 화면에서 은은하게 표시한다(예: 오늘 분량 완료 축하와 뒤섞여 "이것도 스트릭에 잡히나?"로 헷갈리지 않게, 담백한 톤 — 디자인 04 참고).

### 진행·완료 조건 (결정)
- **본 문제는 정답을 직접 입력해 맞혀야 완료로 카운트되어 다음으로 넘어간다.** 시간·건너뛰기로는 넘어가지 않는다(학습 행위 보장).
- **막히면 정답 공개(무낙인 안전판).** 풀리지 않는 본 문제는 사용자가 **정답을 확인**할 수 있다(벌점·비난 없음). **[결정 2026-07-17]** 안전판은 오답 후로 제한하지 않는다 — 사용자가 원하면 **시도 전에도** 열 수 있다(불변식 3: 사용자가 요청한 경우에만 공개하는 규칙은 그대로 유효). 단, **정답을 본 뒤에도 직접 입력해 맞혀야** 다음으로 진행된다("정답 공개 후 직접 입력" — 눈으로만 넘기지 않고 손으로 익히게. 입력이 허용답 집합과 일치하면 통과이며, 대표 정답 문자열을 그대로 베낄 필요는 없다 — **[반영 2026-07-19: C6]**). 정답 공개 여부는 기록해 이후 복습 편입(기능 3)의 신호로 쓸 수 있다.

### 지난 문제 다시 보기 (결정)

## 2. 힌트 (히어로)
### 목적
문제에서 막혔을 때, 사용자가 포기하고 이탈하는 대신 **필요한 만큼만 도움을 받아 스스로 넘게** 한다. 이 서비스가 "5분이면 된다"는 약속을 지키게 하는 핵심 장치.

### 이뤄낼 가치
- 배경지식이 없어도 이탈하지 않는다. 막힘이 곧 포기가 아니라 '한 단계 내려가 이해하고 올라오기'가 된다. (P2 — 히어로)
- 답을 떠먹이지 않으므로 학습 효과가 보존된다(아래 핵심 긴장 참고).

### 핵심 긴장과 도메인 원칙 (결정)
> **진입 장벽 제거 ↔ 학습 효과 보존.** 힌트가 답을 떠먹이면 진입장벽은 사라지지만 학습이 죽고, 너무 약하면 여전히 막혀 포기한다. 이 줄타기를 다음 원칙으로 고정한다.

- **단계적(약→강) 공개.** 한 문제에 여러 힌트가 약→강 순서로 큐레이션되어 있고, 가장 약한 것부터 보인다. 사용자가 '더 보기'를 누를 때만 다음 것이 열린다. 필요한 만큼만 꺼내 쓰므로 학습 효과가 보존된다. (점진성 — [결정])
- **문제별 자유 구성 + '학습적이어야 한다'.** ⭐️ 고정된 종류·개수 사다리(예: 항상 키워드→배경지식)를 두지 않는다. 문제마다 **필요한 힌트만** 큐레이터가 고른다(0~N개). 각 힌트는 **그 개념을 이해하게 돕는 학습적 가치**를 가져야 하며, 단순히 정답 단어를 맞히게만 하는 '풀이용 실마리'는 지양한다. 코드를 안 봐도 되는 문제에는 코드 힌트를 넣지 않는 등, 힌트 구성은 문제 성격을 따른다. (구성 — [결정])
- **떠먹이기 금지.** '정답을 향한 직접 유도(스캐폴딩)'는 지양한다 — 학습을 죽이기 때문. 힌트는 이해를 돕되 정답을 노출하지 않는다.
- **큐레이션 사전 작성.** 모든 힌트는 사람이 문제마다 미리 작성한다. 결정적이고 품질·신뢰성이 보장된다. AI 실시간 생성은 채택하지 않는다(MVP). (생성 방식 — [결정])

> 결정 근거(리뷰 반영): 고정 사다리의 'L1 키워드 힌트가 풀이용일 뿐 학습에 도움이 안 된다'는 리뷰를 반영해, 힌트를 문제별 자유 구성으로 바꾸고 '학습적 가치'를 원칙으로 못박는다.

### 디딤 문제로의 분기 (결정 — **MVP 미포함·로드맵(디딤 묶음)**)
> ⚠️ 이 절은 도메인 개념으로 보존하되 **이번 단계에서 구현하지 않는다**(MVP 경계·로드맵 절 참고). MVP에서 배경지식 보충 힌트는 설명까지만 하고 문제로 이어지지 않는다.

- 배경지식 보충으로도 "그 배경지식 자체를 모른다"가 드러나면, 그 **선행 개념을 다루는 더 쉬운 문제(디딤 문제)** 로 내려갈 수 있다.
- 연결은 **개념을 매개로** 한다: 힌트가 부족한 선행 개념을 가리키고, 그 개념을 다루는 더 쉬운 난이도의 문제로 이어진다. (개념 매개 — [결정]. 개념→문제 선택 규칙의 세부는 **[로드맵 하위 — 디딤 도입 시 결정]**.)
- 디딤 문제를 풀면 **원래 문제로 되돌아온다.** 디딤 문제가 또 어려우면 다시 그 아래로 내려갈 수 있다(파고들기 스택, 여러 겹 재귀). (돌아옴·다층 — [결정])

### 사용자 시나리오
- **성공(약한 힌트로 해결):** 사용자가 막혀 가장 약한 힌트를 열자 개념이 떠올라 스스로 푼다. 더 강한 힌트까진 열지 않는다.
- **성공(디딤 문제로 파고들기 — 로드맵):** 배경지식 보충을 봐도 그 선행 개념을 몰라, 디딤 문제로 내려가 그 개념을 익히고 되돌아와 원래 문제를 푼다. (디딤 도입 시 — MVP 미포함)
- **성공(다층 파고들기 — 로드맵):** 디딤 문제조차 막혀 그 아래 디딤 문제로 또 내려갔다가, 차례로 되돌아오며 원래 문제까지 해결한다. (디딤 도입 시 — MVP 미포함)
- **저부담(막다른 길 없음):** 힌트를 다 봐도 못 풀면, 오답 낙인 없이 무제한 재시도하거나 사용자가 [정답 보기]로 모범답안을 열 수 있다(기능 1의 무낙인·사용자 주도 흐름). 서비스가 강제로 실패를 선고하지 않는다.

### 수용 기준 (초안)

## 2.5 오답 교정 힌트
### 목적
사용자가 특정 개념을 혼동해 흔한 오답을 냈을 때, 무미건조한 "다시 시도하세요" 대신 **그 오답이 왜 다른지 짚어** 스스로 바로잡게 한다. 막힘의 순간을 오개념 교정의 기회로 바꾼다.

### 이뤄낼 가치
- "포기 대신 이해"를 오답을 낸 바로 그 지점에서 즉시 실현한다. 사용자는 자신의 착각을 정확히 지적받고 다시 도전한다. (P2 보강)
- 오답을 처벌이 아니라 학습 신호로 다룬다(무낙인 원칙 강화).

### 도메인 성격 (결정)
- **오답 트리거·자동(push).** 사용자가 요청하지 않아도, 제출한 답이 큐레이션된 예상 오답과 일치하면 그 교정 힌트가 자동 표시된다. (사용자가 여는 기능 2 힌트와 구별)
- **결정적 매칭.** 예상 오답 표기 집합도 허용답 집합과 동일하게 정규화 후 결정적으로 대조한다. AI 판정을 쓰지 않는다(전체 판정 원칙과 일관).
- **무낙인 유지.** 오답 교정 힌트가 떠도 제출을 오답으로 확정하지 않는다. 사용자는 계속 무제한 재시도할 수 있다(기능 1 흐름과 정합).
- **막다른 길 없음(graceful).** 큐레이션되지 않은 임의의 오답은 교정 힌트 없이 일반 불일치 처리(무제한 재시도)로 흐른다. 오답 교정 힌트는 품질 향상 장치이지 모든 오답을 덮는 전제조건이 아니다.
- **정답 비노출.** 교정 힌트는 "왜 그 답이 아닌지"만 짚고 정답을 말하지 않는다(정답은 [정답 보기]로만). 학습 효과 보존.
- **[반영 2026-07-19: L2] 오답 candor(솔직도) 위계는 의도된 차이다.** **큐레이션 오답**(예상 오답 집합에 매칭)은 "왜 그 답이 다른지"를 **명확히 교정**해 짚어 준다 — 큐레이터가 그 오개념을 미리 알고 있기 때문이다. **일반 불일치**(매칭 안 됨)는 원인을 알 수 없으므로 **의도적으로 모호한** "다시 도전해보세요"에 그친다 — 근거 없이 추측해 짚으면 오히려 틀린 교정이 될 수 있다. 이 비대칭은 결함이 아니라 **아는 만큼만 말한다**는 결정적 판정 원칙의 연장이다.

### 사용자 시나리오
- **성공(오개념 교정):** 정답이 '스레드'인 문제에 사용자가 '프로세스'라 적자, "프로세스는 …라 이 문제의 답과는 달라요. 다시 도전해보세요!"가 자동으로 뜬다. 오답 처리는 되지 않고, 사용자는 이를 참고해 '스레드'를 떠올려 맞힌다.
- **성공(예상 밖 오답):** 큐레이터가 예상하지 못한 답을 적으면 교정 힌트 없이 일반 불일치로 흐르되, **[반영 2026-07-19: L2]** 완전히 방향 없는 "다시 도전"에 그치지 않고 "표기를 바꿔보거나 힌트를 열어볼까요?" 정도의 **최소 방향 제시**는 준다(오개념을 짚지는 못해도 다음 행동은 가리킨다). 사용자는 [힌트 보기]나 재시도로 이어간다.

### 수용 기준
- 제출한 답이 허용답 집합과 불일치하되 어떤 오답 교정 힌트의 예상 오답 집합과 정규화 후 일치하면, 그 교정 힌트가 자동으로 표시된다(결정적).
- 오답 교정 힌트가 표시되어도 제출은 오답으로 확정되지 않으며, 재시도 제한이 없다.
- 예상 오답에 매칭되지 않는 불일치는 교정 힌트 없이 일반 재시도 흐름으로 처리된다.
- 오답 교정 힌트는 정답(모범답안)을 노출하지 않는다.

## 3. 복습·정착
### 목적
한 번 풀고 잊는 것을 막고, 약한 개념을 **적절한 시점에** 다시 만나 오래 남는 지식으로 정착시킨다.

### 이뤄낼 가치
- "풀어도 금방 잊는다"(P3)를 개념 단위 재노출로 해소한다.
- 복습은 문제 풀이가 이미 요구하는 **문제↔개념 태깅**을 그대로 쓰므로, 복습을 위한 별도 구조를 새로 만들지 않는다(구조 재사용).
- 복습이 별도 숙제가 아니라 일일 세션에 섞여 들어와, 사용자가 의식하지 않아도 정착이 일어난다(기능 1.5).

### 복습의 축 (결정)
- 복습·정착은 **개념(개념 숙련도)을 축으로** 돈다. "이 문제를 다시 풀라"가 아니라 "이 **개념**이 약하니 다시 만나라"가 기본 단위다.

### 개념 숙련도 산정 (결정)
- 사용자는 각 개념에 대해 **숙련도**와 그로부터 파생되는 **다음 복습 시점**을 가진다.
- 숙련도는 문제 풀이 결과에서 결정적으로 파생된다:
    - **힌트·정답 공개·오답 교정 힌트 없이 스스로 맞힘** → 강한 정착 신호(숙련도↑).
    - **도움받아 맞힘**(힌트를 열고 맞힘) → 약한 정착 신호.
    - **정답 공개(포기)·끝내 못 맞힘** → 숙련도↓.
- 숙련도 산정은 결정적이다(같은 성과 이력 → 같은 숙련도·다음 복습 시점). LLM 호출이 없다. (구체 가중치·간격 공식은 구현 노트/열린 질문으로 분리)

> **관찰(구현 노트) [기록 2026-07-19: O1].** 현재 구현은 숙련도 레벨을 상하한으로 클램프하는데, 그 경계에서 신호 구분이 붕괴하는 지점이 있다 — 최저 레벨에서는 '도움받아 맞힘'과 '정답 공개(포기)'가 같은 값으로 눌리고, 최고 레벨에서는 '스스로 맞힘'과 '도움받아 맞힘'이 같은 값으로 눌린다. 위 "서로 다르게 반영된다"는 결정은 경계 밖 범위에서는 정확히 성립하지만 경계에서는 그 구분이 사라진다. 가중치·간격 공식이 아직 [열린 질문] 7번으로 미확정이라 이 경계 완화가 오너 결정 대상인지 구현 세부인지 아직 갈라지지 않았으므로, 별도 결정 없이 **관찰로만** 남기고 7번 확정 시 함께 다룬다.

### 복습 시점 — 간격 반복 (결정)
- 복습 시점은 개념별로 **간격 반복(spaced repetition)** 으로 정해진다. 맞히면 다음 복습 간격이 멀어지고, 틀리거나 포기하면 가까워진다.
- 간격 반복은 결정적이므로(성과 이력이 같으면 다음 시점이 같다) 우리의 결정적 판정 원칙과 정합한다. (구체 간격 공식은 구현 노트로 분리)

### 복습 문제 선정 (결정)
- 복습은 **이미 푼 그 문제를 다시 낸다.** 간격 반복의 표준 방식이며, 개념당 문제를 여러 개 쌓지 않아도 성립한다(콘텐츠 공급이 병목인 이 서비스의 제약과 정합 — 콘텐츠 신뢰성·비용 가드레일 참고).
  - 같은 문제를 다시 내도 사용자는 **정답을 직접 떠올려 적는다**. 이 서비스가 지키는 재생(recall)은 '보기 중 고르기(재인)'와 대비되는 **답변 형식**의 축이므로(기능 1 문제 형태), 재출제로 훼손되지 않는다.
- **[예외 — 유도형]** **유도형**(기능 1 문제 유형 — 정답이 개념 이름이 아니고 여러 단계를 거쳐 도출하는 문제)에 한해, 그 개념에 **아직 안 낸 다른 문제가 있으면 그것을 우선** 낸다. 유도형은 문항과 답의 연합이 생기면 **유도 과정을 건너뛸** 수 있기 때문이다. 없으면 같은 문제로 낸다.
  - ⚠️ **정의 재생형**에는 이 예외가 적용되지 않는다. 정의 재생형은 **다른 문제를 내도 정답이 같은 개념 이름**이므로, 문제를 바꾸는 것이 문항–답 연합을 막지 못한다.

> 결정 근거: 이전 결정은 "같은 개념의 다른 문제를 우선 낸다"였고 근거는 "이미 푼 문제를 그대로 내면 답을 외운 **재인**이 되어 정착 검증이 흐려진다"였다. 두 가지가 틀렸다. (1) 여기서 쓰인 '재인'은 이 문서가 기능 1에서 정의한 '재인'(보기 중 고르기)과 다른 뜻이라 **용어 규율 위반**이다 — 가리키려던 것은 **문항–답 연합**이다. (2) 그 수단이 목적을 달성하지 못한다 — 정답이 개념 이름인 문제에서는 문제를 바꿔도 답이 같아, 검수 처리량을 두 배로 지불하고도 지목한 위험을 막지 못한다. 위험이 실재하는 범위(**유도형**)에만 예외를 남긴다. (예외의 범위는 처음 "정답이 개념 이름이 아닌 문제"로 **서술**돼 있었으나, 문제 유형이 일급 축으로 확정되면서 **유도형**이라는 유형 이름으로 바꿔 쓴다 — 예외가 요구하던 속성이 이제 문제 형태·유형 절에 정의되어 있다.)

> **문항–답 연합**은 도메인 용어 정리 절에 정의되어 있다. 실제 발생 정도는 미검증 — [열린 질문] 17번.

### 세션 편입 (결정)
- 오늘 세션을 구성할 때, **복습 시점이 도래한 개념이 있으면 그 개념의 복습 문제를 우선 본 문제로 편입**하고, 남은 분량을 새 개념 문제로 채운다(기능 1.5 자동 배정과 연결). (편입 비율·상한은 [열린 질문])

### 사용자 시나리오
- **성공(자연스러운 재노출):** 며칠 전 힌트를 여러 번 써서 겨우 맞힌 개념이 복습 시점이 되어, 오늘 세션에 **그때 푼 그 문제**가 다시 나온다. 이번엔 힌트 없이 맞혀 숙련도가 오르고 다음 복습이 멀어진다.
- **성공(약점 집중):** 계속 틀리거나 포기하는 개념은 복습 간격이 짧게 유지되어 자주 다시 만나며 결국 정착된다.
- **성공(예외 — 유도형):** 코드를 보고 시간 복잡도를 답하는 문제 같은 유도형은, 그 개념에 아직 안 낸 다른 문제가 있으면 그것이 대신 나와 문항–답 연합을 피한다. 없으면 같은 문제로 낸다.

### 수용 기준
- 각 사용자·개념 쌍은 숙련도와 다음 복습 시점을 가지며, 문제 풀이 결과로 결정적으로 갱신된다(같은 이력 → 같은 결과, LLM 없음).
- 힌트·정답 공개 없이 맞힌 경우와 도움받아 맞힌 경우, 포기·오답인 경우가 숙련도에 서로 다르게 반영된다.
- 복습 시점이 도래한 개념은 이후 세션에 그 개념의 문제로 편입된다.
- 복습 문제는 **이미 푼 그 문제를 다시 낸다.** 단, **유도형**에 한해 그 개념에 아직 안 낸 다른 문제가 있으면 그것을 우선한다(없으면 같은 문제).

## 4. 계정·데이터 소유
### 목적
학습 상태(숙련도·복습·이력)를 사용자 단위로 보존해 지속·정착을 보장하되, 진입 부담을 없애기 위해 가입 없이도 바로 시작하게 한다.

### 이뤄낼 가치
- 무가입 즉시 체험으로 진입 마찰 제거. (존재 이유 — 부담 제거)
- 가입하면 학습 상태가 안전히 보존되어 회차를 거듭하며 복습·정착이 이어진다. (P3, 지속성)
- 사용자가 자기 데이터를 통제(삭제)할 권리를 가진다.

### 소유 모델 (결정)
- **콘텐츠(문제·개념·힌트·오답 교정 힌트·허용답 집합)는 서비스 공용 자산**이며 모든 사용자가 공유한다. 사용자 소유가 아니다.
- **학습 상태는 사용자 소유**다: 개념 숙련도·다음 복습 시점, 진행 중 세션(파고들기 스택은 디딤 도입 시 — MVP 미포함), 풀이 이력. 정확히 한 사용자에게 귀속되며 다른 사용자는 접근할 수 없다.

### 게스트 시작과 가입 승계 (결정)
- **게스트로 바로 시작.** 가입 없이 즉시 문제를 풀고 힌트·복습을 쓸 수 있다. 게스트의 학습 상태도 정상적으로 쌓인다.
- **가입 시 승계.** 게스트가 가입하면 그때까지 쌓인 학습 상태가 새 계정으로 **승계**되어 유실 없이 이어진다.
- **가입 전 한계(고지 대상).** 게스트 학습 상태는 그 **기기의 게스트 토큰에 묶여 있어**(서버에는 GUEST 역할 계정으로 보관되지만, 그 계정에 이르는 열쇠는 기기에만 있다) 앱 삭제·기기 이전 시 접근이 유실될 수 있다. 지속·정착이 핵심 가치이므로 서비스는 적절한 시점에 가입을 **권해**(강제하지 않음 — 부담 제거와 정합) 데이터를 안전하게 보존하도록 안내한다. **[반영 2026-07-19: M5]** 사용자에게는 "가입하면 기록을 안전하게 지켜요. 가입 전에는 이 기기에만 저장돼요"처럼, **가입이 곧 보존 조건**임을 정확히 전달한다(서버 보관 여부를 과약속하지 않음). (게스트 상태의 서버 보관 세부는 구현 노트/[열린 질문] 9번 — [결정] 참고)

### 데이터 영속·삭제 (결정)
- **영속:** 가입 사용자의 학습 상태는 명시적으로 지우기 전까지 보존된다. 재로그인해도 숙련도·복습·세션이 그대로다.
- **계정 삭제:** 그 사용자의 모든 학습 상태(숙련도·복습·세션·이력)가 함께 삭제된다. '되돌릴 수 없음'을 고지하고 확인받는다. 공용 콘텐츠는 영향받지 않는다.
  - **[반영 2026-07-19: D10]** 콘텐츠 오류 신고(content_report)는 사용자의 학습 상태가 아니라 콘텐츠 품질 운영 데이터다. 계정 삭제 시 신고는 삭제하지 않고 신고자 식별자를 비워 익명화하여 보존한다(큐레이터의 품질 신호 유지 + 개인정보 정합).
- **인증 수단**(이메일·소셜 등)은 구현·운영 판단으로 두고 도메인에서 고정하지 않는다.

### 사용자 시나리오
- **성공(무가입 체험):** 처음 온 사용자가 가입 없이 바로 오늘의 한입을 풀고 힌트를 쓴다. 숙련도가 쌓이기 시작한다.
- **성공(가입 승계):** 며칠 게스트로 쓰던 사용자가 가입하면, 그동안의 숙련도·복습 스케줄이 계정으로 이어져 끊김 없이 계속한다.
- **성공(재방문):** 가입 사용자가 다른 기기에서 로그인해도 학습 상태가 그대로 보인다.
- **성공(계정 삭제):** 탈퇴를 요청하면 무엇이 사라지는지 안내받고, 확인 후 학습 상태가 완전히 삭제된다.
- **실패(권한 없는 접근):** 다른 사용자의 학습 상태에 접근하려는 시도는 차단된다.

### 수용 기준
- 가입 없이도 문제 풀이·힌트·복습이 동작하고 게스트 학습 상태가 쌓인다.
- 게스트가 가입하면 가입 이전의 학습 상태가 계정으로 승계되어 유실 없이 이어진다.
- 한 사용자의 학습 상태는 다른 사용자에게 노출되지 않는다.
- 재로그인 후 기존 학습 상태(숙련도·복습·세션)가 동일하게 보존된다.
- 계정 삭제 후 그 사용자의 모든 학습 상태가 조회되지 않으며, 공용 콘텐츠는 영향받지 않는다.

## 5. 문제 스크랩
### 목적
사용자가 다시 보고 싶은 문제(어려웠던·중요한·흥미로운)를 저장해, 나중에 스스로 찾아 다시 볼 수 있게 한다.

### 이뤄낼 가치
- 복습(자동·개념 단위)과 별개로, 사용자가 **직접 고른 문제**를 소장·재열람하는 통제감을 준다.
- "이건 나중에 또 봐야지"를 놓치지 않게 한다.

### 도메인 성격 (결정)
- 스크랩은 **학습 상태**에 귀속되는 개인 북마크다(사용자 소유·격리). 공용 콘텐츠(문제)를 가리키는 참조일 뿐 문제 자체를 복제하지 않는다.
- 스크랩 목록에서 언제든 해당 문제를 **읽기 전용**으로 다시 연다(문제·모범답안 등).
- 스크랩은 복습 알고리즘(간격 반복)과 **독립**이다 — 자동 재출제가 아니라 사용자 주도 열람이다(복습과 혼동 금지).
- 계정 삭제 시 스크랩도 함께 삭제된다(학습 상태 일부). 스크랩한 문제가 회수·삭제되면 **[반영 2026-07-19: L3]** 목록에서 "내용 점검을 위해 잠시 내려갔어요(내 학습 기록은 그대로예요)"로 안내한다 — 이전 '더 이상 볼 수 없음'은 마치 사용자 귀책이나 영구 상실처럼 읽혀 과약속·불안을 줄 수 있어 대체한다. 회수는 **콘텐츠 신뢰성 가드레일의 정상 동작**이며(가드레일 절 참고), 회수됐다고 그 문제로 쌓인 숙련도·복습 이력이 사라지거나 영향받지 않는다는 점도 함께 명시한다([열린 질문]과 연결).

### 사용자 시나리오
- **성공(스크랩·재열람):** 헷갈렸던 문제를 스크랩해 두고, 나중에 스크랩 목록에서 다시 열어 모범답안과 함께 복기한다.

### 수용 기준
- 사용자는 문제를 스크랩·해제할 수 있고, 스크랩 목록에서 해당 문제를 다시 열람할 수 있다.
- 스크랩은 사용자 소유이며 타인에게 노출되지 않고, 계정 삭제 시 함께 삭제된다.

## 6. 지속·성취 동기
### 목적
공부 의지가 높지 않은 타깃(P5)이 다시 돌아오게 만든다. 단, **강제·죄책감이 아니라 긍정 동기(성취감)** 로.

### 이뤄낼 가치
- 완결 성취·연속 학습(스트릭) 같은 긍정 신호로 "또 하고 싶다"를 만든다.
- 재접속을 높이되 존재 이유(부담 제거)·무낙인과 충돌하지 않는다.

### 도메인 성격 (결정)
- **긍정 동기 전용.** 스트릭·완결 성취·(선택) 부드러운 리마인더 등 긍정 요소만 쓴다.
- ⛔ **다크 패턴 금지.** 죄책감·불안·강제("안 하면 …를 잃어요" 식 위협)나 스트릭 상실 공포 연출은 쓰지 않는다(UX 핵심 가이드 4). 스트릭이 끊겨도 비난 없이 다시 시작하도록 맞이한다.
- **학습 강제 없음.** 재접속을 유도할 뿐 어떤 학습도 강제하지 않는다(세션·문제 풀이의 무낙인·저부담 유지).

### MVP 경계
- **MVP:** 완결 성취(세션 완료 축하)·연속 학습(스트릭) 표시 등 **앱 내 긍정 신호**.
- **로드맵:** 푸시 알림·리마인더 등 앱 밖 재접속 유도(인프라 필요), 성취 배지 체계 확장.

### 사용자 시나리오
- **성공(성취로 재접속):** 세션을 완료하면 축하와 함께 스트릭이 오르고, 다음 날 "이어가고 싶다"는 긍정 동기로 다시 연다.
- **저부담(스트릭 끊김):** 며칠 쉬어 스트릭이 끊겨도 비난·죄책감 없이 "다시 시작해요"로 맞이한다.

### 수용 기준
- 지속 동기 요소는 긍정 신호(성취·스트릭)만 사용하며, 죄책감·강제·상실 공포 연출을 쓰지 않는다.
- 어떤 지속 동기 요소도 학습을 강제하지 않는다(재접속 유도까지만).

## 7. 문제 카테고리
### 목적
사용자가 지금 푸는 문제가 CS의 어느 **대분류(자료구조·알고리즘 등)** 에 속하는지 알게 하고, 푼 문제를 분야별로 되짚어 보게 한다. 나아가 관심 분야 위주로 학습을 조절할 여지를 준다(2차).

### 이뤄낼 가치
- 지금 문제의 맥락(분야)을 알아 학습의 큰 그림을 잡는다.
- 푼 문제를 분야별로 모아 봐, 어느 영역을 얼마나 다뤘는지 자기 점검한다.
- (2차) 관심 분야를 더 자주 만나 동기·효율을 높이되, 편식은 막는다.

### 카테고리 체계 (결정 2026-07-17)
- **8개 고정 대분류:** 자료구조 · 알고리즘 · 운영체제 · 네트워크 · 데이터베이스 · 컴퓨터구조 · 소프트웨어공학 · 보안.
- 카테고리는 **개념에 귀속**된다(카테고리 1 — 개념 N). 개념이 이미 복습·힌트·진도의 축이므로, 대분류를 개념 위에 얹어 **단일 출처**로 관리한다(문제마다 따로 붙이지 않는다 — 같은 개념의 문제들이 어긋나지 않게).
- **대표 분류 원칙:** 한 문제가 여러 개념을 다뤄 여러 카테고리에 걸칠 수 있으나, 화면 표시·이력 분류에서는 **대표 개념(첫 번째 개념)의 카테고리 하나**를 그 문제의 대표 분류로 삼는다.
- **'준비 중' 분야:** 아직 문제가 없는 카테고리는 목록에 두되 '준비 중'으로 노출한다(빈 목록이 오류로 보이지 않게 — 긍정 빈 상태, UX 가이드 9). **[반영 2026-07-19: D7]** 런칭 시 '준비 중' 카테고리가 과반이어도 **전량 노출을 유지**하기로 확정 — 접기·하단 정렬 등 별도 처리를 두지 않는다. 8개 대분류는 항상 같은 순서·같은 위계로 전부 보인다(로드맵 기대감을 우선하는 선택).

### 표시·노출 시점 (결정 2026-07-17)
- 문제 풀이 화면에 그 문제의 **대표 분류를 풀기 전부터 표시**한다. 개념명은 정답을 스포일하므로 풀기 전 숨기지만(기능 1·디자인 03의 개념 칩 규칙), 카테고리는 대분류라 스포일 위험이 훨씬 낮다.
- ⚠️ **힌트 리스크 수용 [결정 2026-07-17].** 정의 재생형 중 정답이 대분류에 가까운 극소수 문제에서는 대표 분류가 약한 힌트가 될 수 있다. **오너가 이 위험을 인지·수용하고 '풀기 전 표시'를 유지**하기로 했다(원래 요구 유지 — 기획자 추천안이던 '푼 후 표시'는 채택하지 않음). **[반영 2026-07-19: D4] 배지 숨김의 결정적 기준 — 큐레이터 재량 문구를 대체.** 한 문제의 대표 분류 배지는 **정답이 카테고리명 또는 그 직접 파생어와 정규화 기준(위 정답 판정 절과 동일한 정규화)으로 근접할 때** 숨긴다(예: 카테고리 '자료구조'에 정답이 '자료 구조'류인 극소수 문제). 이 기준을 만족하지 않는 한 배지는 항상 노출한다 — '과하다 싶으면'류의 재량 판단을 남기지 않는다. Phase 2 검수 화면(문제 콘텐츠 파이프라인 계획 A4)에 이 기준을 체크 항목으로 반영할 예정.

### 카테고리별 학습 이력 (1차)
- 사용자는 푼 문제를 **카테고리별로 필터·조회**한다(읽기 전용 — 문제·내 답·정답 여부·모범답안). **[반영 2026-07-19: D6]** 사용자가 그날까지 만든 **모든 세션**(하루 여러 세션 포함)에서 푼 문제를 대표 분류로 그룹핑한다(추가 학습 폐지로 '합집합' 규정은 불필요해졌다).
- 복습(간격 반복)과 독립이며, 스크랩과 유사한 재열람 성격이다(자동 재출제 아님).
- **[반영 2026-07-19: D13] 회수 콘텐츠 마스킹은 Phase 2.** 카테고리 이력에 노출되는 문제가 이후 **회수(RETRACTED)** 되면, **스크랩과 동일한 방식**(항목은 이력에 그대로 유지하되 내용만 가림 — 기능 5 참고)을 Phase 2(문제 콘텐츠 파이프라인 구현 계획 §4.2)에서 적용한다. 항목을 이력에서 빼지 않으므로 숙련도·스트릭 집계 수와 화면 표시가 어긋나지 않는다. 이번 단계는 계획 확인이며 신규 결정이 아니다.
- **[반영 2026-07-19: M4] 빈 상태 2종 분리.** 한 카테고리에 이력이 없는 상태는 원인이 다른 두 경우를 구분해 보여준다(긍정 빈 상태, UX 가이드 9) —
  1. **(a) 준비 중 분야:** 그 카테고리에 아직 문제 자체가 없음(위 '준비 중' 노출과 동일 조건).
  2. **(b) 콘텐츠는 있으나 미학습:** 문제는 있지만 이 사용자가 아직 풀지 않음 — "아직 풀어보지 않았어요 — 지금 만나보기"처럼 그 카테고리로 바로 이어지는 행동 유도형 문구를 쓴다.
  두 경우를 하나의 '빈 목록'으로 뭉뚱그리면 (b)에서도 콘텐츠가 없다는 오해를 줄 수 있어 나눈다. (런칭 초기 '준비 중' 노출이 과다할 때의 완화 방식은 별도 오너 결정 사안 — 여기 반영 대상이 아님.)

### 관심 카테고리 기반 출제 (2차)
- 사용자가 **관심 카테고리**를 설정하면(사용자 소유 설정, 게스트 포함), **새 개념 문제 배정에 가중치**로 반영한다 — 관심 분야는 더 자주, 비관심 분야는 덜, 그러나 **완전히 배제하지 않는다**(소량 유지 — 편식·콘텐츠 고갈 방지, 자동 배정 철학과의 타협).
- **복습 편입은 관심과 무관하게 항상 유지된다**(이미 배운 개념의 정착이 취향으로 끊기지 않게).
- 완전 제외(비관심 전면 차단)는 채택하지 않는다.

### 백필·콘텐츠 운영 (결정 2026-07-17)
- 기존 문제(테스트 전용 100여 문제 포함)의 개념을 **AI가 일괄 분류**하고 **오너가 검수**해 카테고리를 부여한다. 개념 단위 분류라 문제 수보다 훨씬 적게 든다.
- 테스트 전용 세트도 함께 분류하되, **큐레이터 검수 전 실서비스 투입 금지** 상태는 그대로 유지한다.
- 개념 배열에 대분류성 용어(예: '자료구조')가 개념으로 섞여 태깅된 경우, 개념에서 빼고 카테고리로 정리한다.

### 수용 기준
- 모든 개념은 8개 카테고리 중 하나에 (검수 후) 귀속되며, 문제의 대표 분류는 대표 개념의 카테고리로 결정적으로 도출된다.
- 문제 풀이 화면은 대표 분류를 **풀기 전부터** 표시한다.
- 사용자는 푼 문제를 카테고리별로 조회할 수 있다(1차, **[반영 2026-07-19: D6]** 사용자의 전체 세션 풀이 이력 기준 — 추가 학습 폐지로 합집합 규정 불필요).
- (2차) 관심 카테고리 설정은 새 개념 문제 배정에 **가중치로만** 반영되고 **복습 편입에는 영향을 주지 않으며**, 비관심 카테고리도 완전히 배제되지 않는다.
- 문제가 없는 카테고리는 '준비 중'으로 노출된다.

# 문제·개념·힌트 구조와 상태 모델
> 이 절은 엔티티·관계·상태 전이를 결정적으로 정의해 개발자가 도메인 모델을 추측 없이 도출하게 하는 것이 목적이다. 현재는 합의된 뼈대만 담고, 나머지는 [논의 예정]으로 표시한다.

## 관계 (결정된 뼈대)
- **문제 — 문제 유형:** 한 문제는 **정확히 하나의 문제 유형**(정의 재생형 / 유도형)을 가진다. 유형은 답변 형식과 독립된 축이며, 근접 신호 적용 여부(기능 1)와 복습 문제 선정(기능 3)이 이 값으로 분기한다.
- **문제 N — M 개념:** 한 문제는 하나 이상의 개념을 다루고, 한 개념은 여러 문제에 나타난다.
- **개념 — 카테고리(대분류) [결정 2026-07-17]:** 한 개념은 **정확히 하나의 카테고리**에 속한다(카테고리 1 — 개념 N). 카테고리는 8개 고정 대분류다(자료구조·알고리즘·운영체제·네트워크·데이터베이스·컴퓨터구조·소프트웨어공학·보안). 문제의 **대표 분류**는 대표 개념(concepts 첫 항목)의 카테고리로 **결정적으로 도출**된다. 카테고리는 개념에 귀속되므로 문제·개념 태깅이 곧 분류의 단일 출처다.
- **개념 — 개념 선행 관계:** 방향 있는 그래프(개념 B는 개념 A의 선행). (**MVP 미포함 — 로드맵(디딤 묶음)**. 깊이·순환 금지 여부 등은 **[로드맵 하위 — 디딤 도입 시 결정]**)
- **문제 1 — N 힌트:** 한 문제는 약→강 순서의 힌트를 문제별로 필요한 만큼(0~N개) 가진다(고정 종류·개수 아님).
- **문제 1 — N 오답 교정 힌트:** 각 오답 교정 힌트는 (예상 오답 표기 집합, 교정 메시지)를 가진다. 사용자 제출이 허용답 집합과 불일치하고 어떤 오답 교정 힌트의 예상 오답 집합과 정규화 후 일치하면 자동 표시된다.
- **문제·개념 0..N — 심화 정보:** 문제·개념은 정답 처리 후 노출되는 심화 정보('더 알아보기')를 가질 수 있다(없어도 됨). 콘텐츠로서 승인 게이트 대상. 심화 정보는 제목·리드 문단·보조 항목 카드(순서 보존)·인용의 구조를 가진다([결정 2026-07-16] — 60행 참고).
- **힌트(배경지식) → 선행 개념 → 디딤 문제:** 배경지식(선행 개념) 힌트는 선행 개념을 가리키고, 그 개념의 더 쉬운 문제가 디딤 문제로 실체화된다. (**MVP 미포함 — 로드맵(디딤 묶음)**)
- **사용자 — 개념 숙련도:** 한 사용자는 각 개념에 대해 숙련도와 다음 복습 시점을 누적한다(사용자 N — M 개념). 문제 풀이 결과로 결정적으로 갱신된다.
- **사용자 — 스크랩:** 한 사용자는 임의의 문제를 스크랩(참조)으로 저장한다(사용자 N — M 문제, 학습 상태 귀속). 복습(간격 반복)과 독립이다.
- **복습 문제 선정:** 복습 시점이 도래한 개념에 대해 **이미 푼 그 문제를 다시 고른다.** 예외적으로 **유도형**에 한해, 그 개념을 다루는 아직 안 낸 문제가 있으면 그것을 우선한다(없으면 같은 문제).

## 파고들기 스택 (결정)
- 사용자가 디딤 문제로 내려갈 때마다 현재 문제가 스택에 쌓이고, 디딤 문제를 마치면 팝(pop)되어 직전 문제로 복귀한다. 여러 겹 재귀가 가능하다.
- 파고들기 스택은 **일일 학습 세션 상태의 일부로 영속**된다. 세션 도중 이탈·재접속해도 진행 중이던 파고들기 지점이 보존되어 이어진다(기능 1.5 중단·재개). 세션 완료 후의 스택 보존 여부는 **[로드맵 하위 — 디딤 도입 시 결정]**.
> ⚠️ 이 절 전체는 **MVP 미포함 — 로드맵(디딤 묶음)**. MVP 세션 상태는 진행 중 본 문제만 영속한다.

## 콘텐츠 승인 상태 (결정)
- 모든 콘텐츠(문제·개념·힌트·오답 교정 힌트·허용답 집합·심화 정보)는 승인 상태를 가진다: **초안 → 검수중 → 승인 / 반려.** 상태 전이는 큐레이터가 주도한다.
- **승인 상태의 콘텐츠만 사용자에게 서빙된다.** 승인 전(초안·검수중)·반려 콘텐츠는 어떤 경로로도 노출되지 않는다.
- 승인으로의 전이는 **결정적 구조 검증 통과**를 전제로 한다(허용답 집합 비어있지 않음·개념 태깅 존재·**문제 유형 태깅 존재**·힌트에 정답 비노출 — 콘텐츠 신뢰성 가드레일 참고).
- 서빙된 콘텐츠는 사용자 신고를 받으면 큐레이터가 수정하거나 **회수**(승인 해제 → 서빙 중단)할 수 있다.

# 가드레일
> 아래 두 축의 정책. 콘텐츠 신뢰성은 확정, 비용은 [논의 예정].

## 콘텐츠 신뢰성 가드레일
### 목적
사용자가 배우는 CS 지식이 틀리지 않도록 보장한다. 학습앱에서 잘못된 지식은 면접에서 무너지고 서비스 신뢰를 깨므로, 콘텐츠 정확성을 도메인 정책으로 지킨다. (P4)

### 핵심 전제 (결정)
- **CS 정확성은 결정적으로 검증할 수 없다.** "이 설명이 맞는가"는 문자 대조가 아니라 도메인 판단이 필요하다. 따라서 정확성 판정은 **사람(큐레이터)** 이 맡고, **또 다른 AI로 정확성을 판정하지 않는다**(판정에 비결정성을 들이지 않는 전체 원칙과 일관).

### AI의 역할과 검수 게이트 (결정)
- **AI는 초안 제안자다.** AI는 콘텐츠(문제·허용답·개념·힌트·심화 정보) 초안을 제안해 제작 속도를 높인다. 그러나 스스로 게시하지 못한다.
- **전량 사람 검수 게이트.** 사용자에게 서빙되는 **모든 콘텐츠는 큐레이터 승인을 통과한 것**이다. AI가 초안을 냈든 사람이 처음부터 썼든, 승인 전 콘텐츠는 사용자에게 노출되지 않는다. 이 불변식이 신뢰성의 근거다.
- 콘텐츠는 승인 상태를 가진다: **초안 / 검수중 / 승인(서빙 가능) / 반려.** 승인 상태만 사용자에게 노출된다.

### 결정적 구조 검증 (결정 — 자동)
의미적 정확성(사람 몫)과 별개로, 결정적으로 판정 가능한 구조 무결성은 자동으로 검사해 승인 전에 거른다.
- 문제에 **허용답 집합이 비어 있지 않다**(정답 없는 문제 금지).
- 문제에 **개념 태깅이 하나 이상 존재한다**(힌트 분기·복습의 근거 확보).
- 문제에 **문제 유형 태깅이 존재한다**(정의 재생형 / 유도형 — 근접 신호 적용 여부·복습 문제 선정의 근거 확보).
  > 추가 근거: 유형은 표시용 메타데이터가 아니라 **런타임 분기의 입력**이다. 근접 신호를 줄지(기능 1)와 복습에 같은 문제를 낼지(기능 3)가 유형으로 갈리므로, 유형이 없는 문제는 두 규칙의 동작이 정의되지 않는다. '개념 태깅 존재'를 검증하는 근거("힌트 분기·복습의 근거 확보")와 정확히 같은 형태이며, 유형은 두 값 중 하나의 존재 여부라 **결정적으로 판정 가능**하다 — 자동 구조 검증의 자격 요건(의미 판단이 아닌 구조 무결성)을 만족한다.
- **힌트·오답 교정 힌트가 정답을 노출하지 않는다**(정답 문자열 비포함 — 학습 효과 보존).
- **콘텐츠의 코드는 모바일 폭에서 읽힌다(큐레이션 가이드).** 문제·힌트·심화 정보의 코드 스니펫은 모바일 세로 화면에서 가로 스크롤 없이 이해되도록 짧은 줄로 작성한다. 컨텍스트가 끊기지 않게 하는 원칙이며, 과도한 줄 길이는 지양한다(리뷰 반영). (자동 소프트 체크 여부는 구현/[열린 질문])
- (그 밖의 구조 규칙·정규화 세부는 구현 노트로 분리.)

> 결정 근거: Resumaker가 "결정적으로 판정 가능한 것만 자동 검증하고, 자유 서술은 생성 지시·사람 검토로 다룬다"고 한 것과 동형이다. CS 정확성은 자유 서술에 해당하므로 사람 검수로, 구조 무결성은 결정적이므로 자동으로 나눈다.

### 사용자 오류 신고 (결정 — MVP 포함)
- 검수 게이트를 통과해 서빙된 콘텐츠에서도 오류가 발견될 수 있다. 사용자는 그 콘텐츠를 **신고**할 수 있고, 큐레이터가 확인해 **수정하거나 회수(서빙 중단)** 한다. 게이트를 뚫은 오류의 안전망이자 지속적 품질 장치.
- **[반영 2026-07-19: D10]** 신고는 사용자의 학습 상태가 아니라 **콘텐츠 품질 운영 데이터**다 — 계정·데이터 소유(기능 4)의 '학습 상태 삭제' 대상에 포함되지 않으며, 계정 삭제 시에도 신고자 식별자만 익명화해 보존한다(품질 신호 유지).
- **[반영 2026-07-19: D14] 회수된 문제의 신고 게이트.** 신고는 **현재 승인(APPROVED) 상태인 콘텐츠만** 수락한다 — 진행 중 세션에 이미 배정된 문제는 회수 후에도 계속 풀 수 있지만(스냅샷 서빙, 열린 질문 10 부분 확정 참고), 그 문제가 회수된 뒤의 신고는 수락하지 않는다(MVP는 이를 수용 — 회수는 대개 큐레이터가 이미 오류를 인지한 결과이므로). Phase 2 신고 콘솔 설계 시 "사용자가 세션·스크랩으로 보유한 문제면 신고를 수락"하는 방식을 재검토한다(**[반영 2026-07-19: D6]** '추가 학습'은 폐지되어 세션에 흡수됨).
- (신고 누적에 따른 자동 임시 숨김 등 운영 정책은 [열린 질문].)

### 사용자 시나리오
- **성공(검수 통과분만 노출):** 사용자는 항상 큐레이터 승인을 통과한 문제·힌트만 만난다. AI가 초안을 낸 문제라도 사람이 정확성을 확인한 뒤에야 서빙된다.
- **성공(오류 신고·회수):** 사용자가 한 문제의 설명 오류를 신고하면, 큐레이터가 확인해 수정하거나 그 콘텐츠를 회수한다. 이후 그 문제는 고쳐진 형태로 다시 서빙되거나 더는 나오지 않는다.
- **실패(초안 유출 차단):** 승인되지 않은 초안·반려 콘텐츠는 어떤 경로로도 사용자에게 서빙되지 않는다.

### 수용 기준
- 사용자에게 서빙되는 모든 콘텐츠는 큐레이터 승인 상태다(초안·검수중·반려 콘텐츠는 서빙되지 않는다).
- 승인 전 자동 구조 검증을 통과하지 못한 콘텐츠(허용답 없음·개념 태깅 없음·**문제 유형 태깅 없음**·힌트에 정답 노출)는 승인·서빙되지 않는다.
- 정확성 판정에 LLM 자동 판정을 쓰지 않는다(사람 검수).
- 사용자는 서빙된 콘텐츠를 신고할 수 있고, 신고된 콘텐츠는 큐레이터가 수정·회수할 수 있다.

## 비용 가드레일
### 목적
AI 호출 비용과 어뷰징을 도메인 정책으로 관리한다. 다만 ByteCS는 런타임 구조상 그 성격이 Resumaker와 근본적으로 다르다.

### 핵심 성격 (결정)
- **런타임에 사용자 트리거 AI 비용이 없다.** 문제 풀이·정답 판정·힌트·오답 교정 힌트·복습은 전부 결정적이거나 큐레이션이라, 사용자가 쓰는 순간 AI를 호출하지 않는다.
- AI가 관여하는 곳은 **콘텐츠 초안 제작(큐레이터 측)** 뿐이며, 사용자가 트리거하지 않고 사람 검수 게이트가 자연히 상한을 정한다. 즉 이 비용은 사용자 어뷰징 벡터가 아니라 **제작(운영) 측 내부 비용**이다.

### 정책 (결정)
- **MVP엔 사용자 단위 비용 쿼터(일일 생성 한도 등)를 두지 않는다.** 사용자 트리거 AI 비용이 0이므로 Resumaker식 쿼터가 불필요하다(과설계 회피).
- **MVP는 전면 무료.** 게스트 시작·부담 제거 철학과 정합한다.
- **제작 측 AI 비용**은 도메인 규칙이 아니라 운영 예산으로 다룬다(사람 검수 처리량에 묶임).
- **조건부 미래 규칙:** 로드맵의 **AI 실시간 힌트** 등 런타임 사용자 트리거 AI가 도입되면, 그 시점에 Resumaker식 사용자 단위 쿼터(카운트 대상·차감 시점·리셋 경계는 도메인으로 고정, 수치는 운영)를 도입한다.

### 수용 기준
- MVP에서 어떤 사용자 동작도 AI 호출로 이어지지 않으며, 사용자 단위 사용량 제한이 존재하지 않는다.
- 전면 무료로, 유료 등급·결제 흐름이 존재하지 않는다.

# MVP 경계와 로드맵

## MVP에 포함 (이번 단계에서 구현)
- 개념을 일급으로 둔 문제 풀이(주관식 단답, 문제↔개념 태깅, **문제 유형 — 정의 재생형·유도형 모두**, 난이도).
- 결정적 허용답 대조 판정 + **근접(오탈자) 신호(정의 재생형에만)** + 무낙인·사용자 주도 흐름(무제한 재시도·[정답 보기]·**정답 공개 후 따라 입력**).
- **문제별 자유 구성 힌트**(학습적·약→강, 큐레이션). ⚠️ 디딤 문제 연결·파고들기 스택은 **로드맵**(아래 참고).
- 오답 교정 힌트(오답 트리거·자동).
- **정답 후 심화 정보**('더 알아보기', 선택·큐레이션).
- 일일 학습 세션(오늘의 한입 — 문제 분량 기반·자동 배정·복습 편입·중단 재개·**지난 문제 다시 보기**).
- 복습·정착(개념 숙련도 기반 간격 반복, **같은 문제 재출제** — 유도형만 예외).
- **문제 스크랩**(개인 북마크).
- **문제 카테고리 1차**(기능 7 — 8개 대분류·대표 분류 표시(풀기 전부터)·카테고리별 학습 이력 필터). 관심 기반 출제 가중치는 2차(로드맵).
- **지속·성취 동기**(완결 성취·스트릭 등 긍정 동기 — 강제·죄책감 배제).
- 계정·데이터 소유(게스트 시작 + 가입 승계, 학습 상태 사용자 소유·격리·삭제).
- 콘텐츠 신뢰성 가드레일(AI 초안 + 전량 사람 검수 게이트 + 결정적 구조 검증 + 사용자 신고).
- 비용 정책(런타임 사용자 트리거 AI 없음 → 사용자 쿼터 없음, 전면 무료).

## 로드맵 (MVP 이후)
- **디딤 문제 연결(개념 매개) + 파고들기 스택 + 개념 간 선행 관계 그래프.** 셋은 한 묶음이다 — 선행 간선 없이 디딤은 성립하지 않는다.
  > MVP에서 뺀 근거: 디딤은 "선행 개념을 익히면 본 문제가 풀린다"를 전제하는데, **정의 재생형**에서는 선행 개념을 익혀도 그 **이름은 나오지 않는다**. 디딤이 값어치를 하는 것은 **유도형**이다 — 선행 개념을 익히면 유도의 재료가 채워져 본 문제에 실제로 다가간다. 또한 선행 간선은 복습과 공유되지 않는 디딤 전용 비용이다(가치 흐름 '개념이 축이다' 참고).
  > **[갱신 — 문제 유형 확정]** 이 항목에 달려 있던 "문제 유형 구성이 확정되면 재검토한다 — [열린 질문]"은 **해소됐다.** 유형이 확정되어 **유도형이 MVP에 포함**되므로(기능 1 문제 유형), 디딤이 값어치를 하는 유형은 이미 존재한다. 즉 **디딤을 뺀 근거 중 '유형' 축은 사라졌고**, 남은 근거는 **선행 간선의 비용**(복습과 공유되지 않는 디딤 전용 구조 — 개념 그래프·순환 금지·깊이 정책 일체)뿐이다. 디딤은 여전히 로드맵이되, 그 이유는 이제 "유도형이 없어서"가 아니라 "**선행 간선의 비용이 MVP 범위를 넘어서**"다. 재검토 시점은 유형 확정이 아니라 **선행 간선을 감당할 콘텐츠·큐레이션 여력이 생길 때**다.
- **맞힌 개념 마인드맵**(개념 그래프 시각화로 학습 지도를 연결해 보기).
- **주제 선택·약점 집중 세션**(원하는 주제만·약한 개념만 풀기 — MVP는 자동 배정).
- **관심 카테고리 기반 출제 가중치(문제 카테고리 2차, 기능 7).** 관심 분야를 새 개념 문제 배정에 가중치로 반영(비관심도 소량 유지·완전 제외 아님, 복습은 무관). 1차의 카테고리 표시·이력 필터 위에 얹는다. (구체 가중치 수치는 2차 구현/운영 판단.)
- **개인 수준 맞춤 적응 출제**(숙련도 기반 난이도·문제 개인화 — MVP는 간격 반복·약점 우선 편입까지).
- **통합·시스템형 지식 문제**(예: "구글에 검색하면 무슨 일이?" 같은 다개념 연결 — 단답 유지 전제의 다개념 연결형. 서술형은 미도입).
  > 새 유형이 아니라 **유도형의 확장**이다(기능 1 문제 유형). 유도형은 이미 MVP에 있고 여러 단계를 거쳐 답을 도출하지만, MVP의 유도형은 대체로 **한 개념 안에서** 유도한다(코드 → 시간 복잡도, 서브넷 → 호스트 수). 로드맵의 통합·시스템형은 **여러 개념을 가로질러** 유도하는 것으로, 문제 유형 축이 아니라 **개념 연결의 폭**을 넓히는 축이다. 답변 형식(주관식 단답)은 그대로 유지된다.
- AI 실시간 힌트(MVP는 큐레이션 고정) — 도입 시 사용자 단위 비용 쿼터를 함께 도입.
- 오답 교정 힌트의 개념·디딤 연결(혼동 개념 → 더 쉬운 문제).
- 간격 반복 공식·개인화의 고도화(MVP는 기본 간격 반복을 이미 포함).
- **앱 밖 재접속 유도**(푸시 알림·리마인더 — MVP는 앱 내 긍정 동기까지, 긍정 톤 유지).
- 게스트 상태의 서버 익명 보관(MVP는 가입 시 기기 로컬 → 계정 승계).
- **인증 남용 방어 강화**(레이트 리밋·로그인 실패 락아웃 — 크리덴셜 스터핑·무차별 대입 방어). MVP는 로그인 응답코드·타이밍 기반 계정 열거만 방어하고 요청 빈도 제한은 두지 않는다(인프라 결정 수반 — Bucket4j·게이트웨이·Redis 등). 저장 토큰 암호화(EncryptedSharedPreferences·Keychain)도 이 축의 후속.
- 유료화·수익 모델(MVP는 전면 무료).
- 모바일/웹 클라이언트 확장 등.

> 경계 근거: MVP는 (1) 5분 세션 안에서 가치가 완결되고(복습에 도달하지 않아도 만남→막힘 넘기→풀이 루프가 닫힘), (2) 힌트 → 정답 공개 → 따라 입력 → 복습 재노출로 '포기'를 막아 진입 장벽을 제거하며, (3) 간격 반복으로 정착을 잇는 핵심 루프에 집중한다. 콘텐츠 확장(AI 실시간)·디딤 묶음(디딤 문제·파고들기 스택·선행 그래프)·개인화·수익화는 본질 가치가 검증된 뒤 더한다.

# 측정 가능한 수용 기준 (전체 요약)
개발자가 명세만으로 pass/fail을 판단할 수 있도록, 각 기능의 핵심 기준을 모은다.

1. 문제 풀이: 제출한 답이 문제의 허용답 집합과 정규화 후 일치하면 정답, 아니면 불일치로 결정적으로 판정된다(같은 입력 → 같은 판정, LLM 없음).
2. 문제 풀이: 불일치는 오답으로 확정되지 않으며, 사용자는 [정답 보기]를 요청하기 전까지 횟수 제한 없이 재시도할 수 있다.
3. 문제 풀이: 모범답안은 사용자가 [정답 보기]를 요청한 경우에만 공개되고, 그 사실이 복습·숙련도 신호로 기록된다.
4. 힌트: 한 문제의 힌트는 약→강 레벨 순으로만, 사용자의 명시적 요청('더 보기')으로만 공개된다.
5. ~~힌트: 배경지식 보충 힌트가 가리키는 선행 개념에는 그 개념의 더 쉬운 디딤 문제로 이어지는 경로가 있고(개념 매개), 디딤 문제를 마치면 직전 문제로 복귀한다(여러 겹이면 역순).~~ **[MVP 미포함 — 로드맵(디딤 묶음)]**
6. 오답 교정 힌트: 제출 답이 허용답과 불일치하되 예상 오답 집합과 정규화 후 일치하면 교정 힌트가 자동 표시되며, 그래도 오답으로 확정되지 않고 정답을 노출하지 않는다.
7. 세션: 세션은 배정된 본 문제 분량을 다 마치면 완료로 전이되고, 시간 경과만으로는 종료되지 않는다.
8. 세션: 세션 완료 카운트에 재시도·힌트 열람은 포함되지 않는다(본 문제만). (디딤 문제도 제외 — 디딤 도입 시 유효, MVP 미포함)
9. 세션: 세션 상태(진행 중 본 문제)는 이탈·재접속에도 보존되어 이어진다. (파고들기 스택 보존은 디딤 도입 시 — MVP 미포함)
10. 복습: 각 사용자·개념 쌍은 숙련도와 다음 복습 시점을 가지며 풀이 결과로 결정적으로 갱신된다(같은 이력 → 같은 결과, LLM 없음).
11. 복습: 스스로 맞힘/도움받아 맞힘/포기·오답이 숙련도에 서로 다르게 반영된다.
12. 복습: 복습 시점이 도래한 개념은 이후 세션에 편입되고, 복습 문제는 **이미 푼 그 문제를 다시 낸다.** 단 **유도형**에 한해, 그 개념에 아직 안 낸 다른 문제가 있으면 그것을 우선한다(없으면 같은 문제).
13. 계정: 가입 없이도 문제 풀이·힌트·복습이 동작하고 게스트 학습 상태가 쌓이며, 가입 시 그 상태가 유실 없이 계정으로 승계된다.
14. 계정: 한 사용자의 학습 상태는 다른 사용자에게 노출되지 않고, 재로그인 후 보존되며, 계정 삭제 후 전부 조회되지 않는다(공용 콘텐츠는 무관).
15. 콘텐츠 신뢰성: 사용자에게 서빙되는 모든 콘텐츠는 큐레이터 승인 상태다(초안·검수중·반려는 서빙되지 않음).
16. 콘텐츠 신뢰성: 자동 구조 검증(허용답 존재·개념 태깅 존재·**문제 유형 태깅 존재**·힌트에 정답 비노출)을 통과하지 못한 콘텐츠는 승인·서빙되지 않으며, 정확성 판정에 LLM을 쓰지 않는다.
17. 콘텐츠 신뢰성: 사용자는 서빙된 콘텐츠를 신고할 수 있고, 신고된 콘텐츠는 큐레이터가 수정·회수할 수 있다.
18. 비용: MVP에서 어떤 사용자 동작도 AI 호출로 이어지지 않으며, 사용자 사용량 제한·유료 결제가 존재하지 않는다.
19. 문제 풀이: 근접(오탈자) 신호는 **정의 재생형** 문제의 결정적 편집거리 임계 내 불일치에서만 표시되고(유도형에는 표시되지 않는다) 정답·개념을 노출하지 않으며, 정답 공개 후에는 정답을 직접 입력해야 다음으로 진행된다(입력이 허용답 집합과 일치하면 통과 — 화면의 대표 정답 문자와 완전히 같을 필요는 없다. **[반영 2026-07-19: C6]**).
20. 심화 정보: 정답 처리(맞힘 또는 정답 공개) 이후에만 선택적으로 노출되고 진행을 막지 않으며, 승인된 것만 서빙된다.
21. 힌트: 힌트는 문제별로 개수·종류가 다를 수 있고 약→강 순으로 공개되며 정답을 노출하지 않는다.
22. 스크랩·지속 동기: 사용자는 문제를 스크랩·재열람할 수 있고(사용자 소유·격리), 지속 동기 요소는 긍정 신호만 쓰며 학습을 강제하지 않는다.
23. 문제 풀이: 모든 문제는 **정의 재생형**·**유도형** 중 하나의 문제 유형을 가지며(답변 형식과 독립된 축), 유형 태깅이 없는 콘텐츠는 자동 구조 검증에서 걸려 승인·서빙되지 않는다.
24. 세션 일원화 **[반영 2026-07-19: D6 — 이전 '추가 학습' 기준을 대체]:** 하루 둘째 이후 세션('조금 더 풀기'로 생성)도 첫 세션과 완전히 동일한 규칙을 따른다 — 결정적 판정·무낙인·무제한 재시도(기준 1·2), 모범답안은 요청 시에만 공개되고 공개 후 따라 입력해야 진행(기준 3·19), 근접 신호는 정의 재생형 한정(기준 19), 심화 정보는 정답 처리 후에만 노출(기준 20), 풀기 전에는 정답·개념·해설·심화·미공개 힌트를 노출하지 않는다(no-leak). 게스트 포함 인증이 강제된다.
25. 세션 일원화 **[반영 2026-07-19: D6]:** 새 세션의 새 개념 선정에서 제외되는 '푼 문제' 풀은 사용자가 그날까지 만든 **모든 세션**의 풀이 이력이다(추가 학습 폐지로 별도 활동과의 합집합 규정이 불필요해졌다).
26. 스트릭 하루 멱등 **[반영 2026-07-19: D6]:** 스트릭은 그날 **첫 세션 완료**로만 기록되며, 같은 날 두 번째 이후 세션 완료는 스트릭을 중복 기록하지 않는다. 각 세션은 완료 시 완료 카운트·완결 화면을 동일하게 받는다.
27. 문제 카테고리 **[결정 2026-07-17]:** 모든 개념은 8개 대분류(자료구조·알고리즘·운영체제·네트워크·데이터베이스·컴퓨터구조·소프트웨어공학·보안) 중 하나에 귀속되고, 문제의 대표 분류는 대표 개념의 카테고리로 결정적으로 도출되며(대표 분류 원칙), 대표 분류는 풀기 전부터 표시된다. 사용자는 푼 문제를 카테고리별로 조회할 수 있다(1차, **[반영 2026-07-19: D6]** 사용자의 전체 세션 풀이 이력 기준 — 추가 학습 폐지로 합집합 규정 불필요). 문제 없는 카테고리는 '준비 중'으로 노출된다.
28. 문제 카테고리 **[결정 2026-07-17]:** (2차) 관심 카테고리 설정은 새 개념 문제 배정에 가중치로만 반영되고 복습 편입에는 영향을 주지 않으며, 비관심 카테고리도 완전히 배제되지 않는다(소량 유지).

# 열린 질문
정말로 서비스 오너만 결정할 수 있는 것들을 모은다. (대화가 진행되며 추가·정리된다.)

1. **힌트 구성 큐레이션 가이드:** 어떤 문제에 어떤 힌트를 몇 개 넣을지의 기준(학습적 가치·문제 성격). (도메인은 '문제별 자유 구성·학습적이어야 함'만 확정)
2. ~~**근접 신호 편집거리 임계:**~~ **[결정 2026-07-19: D11]** 현행 구현을 이 열린 질문의 답으로 확정 기록한다 — **길이 3자 미만은 근접 신호 없음, 7자 이하는 편집거리 1을 오타로 판정, 8자 이상은 편집거리 2까지 오타로 판정**한다(정의 재생형의 개념 이름 표기만 고려하면 되므로 수식·숫자 답을 깨뜨릴 걱정은 없다). 긴 개념 이름에서 드물게 거짓 근접이 발생할 수 있으나, 문제가 관찰되면 그때 재튜닝한다. (**거짓 오답 완충** — 유효답이 허용답 집합에 누락된 경우의 보완 여부·방식 — 은 별도 사안으로 남는다.)
3. **정답 판정 완전성의 큐레이션 기준:** 허용답 집합에 어디까지(유의어·약어·영/한 표기) 등재할지의 큐레이션 가이드라인.
4. ~~**오답 교정 힌트의 개념·디딤 연결:**~~ 오너 판단이 아니라 디딤 도입에 종속된 파생 결정이므로 열린 질문에서 뺀다. **로드맵(디딤 묶음) 하위** — 로드맵 절 '오답 교정 힌트의 개념·디딤 연결' 항목 참고.
5. ~~**세션 크기(본 문제 수):**~~ **[결정]** 기본값 **5개**, 사용자가 설정에서 조정 가능(1~50, 사용자 소유 설정). 도메인의 '고정 소량'을 기본 5로 구체화하되 개인화 여지를 둔다. **[반영 2026-07-19: D3]** "5분" 카피 vs 기본값 10개의 괴리를 해소하기 위해 10에서 하향.
6. ~~**하루 여러 세션 허용 여부:**~~ **[결정 2026-07-19: D6]** 하루 여러 세션을 허용한다 — 완료 후 "조금 더 풀기"는 새 세션을 시작하며, 이전의 '추가 학습'(별도 자유 풀이 활동)은 폐지되어 세션으로 일원화되었다. 스트릭은 그날 첫 세션 완료로만 기록되는 하루 멱등을 유지한다(위 '완료' 절 참고).
7. **간격 반복 공식·숙련도 가중치:** 다음 복습 간격을 늘리고 줄이는 구체 공식과, 스스로 맞힘/도움받아 맞힘/포기·오답이 숙련도에 얼마씩 반영되는지의 수치. (도메인은 '간격 반복·신호 구분'만 확정, 공식은 구현/운영 판단) **[기록 2026-07-19: O1]** 현재 구현의 레벨 클램프 경계에서 신호 구분이 붕괴하는 관찰이 기능 3 '개념 숙련도 산정' 절에 남아 있다 — 이 항목 확정 시 함께 검토.
8. **복습 편입 비율·상한:** 한 세션에서 복습 문제와 새 개념 문제를 섞는 비율, 복습이 도래한 개념이 많을 때의 상한. (도메인은 '복습 우선 편입'만 확정)
9. ~~**게스트 학습 상태 보관 방식:**~~ **[결정]** 게스트도 **서버 User(role=GUEST)로 발급**해 학습 상태를 서버에 보관한다. 가입은 그 게스트 User를 회원으로 **in-place 승격**하여 승계 유실을 없앤다. 기기에는 게스트 토큰만 보관. (인증: Spring Security + JWT, 이메일·비밀번호 BCrypt — 소셜은 로드맵)
10. **콘텐츠 변경과 학습 상태 정합성:** 공용 콘텐츠(문제·개념)가 수정·삭제·회수될 때 이를 참조하는 학습 상태(숙련도·복습 스케줄·복습 폴백)를 어떻게 정합시킬지. (콘텐츠 신뢰성·운영과 함께 판단)
11. **신고 누적 자동 처리:** 사용자 신고가 일정 수 이상 쌓인 콘텐츠를 큐레이터 확인 전 자동 임시 숨김 처리할지. (도메인은 '신고 → 큐레이터 수정·회수'만 확정)
12. **수익 모델·유료화 경계:** MVP 이후 유료화 여부와 경계(무엇을 유료로 둘지). (도메인은 'MVP 전면 무료'만 확정, 이후는 오너의 수익 모델 판단)
13. **심화 정보 제공 범위:** 모든 문제에 심화 정보를 붙일지, 일부만 큐레이션할지의 정책.
14. **스크랩 개수 상한·정리:** 스크랩 보관 개수 상한 여부와 초과 정리 정책.
15. **지속 동기 규칙:** 스트릭 산정·성취 기준 등 구체 규칙(긍정 톤 유지 전제).
16. **통합·시스템형 지식의 문제 형태:** 단답 유지 전제에서 다개념 연결형을 어떻게 구성할지.
17. **문항–답 연합의 실제 발생 정도:** 같은 문제를 복습으로 재출제할 때, **유도형** 문제에서 문항–답 연합이 실제로 얼마나 생기는지는 미검증이다. 관측 후 **유도형 예외**의 범위를 조정할지 판단한다. (도메인은 '예외 존재'만 확정)
