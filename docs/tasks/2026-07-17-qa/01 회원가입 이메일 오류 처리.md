# 01. 회원가입 이메일 오류 처리 수정 (QA #5)

## 배경
"aaa@aaa"로 회원가입하면 서버는 400 + "이메일 형식이 올바르지 않습니다."를 정상적으로 돌려주는데, 클라이언트가 이를 "잠시 연결이 원활하지 않았어요. 다시 시도해 주세요."로 표시한다. 사용자는 검증 실패를 서버 연결 실패로 오인한다.

## 원인 (확인 완료)
서버는 정상이다. 문제는 클라이언트 3지점의 중첩이다.

1. **클라 사전 검증이 느슨함** — `AuthUiState.isEmailValid`(`app/shared/src/commonMain/kotlin/watson/bytecs/account/AuthViewModel.kt:140-141`)가 `@` 포함 여부만 검사. 서버 `Email` VO(`server/src/main/kotlin/watson/bytecs/account/domain/Email.kt:12`)는 `^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$`로 TLD를 요구 → "aaa@aaa"가 클라를 통과해 서버까지 감.
2. **400 미번역** — `KtorAccountRepository.translating()`(`app/shared/.../account/data/KtorAccountRepository.kt:82-91`)이 409→`EmailAlreadyInUseException`, 401→`InvalidCredentialsException`만 매핑하고 나머지는 원본 `ResponseException`을 그대로 던짐.
3. **일괄 폴백** — `AuthViewModel.friendlyMessage()`(AuthViewModel.kt:112-116)의 `else` 분기가 미분류 예외 전부를 연결 오류 문구로 처리.

## 변경 사항

### 클라이언트
1. `AuthUiState.isEmailValid`의 검증을 서버 `Email` VO와 동일한 정규식으로 강화한다.
   - 서버(server 모듈)와 클라(app 모듈)는 코드 공유가 없으므로 정규식이 중복된다. 양쪽에 서로를 가리키는 주석을 남겨 동기화 지점을 명시할 것(예: "서버 Email.kt의 EMAIL_PATTERN과 반드시 동일하게 유지").
2. `KtorAccountRepository.translating()`에 400 분기를 추가한다.
   - 응답 바디의 `ErrorResponse`(message, errorCode)를 파싱해 새 도메인 예외(예: `InvalidInputException(message)`)로 번역한다. 기존 409/401 번역이 바디를 다루는 방식이 있으면 그 패턴을 따른다.
3. `AuthViewModel.friendlyMessage()`에 해당 예외 분기를 추가해 서버 메시지("이메일 형식이 올바르지 않습니다.") 또는 그에 준하는 친절 문구를 표시한다. `else` 폴백(연결 문구)은 진짜 네트워크 예외용으로 유지.

### 서버 (테스트 보강만)
4. `AccountControllerIntegrationTest`에 "잘못된 이메일 형식 → 400 + INVALID_INPUT" 케이스가 없다. 이 케이스를 추가해 현 동작을 사양으로 고정한다. (프로덕션 코드 변경 없음)

## 테스트
- `AuthViewModelTest`: 잘못된 형식 이메일이면 `canSubmit=false`(사전 차단), 400 번역 예외 → 형식 오류 문구 케이스.
- `KtorAccountRepositoryTest`: 400 응답 → `InvalidInputException` 번역 케이스.
- `LoginScreenUiTest`: 형식 오류 문구 노출 케이스(기존 패턴 따라).
- 서버: 위 통합 테스트 케이스.

## 커밋
- `fix: 회원가입 이메일 형식 오류가 연결 실패로 표시되는 문제 수정` (클라 1~3 + 테스트)
- `test: 잘못된 이메일 형식 가입 요청의 400 응답 통합 테스트 추가` (서버)

## 검증
- 클라 `jvmTest`·`commonTest`, 서버 `test` 전체 통과.
- 수동: "aaa@aaa" 입력 시 제출 전 차단(또는 형식 안내), 강제 전송 시나리오에서도 형식 오류 문구 표시. "aaa@aaa.com"은 정상 가입.
