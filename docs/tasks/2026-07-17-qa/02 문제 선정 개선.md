# 02. 문제 선정 개선 — 일일 세션 랜덤화 + 추가 연습 반복 버그 (QA #6, #7)

## 배경
- **QA #6:** 일일 세션의 새 개념 문제가 랜덤이길 원함(이미 푼 문제 제외는 유지). 현재는 id 오름차순 고정이라 모든 사용자가 같은 순서로 같은 문제를 받는다.
- **QA #7:** 일일 할당량 완료 후 '조금 더 풀기'(추가 연습)를 누르면 **항상 같은 문제 하나만 반복 제공**되는 버그. 원인 확정: `GET /api/problems/next`가 stateless로 `findFirstByOrderByIdAsc()`(항상 최소 id 문제)를 반환한다.

## 현재 구조 (확인 완료)
- **일일 세션 배정:** `SessionCreator.assignProblemIds()`(`server/src/main/kotlin/watson/bytecs/session/application/SessionCreator.kt:54-84`)
  - ① 복습 도래 문제 우선(`ReviewService.selectDueReviewProblemIds`) → ② 남은 칸을 **안 푼 문제 id 오름차순**으로 → ③ 다 풀었으면 전체 풀 폴백. 52행 주석이 "같은 학습 상태면 항상 같은 세션"이라는 결정성을 명시.
- **추가 연습:** 클라 `ProblemViewModel.loadNext()` → `GET /api/problems/next` → `ProblemService.findNextProblem()`(`.../problem/application/ProblemService.kt:23-28`) → `ProblemRepository.findFirstByOrderByIdAsc()`(`.../problem/infrastructure/ProblemRepository.kt:10-11`). 사용자 컨텍스트 없음.
- **도메인 명세:** §1.5는 "서비스가 자동 배정, 새 개념+복습을 섞음"만 규정(랜덤 여부 미규정), 추가 연습은 "자유 풀이"로만 규정(선정 규칙 공백). → 이번 변경은 명세 공백을 채우는 개정을 동반한다.

## 변경 사항

### A. 일일 세션 랜덤화 (QA #6)
1. `assignProblemIds()`의 ②단계에서 안 푼 문제 목록을 **셔플 후** 채운다. ③ 전체 풀 폴백도 셔플.
   - 복습 우선 배치(①)와 `LinkedHashSet` 중복 제거·절단 구조는 그대로 유지.
   - 52행의 결정성 주석을 새 정책("복습 우선 + 새 문제는 무작위")으로 수정.
2. **테스트 가능성:** `kotlin.random.Random`을 `SessionCreator`에 주입(생성자 파라미터, 기본값 `Random.Default`)해 테스트에서 시드 고정으로 결정적 검증이 가능하게 한다.

### B. 추가 연습 선정 로직 재작성 (QA #7)
3. `GET /api/problems/next`에 인증 사용자 컨텍스트를 추가한다(세션 API들이 쓰는 `user.userId` 주입 패턴을 따를 것 — `SessionController` 참고).
4. `ProblemService.findNextProblem(userId)` 선정 규칙:
   - 1순위: **아직 풀지 않은 문제** 중 무작위 1건.
   - 2순위(안 푼 문제가 없으면): **복습 도래 개념의 문제** 중 무작위 1건(`ReviewService` 재사용 검토).
   - 3순위(둘 다 없으면): 전체 문제 중 무작위 1건.
   - 직전 제공 문제와의 즉시 중복은 가능하면 회피(전체 1건뿐인 극단 케이스는 허용).
   - "푼 문제" 판정은 세션 배정 로직이 쓰는 `findSolvedProblemIds()` 계열 쿼리를 재사용.
5. `findFirstByOrderByIdAsc()`와 그 "최소 id 반환" 주석·테스트는 제거 또는 대체한다(기존 `ProblemServiceTest`/`ProblemControllerIntegrationTest`가 이 동작을 사양으로 고정하고 있으므로 반드시 갱신).
6. 무작위 추출은 DB `ORDER BY random()`(PostgreSQL) 또는 "후보 id 조회 후 애플리케이션에서 무작위 선택" 중 후자를 권장 — A와 동일하게 `Random` 주입으로 테스트 가능하고, 후보 수가 수백 건 수준(현재 107문제)이라 성능 문제가 없다.

### C. 도메인 명세 개정
7. `docs/도메인 명세서.md` §1.5:
   - 문제 선정(202~205행 부근)에 "새 개념 문제는 아직 풀지 않은 문제 중 무작위로 배정한다" 명시.
   - 완료/추가 연습(217행 부근)에 추가 연습 선정 규칙(위 4번의 1~3순위) 명시.

## 테스트
- `SessionCreatorTest`: 시드 고정 Random으로 ① 복습 우선 유지, ② 안 푼 문제 무작위 배정, ③ 같은 학습 상태라도 시드가 다르면 다른 세션 가능함을 검증. 기존 결정성 검증 테스트는 새 정책에 맞게 수정.
- `ProblemServiceTest` / `ProblemControllerIntegrationTest`: 1~3순위 선정 규칙, 이미 푼 문제 제외, 반복 호출 시 같은 문제만 고정 반환되지 않음.
- 클라 변경은 없어야 함(응답 스키마 동일). `ProblemViewModel` 기존 테스트 통과 확인만.
