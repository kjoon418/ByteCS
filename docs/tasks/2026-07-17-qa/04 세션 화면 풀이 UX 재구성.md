# 04. 세션 화면 풀이 UX 재구성 (QA #1, #2, #3, #4)

## 배경
키보드가 올라오면 가용 화면 높이가 크게 줄어든다. 세션 화면(문제 풀이)에 최소한의 요소만 남기도록 재배치하고, 오답 피드백·정답 보기 흐름을 직관적으로 바꾼다.

주 파일: `app/shared/src/commonMain/kotlin/watson/bytecs/session/SessionScreen.kt`(이하 라인은 이 파일), `SessionViewModel.kt`, `ui/components/FeedbackComponents.kt`, 테스트 `app/shared/src/jvmTest/kotlin/watson/bytecs/session/SessionScreenUiTest.kt`. 서버는 5번 항목만 해당.

6개 커밋으로 나눠 순서대로 진행한다. 각 커밋 후 `jvmTest`(+ 5번은 서버 `test`) 통과 상태를 유지할 것.

---

## 1) 제출 버튼 텍스트 변경 (QA #2)
- 현재: 바텀바 `PrimaryButton` 분기(199~203행)에서 미해결 상태 텍스트가 "정답 확인하기" — 제출 버튼인지 정답 열람 버튼인지 헷갈린다.
- 변경: **"제출하기"** 로 교체. "한입 마치기"/"다음 문제" 분기는 유지.
- 테스트: `빈_입력으로는_제출되지_않는다`(333행) 등 해당 문자열 단언 갱신.
- 커밋: `fix: 제출 버튼 텍스트를 '제출하기'로 변경`

## 2) '오류 신고'를 탑바로 이동 (QA #1)
- 현재: 문제 본문 아래 `Row` 안 `TextLink("오류 신고")`(360~368행).
- 변경: 탑바 `Row`(164~188행)의 "나가기" `TextLink` 옆(왼쪽)으로 이동. `contentDescription`("이 문제의 콘텐츠 오류 신고")과 `onReport(state.problem.id)` 콜백 유지. 본문 쪽 Row는 제거.
- 주의: 탑바에는 `SessionProgressEntry`·나가기가 이미 있다. 좁은 화면에서 겹치지 않게 간격 확인.
- 테스트: `문제_풀이_중에도_오류_신고로_들어갈_수_있다`(599행) 유지 통과 확인.
- 커밋: `feat: 오류 신고 진입점을 탑바로 이동`

## 3) '다시 볼래요'(스크랩)를 난이도 태그 행으로 이동 (QA #1)
- 현재: `Text("다시 볼래요")` + `ScrapToggle`(461~474행)이 `if (state.solved || state.reveal != null)` 게이트(459행) 안에 있어 정답 접근 후에만 보인다.
- 변경: 난이도 태그 행(345~348행, `DifficultyIndicator`)을 `Row`로 만들어 왼쪽에 난이도, 오른쪽 끝에 "다시 볼래요"+`ScrapToggle`을 배치한다. **게이트를 제거해 풀이 중에도 스크랩 가능**하게 한다.
  - 근거: 스크랩은 개인 북마크일 뿐 정답을 노출하지 않는다. 도메인 명세 §5도 스크랩 시점을 제한하지 않는다(정답 유출 가드레일과 무관).
- 낙관적 토글 상태(`mutableStateOf` 맵, 106행 부근)와 `onToggleScrap` 흐름은 그대로.
- 테스트: 스크랩 게이트를 검증하던 기존 케이스를 "풀이 중에도 스크랩 가능"으로 수정. 미해결 상태에서 정답 비노출(no-leak) 단언은 유지.
- 커밋: `feat: 스크랩 토글을 난이도 행으로 이동하고 풀이 중에도 허용`

## 4) 오답 피드백을 카드 색상 이펙트로 교체 (QA #1)
- 현재: 오답(`SessionFeedback.Mismatch`) 시 `RetryNudge`("아직이에요, 다시 해볼까요?", `FeedbackComponents.kt:212`) 텍스트 카드가 입력창 아래에 나타난다(406~409, 518행). QA: 문구가 모호하고 세로 공간을 차지한다.
- 변경 (2026-07-17 사용자 확정 — 무낙인 유지, **주황 계열**):
  1. 문제 영역 `Column`(344~356행)에 배경색을 입히고, `Mismatch` 피드백 발생 시 **주황 톤으로 잠깐 물들었다 돌아오는** 이펙트를 넣는다.
     - 구현: `animateColorAsState` + `tween(BcsMotion.durFast, BcsMotion.easing)`. 트리거는 기존 햅틱 훅과 같은 `LaunchedEffect(state.feedback)`(332행)에 얹는다. 잠깐 유지 후 원복(예: 주황 → 300ms 내 원색).
     - 색: raw hex 금지. `BcsColors`(`ui/theme/Color.kt:76~`)에 주황 계열 피드백 토큰이 없으면 라이트/다크 쌍으로 신설(예: `retryFlash`). danger(빨강)는 금지(무낙인).
  2. `RetryNudge` 텍스트 카드는 렌더링에서 제거한다(컴포넌트가 다른 곳에서 안 쓰이면 삭제).
  3. **접근성:** 색만으로는 스크린리더가 인지할 수 없다. 기존 라이브 리전 패턴(499행, `liveRegion = Polite`)으로 시각적으로 표시되지 않는 짧은 안내(예: "정답과 달라요, 다시 시도해 보세요")를 유지·낭독시킨다. '오답/틀림/실패' 어휘 금지.
- **유지하는 것:** `NearMiss`("거의 맞았어요…" — 명세 불변식 19), `MisconceptionHintCard`(오답 교정 힌트 — 명세 불변식 6)는 텍스트 그대로 유지. 이펙트는 `Mismatch`·`NearMiss` 공통으로 적용해도 좋으나 `NearMiss`는 기존 info 톤 유지 고려.
- 테스트: "아직이에요, 다시 해볼까요?" 문자열 단언(오답 no-leak 테스트 119행 등)을 새 구조에 맞게 수정 — no-leak 단언(정답 문자열 부재)은 반드시 유지. 무낙인 어휘 부재 단언 유지.
- 커밋: `feat: 오답 알림을 텍스트 카드 대신 문제 영역 색상 이펙트로 교체`

## 5) '정답 보기'를 시도 전에도 허용 (QA #3) — 서버+명세 포함
- 현재: 오답 1회 이후에만 노출/허용.
  - 서버: `Session.reveal()`(`server/.../session/domain/Session.kt:125-134`)이 `wrongAttemptCount < MIN_WRONG_ATTEMPTS_TO_REVEAL`이면 `RevealNotAllowedException.beforeAnyWrongAttempt()`.
  - 클라: `canReveal`(`SessionViewModel.kt:346-348`) = `reveal == null && !solved && (feedback is Mismatch || feedback is NearMiss)`.
- 변경 (2026-07-17 사용자 확정):
  1. **서버:** `Session.reveal()`에서 선행 오답 요구 가드를 제거한다(완료 세션 가드는 유지). `MIN_WRONG_ATTEMPTS_TO_REVEAL` 상수·`RevealNotAllowedException.beforeAnyWrongAttempt()`·관련 주석(120~123행, `SessionService.kt:123`) 정리. `RevealNotAllowedException`이 다른 사유 없이 비게 되면 예외 클래스·`ErrorCode.REVEAL_NOT_ALLOWED`까지 제거 여부 판단.
  2. **클라:** `canReveal`에서 feedback 조건을 제거 — `reveal == null && !solved`.
  3. **명세 개정** (`docs/도메인 명세서.md`): 221행 "막히면 정답 공개(무낙인 안전판) … 오답 후 제공되는 안전판" → "사용자가 원하면 시도 전에도 열 수 있다"로 수정. §1 무낙인 흐름(158행 부근)도 같은 취지로 손질. 불변식 3("사용자가 요청한 경우에만 공개")은 그대로 유효.
  4. 숙련도 신호는 변경 없음 — 정답 공개는 기존대로 '도움/포기' 신호로 기록된다(시도 전 공개도 동일).
- 테스트:
  - 서버: `SessionTest`·`SessionControllerIntegrationTest`의 "시도 전 공개 거부" 케이스를 "시도 전에도 공개 허용"으로 반전. 완료 세션 공개 거부 케이스는 유지.
  - 클라: `정답_보기는_시도_전에는_제안하지_않는다`(181행)를 "시도 전에도 정답 보기가 보인다"로 반전. 공개 전 정답 비노출(no-leak) 단언 유지.
- 커밋: `feat: 정답 보기를 오답 제출 전에도 허용` (서버+클라+명세 한 커밋 — 정책 하나의 변경)

## 6) 정답 공개 레이아웃을 정답 시와 유사하게 (QA #4)
- 현재: 공개 시(428~455행) `ModelAnswerBlock`(정답+해설 큰 블록)이 문제와 입력 사이에 삽입돼 `TypeAlongField`가 아래로 밀린다 → 사용자가 입력창이 사라졌다고 착각. 정답을 맞혔을 때(373~395행)는 입력창 자리가 `ConfirmedAnswerField`로 교체되고 해설·개념칩·심화가 그 아래 온다.
- 변경: 공개 시 배치를 정답 시와 같은 골격으로 맞춘다.
  1. 문제 영역에 동일한 `alpha(0.6f)` 감쇠 적용(현재 solved에만 적용, 344행).
  2. 기존 입력창 위치에 **정답 표시 필드**(`ConfirmedAnswerField` 변형 — "이 정답을 따라 적어 보세요" 안내 톤, 축하 문구 없음)를 두고, 그 **바로 아래에 `TypeAlongField`** 를 붙인다 → 입력 요소가 원래 자리 근처에 남는다.
  3. 해설·`ConceptChips`·`EnrichmentBlock`은 그 아래에 정답 시 `FeedbackCard`(Correct)와 같은 flat 순서로 배치.
  4. `ModelAnswerBlock`의 큰 블록 형태는 위 구조로 대체(해설 부분은 3번 위치로).
- 불변 유지: "정답 공개 후 따라 입력해야 진행"(명세 불변식 19) 로직 변경 없음. 좌표 테스트 `모범답안은_따라_입력_칸보다_위에_있다`(209행)는 새 구조에서도 성립(정답 표시 필드가 따라 입력 칸 위).
- 테스트: 공개 상태의 배치·문구 단언 갱신, 공개 전 no-leak 단언 유지.
- 커밋: `feat: 정답 공개 레이아웃을 정답 시 배치와 통일`

## 공통 주의
- `SessionScreenUiTest.kt`는 도메인 가드레일 테스트(정답 비노출·무낙인 어휘 부재·'세션' 용어 미노출)가 촘촘하다. **가드레일 단언을 삭제로 풀지 말고, 새 UI에 맞게 옮겨서 유지**할 것.
- 키보드 대응이 동기이므로, 각 단계 후 실기기(또는 에뮬레이터)에서 키보드 올라온 상태의 세로 공간을 확인한다.

## 검증
- 클라 `jvmTest` + 서버 `test` 전체 통과.
- 수동 시나리오: ① 오답 제출 → 주황 플래시 확인·텍스트 카드 없음 ② 시도 전 정답 보기 → 공개·따라 입력 진행 ③ 공개 시 입력창 위치 유지 ④ 스크랩을 풀이 중에 토글 ⑤ 탑바에서 오류 신고 진입 ⑥ 제출 버튼 문구.
