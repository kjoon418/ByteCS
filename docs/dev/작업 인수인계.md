## 1. 이번 세션에서 확정된 도메인 결정

세 개의 도메인 커밋(`9fd8b87`, `fcb6422`)과 그 뒤를 따른 코드 커밋(`82a47d1`)이 이 절의 내용입니다. **결론만 알면 다음 사람이 되돌립니다.** 근거를 같이 읽어 주세요.

### 1.1 디딤 문제·파고들기 스택·선행 간선 → 로드맵

세 가지는 **한 묶음**으로 MVP에서 빠졌습니다(`도메인 명세서.md` 로드맵 절 첫 항목).

> **결정 근거 (명세서 로드맵 절 인용):** "디딤은 '선행 개념을 익히면 본 문제가 풀린다'를 전제하는데, **정의 재생형**에서는 선행 개념을 익혀도 그 **이름은 나오지 않는다**. (…) 또한 선행 간선은 복습과 공유되지 않는 디딤 전용 비용이다."

시드 문제 7개(`ProblemSeeder.kt`)를 전수 확인했을 때 "선행 개념을 익히면 본 문제에 다가간다"가 성립하는 사례가 **0개**였던 것이 발단입니다. 대부분이 정의 재생형(정답 = 개념 이름)이라, 배경지식을 아무리 채워 줘도 답인 '이름' 자체는 나오지 않습니다.

⚠️ **주의 — 근거가 한 번 갱신됐습니다.** 명세서에 `[갱신 — 문제 유형 확정]` 블록이 붙어 있습니다. 유도형이 MVP에 포함되면서 "디딤이 값어치를 하는 유형이 없다"는 근거는 **사라졌습니다**. 지금 디딤이 로드맵인 이유는 **선행 간선의 비용**(개념 그래프·순환 금지·깊이 정책 일체가 복습과 공유되지 않는 디딤 전용 구조) 하나뿐입니다. 재검토 시점도 "유형이 확정되면"이 아니라 **"선행 간선을 감당할 콘텐츠·큐레이션 여력이 생길 때"** 로 바뀌었습니다. 이 구분을 놓치고 "유도형 생겼으니 디딤 부활"로 읽지 마세요.

### 1.2 복습 = 같은 문제 재출제가 기본

옛 규칙은 "같은 개념의 **다른** 문제를 우선 낸다"였고, 뒤집혔습니다. 예외는 **유도형에 한해서만** 남습니다.

> **결정 근거 (명세서 '복습 문제 선정' 절 인용):** "이전 결정은 (…) 근거는 '이미 푼 문제를 그대로 내면 답을 외운 **재인**이 되어 정착 검증이 흐려진다'였다. 두 가지가 틀렸다. (1) 여기서 쓰인 '재인'은 이 문서가 기능 1에서 정의한 '재인'(보기 중 고르기)과 다른 뜻이라 **용어 규율 위반**이다 (…) (2) 그 수단이 목적을 달성하지 못한다 — 정답이 개념 이름인 문제에서는 문제를 바꿔도 답이 같아, 검수 처리량을 두 배로 지불하고도 지목한 위험을 막지 못한다."

(2)가 결정적입니다. 용어 위반은 고쳐 쓰면 그만이지만, **수단이 목적을 달성하지 못한다**는 것은 규칙 자체를 무효로 만듭니다. 옛 규칙이 막으려던 것의 정확한 이름은 **문항–답 연합**(개념 이해가 아니라 그 문항의 정답 문자열을 외워서 맞히는 상태)이고, 이 위험이 실재하는 범위는 유도형뿐이라 예외도 거기에만 남았습니다.

콘텐츠 공급이 이 서비스의 병목이라는 제약과도 정합합니다 — 개념당 문제를 여러 개 쌓지 않아도 복습이 성립합니다.

### 1.3 문제 유형 축 신설 — 정의 재생형 / 유도형

**둘 다 MVP에 포함**됩니다. '주관식 단답'이 정하는 **답변 형식**과는 **독립된 축**이라는 점이 핵심입니다(둘을 섞어 쓰면 명세가 무너집니다).

- **정의 재생형(`DEFINITION_RECALL`):** 지문이 정의를 주고 사용자가 **개념의 이름**을 떠올린다. 정답 = 개념 이름.
- **유도형(`DERIVATION`):** 개념을 재료로 여러 단계를 거쳐 답을 도출한다. 정답 ≠ 개념 이름. 수식 답을 허용한다.

코드: `server/src/main/kotlin/watson/bytecs/problem/domain/ProblemType.kt` (KDoc에 근거가 들어 있습니다).

이 축을 참조하는 곳이 현재 둘입니다 — **근접 신호 적용 여부**(§1.4)와 **복습 문제 선정**(§1.2 예외). 또 자동 구조 검증이 **유형 태깅 존재**를 요구합니다(수용 기준 23번).

### 1.4 근접(오탈자) 신호는 정의 재생형에만

> **결정 근거:** 답이 자연어 단어일 때만 "편집거리가 작다 ⇒ 오타다"가 참입니다. 개념명끼리는 편집거리가 멉니다(`TCP`↔`UDP` = 2)이므로 편집거리 1은 실제로 오타입니다(`collsion` → `collision`). 반면 유도형의 정답은 수식·숫자처럼 **밀집한 공간의 한 점**이라 이웃이 전부 '유효한 다른 답'입니다. `o(n²)`에 `o(n)`은 오타가 아니라 이중 반복문을 하나로 잘못 센 **오답**이고, 근접으로 알려주면 **틀린 유도를 맞았다고 알려주는 셈**이 됩니다.

이건 도메인 결정이자 실제로 있던 버그입니다 — `82a47d1 fix: 계산을 틀린 답에 오타라고 알려주던 문제`.

구현은 `Problem.judge()` / `Problem.isNearMissCandidate()`에 캡슐화돼 있고 **두 관문의 AND**입니다: `type == DEFINITION_RECALL && acceptable.length >= 3`. 테스트를 만질 일이 있으면 §5.2를 먼저 읽으세요(합성 조건은 절반씩 따로 죽여야 합니다).

## 3. 힌트 슬라이스 착수 정보

> ⭐️ **이 슬라이스의 특이점을 먼저 말합니다.** 보통 슬라이스는 서버가 먼저 있고 클라이언트가 따라옵니다. **여기는 정반대입니다.** 서버는 백지이고, 클라이언트는 이미 완성돼 있습니다. 즉 **UI 계약이 서버 계약을 이미 규정해 놨습니다.** 서버 DTO를 새로 상상하지 말고, 먼저 UI가 뭘 요구하는지 읽으세요.

### 3.1 현재 상태 — 서버는 백지, 클라이언트는 완성

**서버:** `server/` 전체에 `Hint`/`hint`/`힌트` 문자열이 **하나도 없습니다**(grep 확인). 엔티티·필드·API·시드 전부 신규입니다.

**클라이언트:** `app/shared/src/commonMain/kotlin/watson/bytecs/ui/components/HintComponents.kt`에 `HintStepper`·`MisconceptionHintCard`가 **완성돼 있고 가드레일 테스트까지 있는데**, 호출자가 **0곳**입니다. 참조하는 파일은 자기 자신과 테스트 2개(`HintStepperUiTest.kt`, `BcsComponentsUiTest.kt`)뿐입니다.

**가장 먼저 읽을 것:** `HintComponents.kt`의 `BcsHint` 데이터 클래스와 `HintStepper`의 KDoc. 지켜야 할 규칙이 KDoc에 명시적으로 적혀 있고, 그게 곧 서버가 맞춰야 할 계약입니다.

```kotlin
@Immutable
data class BcsHint(
    val text: String,
    val codeSnippet: String? = null,
    val drilldownLabel: String? = null,
)
```

### 3.2 서버가 `revealedCount`를 영속해야 합니다

**힌트 열람은 화면 상태가 아니라 학습 기록입니다.**

> **명세서 인용 (개념 숙련도 산정 절):** "**힌트·정답 공개·오답 교정 힌트 없이 스스로 맞힘** → 강한 정착 신호(숙련도↑). **도움받아 맞힘**(힌트를 열고 맞힘) → 약한 정착 신호."

> **`HintStepper` KDoc 인용:** "공개 상태를 [revealedCount]로 끌어올린 이유: 힌트를 몇 개 열었는지는 화면 장식이 아니라 학습 기록이다(숙련도 반영·재진입 시 복원). 소유는 호출자(뷰모델)에 둔다."

즉 **'힌트 열기'는 서버 왕복이 필요한 동작**입니다. 로컬 상태로만 두면 재진입 시 복원이 안 되고, 기능 3(복습)이 쓸 신호가 사라집니다.

**`SessionItem`에 `revealedHintCount` 추가가 필요합니다.** 현재 필드는 `position`·`problemId`·`solved`·`revealed`·`wrongAttemptCount`·`submittedAnswer`뿐입니다(확인함). `SessionItem`은 `@Embeddable`이고 **상태 변경은 `Session` 애그리거트가 주도**하도록 세터가 `protected`입니다 — 기존 `recordWrongAttempt()`/`markRevealed()` 패턴을 그대로 따르세요.

> ⚠️ **이름 충돌 주의:** `SessionItem.revealed`는 **정답 공개**(모범답안)를 뜻합니다. 힌트 공개가 아닙니다. 새 필드 이름을 `revealed*`로 지을 때 헷갈리기 쉬우니 `revealedHintCount`처럼 명확히 하세요.

### 3.3 ⚠️ 힌트 본문을 공개 전에 내려보내면 안 됩니다

> **`SessionProblemResponse` 주석 인용:** "개념명·허용답·해설 등 **정답을 유추할 수 있는 정보는 절대 포함하지 않는다**(문제 슬라이스의 다음 문제 형태와 동일한 계약)."

힌트 본문을 문제와 함께 미리 내려주면 이 계약이 깨집니다. 클라이언트가 안 그리더라도 **응답 바디에 들어 있으면 유출**입니다.

**단, 힌트 '개수'는 알려야 합니다.** `HintStepper`가 `hints.isEmpty()`면 **진입점 자체를 노출하지 않기** 때문입니다(KDoc 규칙 3: "눌러 봐야 아무것도 없는 버튼을 두지 않는다"). 개수를 모르면 `[힌트 보기]`를 띄울지 결정할 수 없습니다.

→ 설계 방향: `SessionProblemResponse`에 **개수만**(예: `hintCount`) 싣고, **본문은 '열기' 요청의 응답으로만** 내려주는 형태. 이러면 §3.2의 영속 요구와도 자연스럽게 맞아떨어집니다(열기 = 서버 왕복 = 기록 + 본문 반환).

### 3.4 오답 교정 힌트는 신규 API가 불필요합니다

기능 2.5는 **push(오답 트리거·자동)** 입니다 — 사용자가 요청해서 여는 게 아니라 **제출의 결과로** 뜹니다. 따라서 신규 엔드포인트가 아니라 **기존 `SessionAttemptResponse`에 필드를 추가**하는 것이 맞습니다.

- 서버: `server/src/main/kotlin/watson/bytecs/session/application/dto/SessionAttemptResponse.kt`
- 클라이언트: `SessionViewModel.kt`의 `sealed interface SessionFeedback` 확장이 필요합니다. 현재:
  ```kotlin
  sealed interface SessionFeedback {
      data class Correct(val concept: String?, val explanation: String?) : SessionFeedback
      data object Mismatch : SessionFeedback
      data object NearMiss : SessionFeedback
  }
  ```
  교정 힌트는 `Mismatch`(그리고 명세상 `NearMiss`도 불일치)에 실려 오므로, 어느 쪽에 어떻게 붙일지 판단이 필요합니다.

**지켜야 할 규칙(명세 기능 2.5):** 무낙인 유지(오답 확정 아님·무제한 재시도) / 정답 비노출("왜 그 답이 아닌지"만) / **막다른 길 없음** — 큐레이션 안 된 오답은 교정 힌트 없이 일반 불일치로 흐릅니다. 교정 힌트는 품질 향상 장치이지 모든 오답을 덮는 전제조건이 **아닙니다**.

### 3.5 ⚠️ `BcsHint.drilldownLabel`을 어떻게 할지 판단이 필요합니다

이 필드는 **디딤 문제 진입 버튼의 라벨**입니다. 그런데 **디딤이 로드맵으로 갔습니다**(§1.1). `HintCard`가 `hint.drilldownLabel?.let { GhostButton(...) }`로 버튼을 그리고 있어, 지금 상태로 서버가 이 값을 채우면 **MVP에 로드맵 기능이 들어옵니다.**

선택지(트레이드오프):

- **(A) 남기고 항상 null.** 변경 0. 디딤 도입 시 서버가 값만 채우면 됩니다. 대신 **죽은 코드와 죽은 분기가 남고**, 다음 사람이 "왜 아무도 안 채우지?"를 다시 조사합니다. `GhostButton` 경로는 테스트로만 살아 있게 됩니다.
- **(B) 제거.** 컴포넌트가 MVP 범위와 정확히 일치해 오해가 없습니다. 대신 디딤 도입 시 컴포넌트·테스트를 되돌려야 합니다.

§2.2에서 **시안이 아직 디딤 버튼을 갖고 있다**는 점과 묶여 있으니, **기획자 확인과 함께 처리하세요.** 어느 쪽이든 **선택한 이유를 KDoc에 남기세요** — 이 필드는 이미 한 번 "왜 있지?"를 유발했습니다.

### 3.6 스키마·시드

- **Flyway가 없습니다.** `ddl-auto: update`(prod) / `create-drop`(local). **엔티티만 추가하면 스키마가 따라옵니다.**
- ⚠️ **그 대가:** `ddl-auto`는 **기존 행을 백필하지 못합니다.** 행이 있는 테이블에 NOT NULL 컬럼을 추가하면 실패합니다. `Problem.type`이 nullable인 이유가 정확히 이것입니다(§4). 새 컬럼을 non-null로 만들고 싶다면 이 제약을 먼저 확인하세요.
- **시드:** `server/src/main/kotlin/watson/bytecs/problem/infrastructure/ProblemSeeder.kt`. 현재 문제 7개, 힌트 0건.
  - ⚠️ **가드:** `if (problemRepository.count() > 0) return`. **이미 문제가 있는 DB에는 시드가 반영되지 않습니다.** 힌트를 시드에 추가해도 기존 로컬 DB에서는 안 보입니다 — DB를 비우거나 로컬 프로파일(`create-drop`)로 확인하세요. 이걸 모르면 "시드를 넣었는데 왜 안 뜨지"로 시간을 씁니다.
  - `@Profile("!test")` — 테스트에서는 동작하지 않습니다. 테스트는 각자 데이터를 준비합니다.
  - ⭐️ **힌트가 0개인 문제를 시드에 최소 1개 남기세요.** 도메인상 힌트는 **0~N개가 정상**이고("문제별 자유 구성" — [결정]), UI가 이 경우를 분기합니다(진입점 미노출). 전부 힌트를 달아 두면 **그 분기가 시드로 한 번도 실행되지 않습니다.**

### 3.7 착수 전 마지막 점검

- 명세서 **`## 2. 힌트 (히어로)`** 절(239행~)과 **`## 2.5 오답 교정 힌트`** 절(278행~)을 통째로 읽으세요. 특히 **`### 디딤 문제로의 분기` 절은 통째로 MVP 미포함**입니다(제목에 명시돼 있습니다).
- 수용 기준 4·6·21번(전체 요약 절)이 pass/fail 기준입니다.
- **고정 사다리를 만들지 마세요.** 명세가 명시적으로 금지합니다: "고정된 종류·개수 사다리(예: 항상 키워드→배경지식)를 두지 않는다." 힌트 종류를 enum으로 못박고 싶은 유혹이 들면 `HintStepper` KDoc 규칙 1을 다시 읽으세요.

## 4. 알려진 버그·기술 부채

### 첫 실행 레이스 (재현됨)

**증상:** 데이터 초기화 후 첫 실행에 "오늘의 한입을 불러오지 못했어요". 재시도하면 정상.

**원인(코드 확인함):** `app/shared/src/commonMain/kotlin/watson/bytecs/App.kt`의 `AppNavHost`가 `AuthState.BootstrapFailed`만 걸러내고 **`AuthState.Loading`은 그대로 통과시켜** HomeScreen을 그립니다. 그 결과 게스트 토큰이 저장되기 전에 `getToday()`가 호출돼 401이 납니다.

```kotlin
if (authState is AuthState.BootstrapFailed) {
    BootstrapErrorScreen(...)
    return
}
// ← Loading이 여기로 흘러 HomeScreen을 그린다
```

`AuthState`는 `Loading`·`BootstrapFailed`·`Guest`·`Member` 4상태입니다(`account/SessionManager.kt`). `BootstrapFailed` 분기를 넣을 때 `Loading`을 빠뜨린 것으로 보입니다.

### 도메인 구조 부채

- ⚠️ **`Problem.concept`가 `@ManyToOne` 단일**인데, 명세는 **"문제는 하나 이상의 개념에 연결된다(문제 N — M 개념)"** 로 [결정]했습니다(문제 형태 절 120행). **힌트 분기·복습이 모두 개념 축 위에서 돕니다** — 명세: "**힌트 분기·복습·진도**가 모두 이 축 위에서 돈다". **기능 3(복습) 착수 전에는 정리해야 합니다.** 힌트 슬라이스도 개념을 건드리므로 영향 범위를 미리 보세요.
- **`Problem.type`이 nullable.** 이유는 `ddl-auto`의 백필 불가(§3.6)이고, KDoc에 근거가 적혀 있습니다. **의도적으로 안전한 방향으로 틀리게** 설계돼 있습니다 — 미상이면 근접 판정이 꺼지는 쪽(정확 일치만)으로 퇴화합니다. `DEFINITION_RECALL`을 기본값으로 두면 태깅을 빠뜨린 유도형에 근접 판정이 되살아나 §1.4 버그가 재발합니다. **기본값을 넣고 싶은 유혹을 참으세요.** 승인 파이프라인이 들어오면 구조 검증(수용 기준 23번)에서 유형 태깅을 강제하고, 백필 후 non-null로 조일 수 있습니다.

### 소소한 것들

- 비밀번호 표시 토글 부재 — `BcsComponents.kt`의 `PasswordVisualTransformation`이 하드코딩.
- EmptyState 3벌 중복.
- `PrimaryButtonRole` KDoc의 과장 — `GhostButton(contentColor)`가 규칙을 우회할 수 있음.
- `"분"` substring 가드가 `"분량"`에 오발할 위험.
- 토큰 갭 — `buttonHeightInline`·`dangerPressed`.
- **iOS 빌드 미검증.** Windows에서 컴파일 불가 — macOS가 필요합니다. **"빌드된다"고 가정하지 마세요.**

## 5. 이 프로젝트에서 통한 작업 방식

> 다음 세션이 같은 함정을 다시 밟지 않도록 남깁니다.

### 5.1 가짜 테스트를 조심하세요

이번 작업에서 **가짜 테스트가 5건** 나왔습니다. **이름은 그럴듯한데 본문이 아무것도 지키지 않는** 유형입니다. 전부 **변이 테스트**로 잡혔습니다 — 통상적인 리뷰로는 안 걸렸습니다.

### 5.2 변이로 검증하는 법

- **"고치기 전엔 변이가 살고, 고친 뒤엔 죽는다"를 양쪽 다 보이세요.** 한쪽만 보이면 증명이 아닙니다.
- ⭐️ **조건이 합성(`A && B`)이면 각 절반을 따로 죽이세요.** 전체만 죽이면 한쪽이 안 잠긴 걸 놓칩니다. `Problem.isNearMissCandidate()`가 정확히 `type == DEFINITION_RECALL && length >= 3` 형태입니다 — 유형 조건과 길이 조건을 **각각** 죽여 보세요.
- **가드레일 규칙(색 매핑 등)은 순수 함수로 빼서 테스트하세요.** `ComponentTones.kt`가 선례입니다. Compose 트리 안에 묻어 두면 규칙이 규칙인지 알 수 없습니다.

### 5.3 ⚠️ 동시 실행 함정

**여러 에이전트가 같은 워킹트리에서 Gradle을 동시에 돌리면 `build/` 캐시와 `test-results/`가 서로를 덮어씁니다. 남의 결과를 자기 것으로 읽게 됩니다.** 격리 워크트리를 쓰거나 직렬화하세요. (이건 "느려진다"가 아니라 **틀린 초록불을 본다**는 문제입니다.)

### 5.4 테스트·실행 명령

```bash
# 클라이언트
./gradlew :app:shared:jvmTest :app:shared:testAndroidHostTest
# 서버
./gradlew :server:test
```

- ⭐️ **UI 테스트는 `jvmTest`에 둡니다.** `commonTest`에 두면 에뮬레이터를 요구합니다. `androidx.compose.ui.test.v2.runComposeUiTest`를 씁니다.
- **에뮬레이터 실행:** base URL은 런타임 자동 감지입니다 — 에뮬레이터는 그냥 Run, **실기기는 `adb reverse tcp:8080 tcp:8080`**.
- ⚠️ **APK 경로는 `app/androidApp/build/outputs/apk/debug/`** 입니다. `intermediates` 아래 것은 stale이라 **옛 빌드를 설치하고 "고쳤는데 왜 그대로지"** 를 하게 됩니다.
- 자세한 것은 `docs/dev/Android 실기기 테스트 가이드.md`.
