# 문제 데이터 생성 가이드

이 문서는 CS한입 서비스의 본격 테스트를 위해 `server/src/main/resources/seed/problems-generated.json`에 약 100개의 문제를 생성할 때 따라야 할 명세를 담는다.

**독자**: 이 저장소를 처음 여는 새 AI 세션. 이 문서만으로 100문제를 생성·검증·커밋할 수 있어야 한다.

> **중요 고지**: 이 콘텐츠는 **테스트 전용**이며 승인 게이트(명세 승인 파이프라인)을 통과하지 않았다. 실서비스 투입 전에 큐레이터의 검수를 거쳐야 한다.

---

## 1. 목적

약 100개의 CS 문제를 생성해 서비스의 학습 세션·복습·검증을 본격적으로 테스트한다. 생성 방식:
- **AI 초안 제안** (이 세션에서 수행)
- **자동 구조 검증** (서버 로더가 기동 시 도메인 불변식으로 전수 검증)
- 테스트 통과 확인 (단위/통합 테스트 — 기존 분포 검증·no-leak 스윕·불변식 검증 포함)

---

## 2. 산출물: JSON 스키마와 필드별 도메인 규칙

파일: `server/src/main/resources/seed/problems-generated.json`

### JSON 구조

다음은 하나의 문제 객체 형태다. 각 필드를 생성할 때 뒤따르는 규칙을 정확히 지켜야 한다.

```json
{
  "problems": [
    {
      "question": "다음 … 무엇이라고 하나요?",
      "type": "DEFINITION_RECALL",
      "difficulty": "EASY",
      "codeSnippet": null,
      "concepts": ["해시 충돌", "자료구조"],
      "acceptableAnswers": ["해시 충돌", "hash collision", "해시 충돌 (hash collision)"],
      "representativeAnswer": "해시 충돌 (hash collision)",
      "explanation": "서로 다른 키가 같은 해시 값으로 매핑되는 현상입니다.",
      "hints": [
        { "text": "두 개의 서로 다른 입력이 같은 결과를 갖는 상황을 생각해보세요.", "codeSnippet": null },
        { "text": "이를 해소하기 위해 체이닝이나 개방 주소법을 사용합니다.", "codeSnippet": null }
      ],
      "misconceptionHints": [
        { "expectedAnswers": ["해시 함수"], "message": "해시 함수는 이 현상을 일으키는 도구일 뿐이고, 현상 자체의 이름은 아닙니다. 다시 생각해보세요." }
      ],
      "enrichment": {
        "title": "왜 충돌이 발생할까요?",
        "body": "해시 테이블은 무한한 데이터를 유한한 크기의 배열에 매핑하기 때문에 충돌이 반드시 생깁니다.",
        "items": [
          { "title": "해결책 01. 체이닝", "description": "같은 인덱스에 데이터를 연결 리스트로 매다는 방식입니다." }
        ],
        "quote": null
      }
    }
  ]
}
```

### 필드별 규칙

#### `question` (필수, 문자열)
**역할**: 사용자가 풀 문제의 지문.

**규칙**:
- 빈 문자열 금지.
- 명확한 질문 형태여야 함 (질문이 애매하면 허용답과 정답 판정이 흔들린다).
- 한국어 사용 권장. 영문·코드는 필요할 때만.
- **도메인 근거** (명세 119~127행): 주관식 단답 형식. 사용자가 정답을 직접 떠올려 적는다.

#### `type` (선택, "DEFINITION_RECALL" 또는 "DERIVATION" 또는 null)
**역할**: 문제가 사용자에게 요구하는 사고의 종류. 근접(오탈자) 신호 적용 여부와 복습 문제 선정이 이 값으로 분기한다.

**두 가지 유형**:

1. **`"DEFINITION_RECALL"`** (정의 재생형)
   - 지문이 개념의 정의·성질을 주고, 사용자가 그 **개념의 이름(용어)**을 떠올린다.
   - 정답 = 개념 이름.
   - **예**: "가장 나중에 넣은 데이터가 먼저 나오는 후입선출(LIFO) 자료구조는?" → 스택
   - **정규화 후 정확 일치**로만 판정되고, 편집거리가 1~2로 가깝다면 **근접 신호**(오타 알림)가 표시된다.
   - **도메인 근거** (명세 129~137행, 149행): 정답이 자연어 단어(개념 이름)라 개념명끼리 편집거리가 멀고(`TCP`↔`UDP`는 2), 편집거리 1은 실제로 오타다(`collsion`→`collision`).

2. **`"DERIVATION"`** (유도형)
   - 지문이 재료(코드·구성·상황)를 주고, 사용자가 개념을 적용해 **여러 단계를 거쳐 답을 도출**한다.
   - 정답 ≠ 개념 이름 (수식·숫자·계산 결과).
   - **예**: 코드를 보고 시간 복잡도 답하기 → `o(n^2)`, 또는 /26 서브넷의 호스트 수 → `62`.
   - **근접 신호가 적용되지 않는다** (유도형에서 편집거리 1의 차이는 오타가 아니라 유도 실패를 뜻함 — 예: `o(n²)`에 `o(n)`은 이중 반복을 하나로 잘못 센 오답).
   - **도메인 근거** (명세 133~141행, 149행): 정답이 수식·숫자처럼 밀집한 공간의 한 점이라 이웃이 전부 유효한 다른 답이고, 한 글자가 곧 의미(지수·차수·자릿수).
   - **[반영 2026-07-19: D1] 예외는 태깅과 무관한 런타임 상태다.** 정답 공개 후 '따라 입력' 구간에서는 이 유형도 근접 신호를 받는다(명세 '정답 판정' 절 참고). 이는 **type 태깅과 무관한 별도 런타임 분기**이므로, 위 태깅 지침(유도형=근접 미적용)은 그대로 유효하다 — 큐레이터가 이 예외 때문에 태깅을 바꿀 필요는 없다.

**선택 허용 (null)**:
- 유형을 모르면 `null`로 두어도 된다.
- `null`일 때의 동작: 근접 신호가 표시되지 않고 정확 일치만 정답이 된다(안전 방향으로 퇴화).
- **주의**: 유도형인데 `"DEFINITION_RECALL"`로 태깅하면 근접 신호가 활성화되어 버그가 재발한다. 모르면 차라리 `null`로 두는 게 낫다.

**도메인 근거** (명세 450~451행): 근접 신호 적용 여부(기능 1)와 복습 문제 선정(기능 3)이 유형으로 분기한다. 기능 3에서 유도형에만 "다른 문제 우선" 예외를 적용하므로, 유형 태깅이 없으면 이 규칙들의 동작이 정의되지 않는다.

#### `difficulty` (선택, enum 값)
**역할**: 문제의 난이도.

**허용되는 값** (Difficulty.kt에서 실제 코드):
- `"EASY"` — 기초 개념 이해
- `"MEDIUM"` — 중급 적용
- `"HARD"` — 고급 복합

선택값이므로 없어도 되지만, 로드맵의 개인화 난이도 선정에 쓰일 예정이므로 생성할 때 지정하는 것이 좋다.

**도메인 근거** (명세 123~124행): 난이도는 표시용일 뿐 아니라 로드맵의 '개인 수준 맞춤 적응 출제'와 '더 쉬운 문제(디딤)'가 이 속성을 소비한다.

#### `codeSnippet` (선택, 문자열 또는 null)
**역할**: 문제에 포함된 코드 예시.

**규칙**:
- 코드 없으면 `null`.
- 있으면 모바일 세로 화면(약 40~50자 폭)에서 가독성을 유지해야 한다.
- 라인이 과도하게 길면 안 됨 (컨텍스트 끊김, 가독성 악화).
- **도메인 근거** (명세 138, 496행): 문제 형태가 "주관식 단답"이지 "코드 작성형"이 아니므로, 코드는 **지문의 일부**(문제를 이해하게 돕는 재료)일 뿐이다. 사용자가 코드를 작성하지는 않는다.

#### `concepts` (필수, 배열 1개 이상)
**역할**: 이 문제가 다루는 CS 개념들. 문제 N — M 개념 연결이며, 힌트 분기·복습의 근거가 된다.

**규칙**:
- 배열은 **최소 1개, 최대 N개**.
- **순서가 중요** — 첫 번째 개념이 **대표 개념**이고, 다른 곳에서 이를 참조한다 (화면 표시, 복습 선정).
- **개념명은 실제 CS 용어** (예: "해시 충돌", "TCP", "캐시 미스").
- 개념 이름 기준 find-or-create로 문제 간에 공유된다 — 같은 이름의 개념은 같은 개념 노드로 통합된다.

**개념 풀 설계 (중요)**:
- 약 100문제를 생성할 때 개념 풀을 30~40개 설계하는 것을 권장한다.
- **개념당 1~5문제를 재사용**하는 것이 복습 테스트의 전제다 (명세 51행).
  - 개념 1개에 문제가 1개뿐이면 복습으로 다시 만날 가능성이 없다.
  - 개념 1개에 유도형 문제 여럿이 있으면 다른 유도형 문제를 먼저 선정할 수 있다 (복습 규칙).
- **권장 순서**: 먼저 개념 풀(30~40개)을 설계한 뒤, 그 위에 문제를 얹는다.

**도메인 근거** (명세 51, 63행, 316~317행): 복습·정착은 개념을 축으로 돈다. "이 문제를 다시 풀라"가 아니라 "이 **개념**이 약하니 다시 만나라"가 기본이므로, 개념 풀의 충실함이 복습 시스템의 가치를 좌우한다.

**대분류성 용어를 개념으로 쓰지 않는다** (명세 §7 '백필·콘텐츠 운영', 결정 2026-07-17):
- 개념 배열에 8개 대분류 이름 자체(예: "자료구조", "알고리즘", "운영체제", "네트워크", "데이터베이스", "컴퓨터구조", "소프트웨어공학", "보안")를 개념으로 태깅하면 안 된다.
- 대분류는 개념에 귀속되는 상위 메타데이터이지, 그 자체가 힌트·복습의 축이 되는 "개념"이 아니다("자료구조"라는 이름으로는 복습 문제를 특정할 수 없다).
- 대분류를 표시하고 싶다면 개념 배열이 아니라 아래 `conceptCategories` 필드로 개념 각각에 카테고리를 부여한다.
- 실수로 섞였다면: 그 대분류성 태그를 개념 배열에서 빼고, 문제에 남은 개념이 하나도 없다면 그 문제의 적절한 최소 단위 개념(예: "자료구조" → "스택"·"큐" 등 실제로 묻는 개념)으로 교체한다(Problem은 개념 1개 이상 불변식이 있다).

#### `conceptCategories` (선택, `ProblemSeedFile` 최상위 필드 — 문제 객체 안이 아님)
**역할**: 개념 이름 → 8개 대분류 중 하나(enum 이름 문자열)의 전역 매핑. 카테고리는 **개념에 귀속되는 단일 출처**이므로(명세 §7 '카테고리 체계'), 문제마다 반복해서 붙이지 않고 시드 파일 전체에 한 번만 선언한다.

**형태**:
```json
{
  "problems": [ ... ],
  "conceptCategories": {
    "스택": "DATA_STRUCTURE",
    "TCP": "NETWORK",
    "SQL 인젝션": "SECURITY"
  }
}
```

**규칙**:
- 허용값은 `ProblemCategory` enum 이름 8개 중 하나: `DATA_STRUCTURE`, `ALGORITHM`, `OPERATING_SYSTEM`, `NETWORK`, `DATABASE`, `COMPUTER_ARCHITECTURE`, `SOFTWARE_ENGINEERING`, `SECURITY`.
- **선택 필드다** — 이 맵에 이름이 없는 개념은 미분류(`null`)로 조용히 생성된다. 카테고리 필드 도입 이전 시드와의 하위 호환이 이렇게 보장된다.
- 개념 하나는 정확히 하나의 카테고리만 가진다(카테고리 1 — 개념 N). 같은 개념명을 서로 다른 카테고리로 두 번 매핑하면 안 된다(JSON 객체 특성상 뒤 항목이 앞을 덮어써 조용히 잘못된 값이 남을 수 있으니 주의).
- 신규 개념을 추가할 때는 이 맵에도 함께 등록하는 것을 원칙으로 한다(미분류 방치 최소화). 다만 분류가 애매하면 비워 두고 '준비 중'으로 남기는 편이 안전하다(명세 §7 '준비 중' 분야 참고).

**도메인 근거** (명세 461~490행, 결정 2026-07-17): 문제의 대표 분류는 대표 개념(concepts 첫 항목)의 카테고리로 결정적으로 도출된다. 카테고리를 개념 위에 얹어 단일 출처로 관리하므로, 같은 개념을 다루는 문제들 사이에 분류가 어긋나지 않는다.

#### `acceptableAnswers` (필수, 배열 1개 이상)
**역할**: 이 문제의 정답으로 인정되는 모든 표기의 집합.

**규칙**:
- **배열 원소는 모두 비어 있지 않아야 함**.
- 한/영 표기, 대표 유의어, 오타 변형 등을 포함한다.
- **예**: `["해시 충돌", "hash collision", "해시 충돌 (hash collision)"]`
- 병기 표기("스레드 (thread)")는 그 **문자열 자체를 등재**해야 한다 (아래 정규화 규칙 참고).
- **정규화 기준으로 중복 제거** — 정규화 후 같은 값은 등재하지 않는다.

**정규화 규칙 (AnswerText.kt에서 실제 코드)**:
정사용자 입력과 허용답은 다음 순서로 정규화되어 대조된다:
1. **유니코드 NFC 조합** — 한글 자모 분해형(NFD)을 조합형으로 통일 (입력 인코딩 무관 결정성 보장).
2. **공백 트림** — 앞뒤 공백 제거.
3. **소문자화** — 대문자를 소문자로.
4. **내부 공백 축약** — 연속된 공백을 한 칸 공백으로.
5. **구두점은 제거하지 않음** (중요!) — `o(n^2)`, `TCP/IP`, `C++` 같이 기호가 의미를 갖는 답이 깨지기 때문. 구두점 표기 흔들림은 허용답 큐레이션(변형 등재)으로만 대응.

**예시**:
- 입력 "스택 (stack)"  → 정규화: "스택 (stack)"
- 입력 "STACK   " → 정규화: "stack"
- 입력 "o(n²)" vs 허용답 "o(n^2)" → 정규화 후 "o(n²)" ≠ "o(n^2)" (기호 유지) → 불일치 (별도 등재 필요)

**도메인 근거** (명세 143~149행, 55행): 사용자 입력을 정규화한 뒤 이 집합과 문자열 대조해 결정적으로 판정한다. AI 판정이 아니라 큐레이션된 집합으로 검증하므로, 완전성과 정확성을 보장하려면 허용답 큐레이션이 잘 되어야 한다.

#### `representativeAnswer` (필수, 문자열)
**역할**: 화면(정답 공개·지난 문제·스크랩 상세, 정답 처리 응답)에 보여줄 **대표 정답 하나**.

**규칙**:
- 비어 있으면 안 됨.
- **불변식**: 정규화([AnswerText]) 기준으로 `acceptableAnswers`에 포함되어야 한다.
  - **근거**: 정답 공개 후 '따라 입력'에서 사용자가 화면의 대표 정답을 그대로 치면 반드시 통과해야 한다 (명세 56~58행).
  - 즉, `AnswerText(representativeAnswer).value in acceptableAnswers.map { AnswerText(it).value }` 가 **true**여야 한다.

**병기 표기 처리** (주의):
- 대표답이 "스레드 (thread)" 같이 한/영 병기라면?
- AnswerText는 구두점을 제거하지 않으므로, 정규화되어도 "스레드 (thread)" 형태로 남는다.
- 따라서 `acceptableAnswers`에 **"스레드 (thread)" 문자열 자체도 등재**해야 한다.
- **예**: 
  ```json
  "representativeAnswer": "스레드 (thread)",
  "acceptableAnswers": ["스레드", "thread", "스레드 (thread)"]
  ```

**도메인 근거** (명세 55~58행): 오너 결정 2026-07-16. 화면은 허용 표기를 나열하지 않고 대표 정답 하나만 보여준다(간결함, 학습자 친화).

#### `explanation` (선택, 문자열)
**역할**: 정답 후 사용자에게 공개되는 짧은 해설.

**규칙**:
- 없으면 생략 가능.
- 정답과 개념을 다시 확인시켜주는 수준이면 충분.
- 관련된 다른 개념을 언급해도 됨 — explanation도 정답 처리 후 공개되는 큐레이션 콘텐츠라 심화 정보와 같은 논리가 적용된다. 의미적 스포일 여부는 큐레이터 승인 게이트 소관([결정 2026-07-17]).

**도메인 근거** (명세 60행): 큐레이션 콘텐츠.

#### `hints` (선택, 배열)
**역할**: 문제에서 막혔을 때 포기 대신 넘게 돕는, 약→강 순서의 힌트 (0~N개).

**규칙**:
- **배열 원소 구조**: `{ "text": "...", "codeSnippet": null }`
  - `text`: 힌트 본문 (필수, 비어 있지 않아야 함).
  - `codeSnippet`: 코드 예시 (선택, 없으면 null 또는 생략).
  - 순서: 배열 인덱스가 약→강을 정함 (첫 번째가 가장 약함).

- **개수 제약 없음** — 0개(힌트 없음)부터 N개까지 필요한 만큼만. 고정 "L1·L2·L3" 사다리는 없다.

- **학습적 가치** (중요):
  - 각 힌트는 **그 개념을 이해하게 돕는** 학습적 가치를 가져야 한다.
  - 단순히 정답을 맞히게만 하는 '풀이용 실마리'는 지양한다 (예: 첫 글자 힌트 "스"는 학습 가치 없음).
  - 배경지식 보충, 접근 관점, 관련 개념 등 문제 성격에 맞는 힌트를 고른다.
  - 코드를 안 봐도 되는 문제에는 코드 힌트를 넣지 않는다.

- **정답 문자열 절대 비노출** (무낙인 유지):
  - 힌트에 정답 표기(예: "스택", "stack", "스택 (stack)")를 담으면 안 됨.
  - 자동 검증 스윕이 이를 잡음 (no-leak 검사 — 기존 로드 검증 상속).
  - **예**: 정답이 "해시 충돌"인 문제에서 "체이닝이나 개방 주소법으로 **충돌**을 해소한다"는 OK (일반 개념), "**해시 충돌**을 해소한다"는 금지.

- **고정 사다리 금지** (명세 252행):
  - "항상 키워드 힌트 → 배경지식 → 풀이"처럼 고정된 순서·개수를 두지 않는다.
  - 각 문제의 성격에 맞는 힌트만, 필요한 만큼만 구성한다.

**도메인 근거** (명세 65~68행, 249~256행): 힌트는 '포기 대신 이해'를 실현하는 히어로. 점진성·필요한 만큼만·학습적 가치가 핵심이다.

#### `misconceptionHints` (선택, 배열)
**역할**: 사용자가 특정 오답을 제출했을 때 자동으로 표시되는 교정 힌트.

**규칙**:
- **배열 원소 구조**: `{ "expectedAnswers": [...], "message": "..." }`
  - `expectedAnswers`: 예상 오답 표기의 배열 (1개 이상).
  - `message`: 교정 메시지 (비어 있지 않아야 함).

- **예상 오답 집합**:
  - 허용답처럼 정규화([AnswerText]) 기준으로 대조된다.
  - 다른 개념의 답이거나 흔한 혼동을 대상으로 한다.
  - **예**: 정답이 '스레드'인 문제에서 예상 오답 `["프로세스", "process"]`.

- **교정 메시지**:
  - "왜 그 답이 **다른지**"만 짚는다 — 정답을 말하지 않는다.
  - 무미건조하지 않게, 학습자가 스스로 바로잡게 한다.
  - **예**: "프로세스는 실행 중인 프로그램 그 자체예요. 이 문제는 그 '안에서' 자원을 공유하며 도는 더 작은 실행 흐름을 묻고 있어요. 다시 도전해보세요!"

- **정답 비노출** (무낙인 유지):
  - 교정 메시지에 정답을 담으면 안 됨 ("답은 **스레드**다" 금지).
  - 기존 no-leak 자동 검증이 이를 잡음.

- **개수 제약 없음** — 있는 만큼만 (0개도 가능).

**도메인 근거** (명세 280~304행, 기능 2.5): 오답을 처벌이 아니라 **학습 신호**로 다룬다. 오답이 자동으로 표시되어도 제출은 오답으로 확정하지 않고, 사용자는 무제한 재시도할 수 있다.

#### `enrichment` (선택, 객체 또는 null)
**역할**: 정답 처리 후 선택적으로 노출되는 '더 알아보기' 심화 정보.

**구조**:
```json
{
  "title": "왜 충돌이 발생할까요?",
  "body": "해시 테이블은 무한한 데이터를 유한한 크기의 배열에 매핑하기 때문에…",
  "items": [
    { "title": "해결책 01. 체이닝", "description": "…" },
    { "title": "해결책 02. 개방 주소법", "description": "…" }
  ],
  "quote": "좋은 해시 함수는 충돌을 최소화하여…"
}
```

**규칙**:
- **title** (필수): 카드 제목. 질문형 추천 (예: "왜 충돌이 발생할까요?").
- **body** (필수): 리드 문단. 핵심 내용 1~2문장.
- **items** (선택, 배열 0~N개): 보조 항목 카드. 각각 `title`과 `description` 필수. 순서 보존.
- **quote** (선택): 인용문 또는 참고 문구.

- **스키마상으로는 선택 필드** — `enrichment: null`이면 노출 자체가 생략되는 graceful 분기가 존재한다(클라이언트·역직렬화 레벨의 안전장치).
  - 다만 **콘텐츠 품질 기준으로는 전 문제 작성이 원칙**이다 — 심화 정보가 없는 문제는 테스터·사용자에게 품질 저하로 인식되므로, 신규 생성 시 `enrichment: null`을 남기지 말고 전 문제에 작성한다.
  - [결정 2026-07-17] 심화 정보 전 문제 작성 원칙 — 심화 없는 문제가 테스터/사용자에게 품질 저하로 인식되어 상향평준화. 시드의 '심화 없는 문제 최소 1개' 테스트는 폐기(null 분기는 UI·역직렬화 전용 테스트가 커버).

- **타 개념 언급 허용, 의미적 스포일은 큐레이터 소관**:
  - 이 문제의 개념은 물론, 관련된 다른 CS 개념을 자연스럽게 언급해도 됨 — CS 개념은 서로 연관되어 있고, 좋은 심화 정보일수록 이웃 개념을 끌어온다.
  - **예**: 이 문제의 정답이 "해시 충돌"이면 "해시 충돌은 …"이라고 설명할 수 있고, 다른 문제의 정답 "체이닝"이나 "로드 팩터"를 여기서 함께 언급해도 무방함.
  - 다만 다른 문제의 정답을 대놓고 흘려 그 문제를 무의미하게 만드는(의미적 스포일) 수준인지는 자동 검증 대상이 아니며, 명세의 전량 큐레이터 승인 게이트에서 판단한다.
  - [결정 2026-07-17] 심화 정보의 전역(타 문제 정답) no-leak 제약 제거 — 개념 간 연관 언급 허용, 스포일 여부 판단은 큐레이터 승인 게이트 소관.

- **모바일 가독성**:
  - 코드 스니펫이 있으면 모바일 화면(약 40~50자)에 맞춰야 함.
  - 라인이 과도하게 길면 안 됨.

**도메인 근거** (명세 175~183행, 60~61행): 정답 후 심화 정보는 학습을 강화하고 흥미로 이어가는 선택적 콘텐츠. 진행을 막지 않으므로, 보고 싶으면 보고 안 보면 넘어간다.

## 4. 품질 기준

**생성 목적이 테스트이므로** 모든 콘텐츠가 CS 사실과 정확히 일치할 필요는 없다. 다만 다음은 필수다:

1. **논리적 자기모순 없음**:
   - 정의 재생형인데 정답이 개념 이름이 아니면 안 됨.
   - 유도형인데 정답이 개념 이름만인 문제도 논리적으로 맞지 않음 (유도 과정이 없으므로).
   - 정답과 허용답 간 의미 관계가 일관되어야 함 (허용답이 정답과 다른 개념을 가리키면 안 됨).

2. **정답 판정 결정성**:
   - 같은 입력에는 같은 판정이 나와야 한다 (정규화 규칙 일관 적용).
   - 정규화 후 허용답 집합에 포함되지 않으면 불일치.

3. **허용답 충실성**:
   - 정답의 흔한 표기 변형을 대부분 포함해야 한다.
   - 허용답 누락으로 인한 '거짓 오답'(사실상 맞는데 불일치로 판정)을 최소화.

## 5. 검증 절차

   ```bash
   ./gradlew :server:test
   ```

   - **no-leak 스윕 테스트**:
     - 모든 문제의 힌트·오답 교정 힌트에서 **자기 문제**의 정답 문자열 비포함 검증.

   - **구조 검증 (자동)**:
     - 정의 재생형 근접 신호 임계값 검사.
     - 편집거리 계산 정상 작동 여부.

3. **테스트 통과 확인**:
   - 기준선: `server 255 / jvmTest 382 / 실패 0` 유지 (시더 테스트 개정분 증감 설명).

4. **로컬 확인** (선택):
   ```bash

   ```
   - 애플리케이션이 정상 기동하고 7개 + 신규 문제가 시딩되는지 확인.

5. **배치 반복**:
   - 테스트 통과 시 다음 배치 진행.

   - 약 100문제에 도달할 때까지 반복.

## 6. 금지 사항과 주의

다음은 **반드시 하지 말아야 할 것**이다.

### 금지

1. **ProblemSeeder.kt 부활 금지**:
   - 기존 ProblemSeeder는 제거되었다.
   - 새 문제를 추가하려고 다시 만들면 안 됨.
   - JSON 데이터 방식으로 통일되어야 함 (로더가 담당).

2. **기존 7개 문제 수정 금지**:
   - 기존 `problems-generated.json`의 초기 7문제(기존 ProblemSeeder 이관분)는 수정하면 안 됨.
   - 분포 검증과 no-leak 검증이 이 7문제의 의도된 구조를 신뢰함.

3. **JSON 외 파일 수정 최소화**:
   - 코드 수정은 특별한 이유가 없으면 하지 않음 (gradle 금지이므로 어차피 어려움).
   - 새 로더·도메인 클래스 추가도 최소화 (로더는 worker-seed가 구현).
   - JSON만 수정하는 것이 원칙.

4. **`작업 인수인계.md` 커밋 금지**:
   - 이 파일은 본인의 워크플로우 기록이므로 커밋하면 안 됨.
   - 최종 커밋은 `docs: 문제 데이터 생성 가이드 작성`만 (신규 가이드 문서 1개).

5. **gradle 실행 금지**:
   - worker-guide는 docs/dev 전용 (다른 워커가 워크트리에서 빌드 중).
   - test는 본인이 직접 실행할 수 없음 (테스트 검증은 이 세션 또는 운영 파이프라인에서).

6. **푸시 금지**:
   - 모든 커밋은 로컬에만 남김.
   - 원격 저장소에 푸시하지 않음.

### 주의

1. **개념 풀을 먼저 설계**:
   - 문제를 무작정 100개 생성하는 것보다, 먼저 개념 30~40개를 정하고 그 위에 문제를 구성하는 게 복습 시스템의 가치를 높인다.

2. **`type` 태깅 신중**:
   - 모르면 `null`로 두는 게 낫다.
   - 유도형을 정의 재생형으로 잘못 태깅하면 근접 신호 버그가 재발한다.

3. **정규화 규칙 정확히**:
   - AnswerText에서 구두점을 제거하지 않으므로, `o(n^2)`와 `o(n²)`는 다른 문자열이다.
   - 허용답 큐레이션에서 표기 변형을 모두 등재해야 한다.

4. **대표 정답 불변식 확인**:
   - 병기 표기("스레드 (thread)")를 사용하면, 그 문자열 자체가 허용답에도 등재되어야 한다.
   - 정규화 후 대표 정답이 허용답 집합에 포함되는지 수동으로 검증하면 좋다.

5. **no-leak 검증**:
   - 자동 검증이 이를 잡지만, 생성 시부터 의식해서 작성하면 수정 반복이 줄어든다.
   - 힌트·오답 교정 힌트에 **자기 문제**의 정답 문자열을 담지 않도록 주의(심화 정보는 대상 아님).

## 8. 참고: 기존 7문제

현재 `problems-generated.json`에는 기존 ProblemSeeder에서 이관한 7개의 시드 문제가 있다:
1. 스레드 (복수 개념, 힌트 2개, 심화 정보, 오답 교정)
2. 스택 (단순, 힌트 없음)
3. 큐 (단순, 힌트 없음)
4. 해시 충돌 (정의 재생형, 힌트 2개, 심화 정보)
5. TCP (유도형, 힌트 1개)
6. 캐시 (유도형, 심화 정보 있음)
7. 시간 복잡도 (유도형)

이들은 **분포 검증 기준선**이자 **도메인 규칙의 살아있는 예시**다. 신규 100문제를 생성할 때 이 7개와의 개념 공유, 힌트·심화의 톤과 구성을 참고하되, **수정하지 말 것**.

## 참고: 용어 정의

- **허용답 (acceptable answer)**: 사용자가 제시해도 정답으로 인정되는 모든 표기. 정규화 후 집합 대조.
- **대표 정답 (representative answer)**: 화면에 보여주는 하나의 표기. 정규화 기준으로 허용답 집합에 포함되어야 함.
- **개념 재사용**: 한 개념이 여러 문제에 나타남 (복습 스케줄링의 전제).
- **정규화**: 사용자 입력과 허용답을 NFC, 트림, 소문자화, 공백 축약으로 통일된 형태로 변환해 비교.
- **no-leak**: 힌트·오답 교정 힌트에서 **자기 문제**의 정답 문자열이 새지 않도록 자동 검증하는 기능(심화 정보는 대상 아님 — [결정 2026-07-17]).
- **근접 신호 (near-miss)**: 정의 재생형 문제에서 편집거리 1~2 범위 불일치 시 "거의 맞았어요, 오타를 확인해보세요" 안내. **[반영 2026-07-19: D1]** 정답 공개 후 '따라 입력' 구간에서는 유도형에도 적용되는 예외가 있다(타입 태깅과 무관한 런타임 분기 — 위 `type` 절 참고).
